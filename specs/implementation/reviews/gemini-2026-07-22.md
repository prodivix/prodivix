## [HIGH] Client clock drift causes asset delivery session validation failure

**File:** apps/web/src/editor/editorApi.ts:344

**Category:** correctness

**Description:**
In `editorApi.ts`, `readWorkspaceAssetDeliverySession` validates the server's response for `createWorkspaceAssetDeliverySession`. It checks that `value.expiresAt` is a safe integer and `value.expiresAt <= Date.now()`. Wait!!!

```typescript
    typeof value.expiresAt !== 'number' ||
    !Number.isSafeInteger(value.expiresAt) ||
    value.expiresAt <= Date.now() ||
```

Wait, if `value.expiresAt <= Date.now()`, it THROWS A TYPE ERROR?!
Yes! If `expiresAt` is in the past, it throws! The condition is `|| value.expiresAt <= Date.now() ||`.
This means it expects `value.expiresAt` to be in the FUTURE.
Wait, what if the client's clock is slightly ahead of the server's clock?
If the server generates an `expiresAt` that is 5 minutes in the future, but the client clock is 10 minutes ahead, `value.expiresAt <= Date.now()` will evaluate to TRUE, and it will throw `TypeError('Workspace asset delivery session is invalid.')`.
This is a classic time-sync bug! The client clock cannot be trusted to strictly evaluate short-lived server expirations at the millisecond level, especially throwing a validation error.
Wait, `expiresAt` is usually a Unix timestamp in milliseconds. Does the server return milliseconds?
If the server returns seconds, `value.expiresAt` (e.g. 1700000000) will be WAY LESS than `Date.now()` (e.g. 1700000000000), and it will ALWAYS throw!
Let's check if the server returns seconds or milliseconds.
Assuming it returns milliseconds, the clock drift issue still exists. If the expiration is short (e.g., 60 seconds), a client with a fast clock will fail to validate the session. A validation schema should not reject a payload strictly based on `Date.now()`. Validating structural integrity should be decoupled from evaluating business logic (like expiration), or at least tolerate clock drift.

**Scenario:**
If the server returns a delivery session that expires in 60 seconds, but the user's OS clock is 2 minutes fast, `value.expiresAt <= Date.now()` is true. The API client throws a `TypeError: Workspace asset delivery session is invalid`, breaking asset loading completely for that user, even though the server would have accepted the URL. If the server returns Unix seconds, it will fail 100% of the time.

For a document with N tokens and R references, this results in O(R * N log N) time complexity. On large design systems (e.g., 5,000+ tokens), this will block the main thread for seconds. The lookup should be optimized to O(K) by iteratively slicing the pointer path and querying the map directly.

**Scenario:**
Loading a large design token document with thousands of tokens and references causes the application UI thread to freeze for multiple seconds during the codec parsing phase.

## [HIGH] DOM XSS vulnerability via innerHTML injection of unsanitized value in PdxRichTextEditor

**File:** D:\Projects\prodivix\packages\ui\src\form\PdxRichTextEditor.tsx:68

**Category:** security

**Description:**
`PdxRichTextEditor` allows a `value` prop to be set. This value is assigned to `currentValue`, and the component updates `editorRef.current.innerHTML = currentValue;` directly. If the provided `value` string contains malicious `<script>` tags or inline event handlers (e.g., `<img src=x onerror=alert(1)>`), it will be injected straight into the DOM as HTML, leading to a DOM-based XSS attack. The `value` prop (or `defaultValue`) must be sanitized, or the usage of `innerHTML` must be protected.

**Scenario:**
If data sourced from an external, untrusted environment (e.g. user-generated content from a database) is passed as the `value` prop to `<PdxRichTextEditor value={untrustedData} />`, the untrusted data will be inserted via `.innerHTML` directly into the DOM, executing any malicious payload embedded within it.
