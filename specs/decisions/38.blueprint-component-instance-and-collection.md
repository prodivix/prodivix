## 决策摘要

Blueprint 的复用模型由三个一等概念组成：

1. **Component Definition**：`pir-component` Workspace document，拥有实现 graph 与公开 `ComponentContract`。
2. **Component Instance**：消费方 PIR document 中以稳定 `componentDocumentId` 引用 Definition 的节点；props、events 和 variants 只能绑定 Contract member，slot 内容通过 Instance 所拥有的 named regions 提供。
3. **Collection**：拥有显式 item / empty / loading / error regions 和稳定 item/index scope 的结构节点；它可以在 item region 中复用 Component Instance。

“抽取为组件”不是复制 JSON 或 UI 局部操作。它必须先分析选中子图的引用和作用域，再用一个原子 Workspace Transaction 创建 Definition、生成 Contract、移动子图、插入 Instance 并重写所有受影响引用。

Component Instance 与 Collection 只属于无版本号的 PIR-current 领域模型。数字 wire version 不参与其 validator、Renderer、Compiler、Workspace 或 Blueprint 架构。

以上能力属于唯一 PIR-current 生产架构。数字版本只由 wire codec/migration 与 persistence 消费；S5 继续完成 Blueprint 产品表面，S6 只负责 Golden conformance。

Route-level document reuse 负责路由模块复用；Component Definition、Component Contract、Component Instance 与 Collection 共同构成 Blueprint 的 G1 复用模型。

## 术语与所有权

### Component Definition

Component Definition 是 `type: 'pir-component'` 的 Workspace document。它拥有：

- 稳定 `documentId`；
- normalized PIR graph；
- 公开 `ComponentContract`；
- 本组件拥有的 state、regions 和结构语义；
- 对 Code、Token、Asset、NodeGraph、Animation 与其他 Component 的类型化引用。

文件 path 和显示名称可以变化；`documentId` 是实例与跨文档引用的稳定主键。

### Component Contract

```ts
type JsonValue =
  null | boolean | number | string | JsonValue[] | { [key: string]: JsonValue };

type ComponentContract = {
  version: 1;
  propsById: Record<string, ComponentPropContract>;
  eventsById: Record<string, ComponentEventContract>;
  slotsById: Record<string, ComponentSlotContract>;
  variantAxesById: Record<string, ComponentVariantContract>;
  partsById?: Record<string, ComponentPartContract>;
  tokenBindings?: ComponentTokenContract[];
  accessibility?: ComponentAccessibilityContract;
};

type PirValueBinding =
  | { kind: 'literal'; value: JsonValue }
  | { kind: 'param'; paramId: string; path?: string }
  | { kind: 'state'; stateId: string; path?: string }
  | { kind: 'data'; dataId: string; path?: string }
  | { kind: 'collection-symbol'; symbolId: string; path?: string }
  | { kind: 'component-prop'; memberId: string; path?: string }
  | { kind: 'code'; reference: CodeReference };
```

Component Contract 是 Definition 的类型化公开边界，不是从 UI 临时推断的缓存，也不是插件 palette metadata 的别名。

在 PIR-current 中，Contract 与 Definition graph 同属 `pir-component` document 的 canonical content；所属 Component identity 直接使用外层 `WorkspaceDocument.id`，Contract 内不重复保存第二个 document id。Registry 和 Workspace Semantic Index 只提供查询投影，不另存一份可独立演化的 contract。

规则：

1. prop 声明类型、required/default 与可绑定能力。
2. event 声明 payload contract；复杂 handler 通过 CodeSlot / CodeReference 接入。
3. slot 声明消费者可提供内容的命名 region、数量和允许能力。
4. variant 是公开选择，不是任意内部 node patch。
5. part 是显式公开的样式、动画、测试或定位 target；它不允许消费者任意遍历 Definition 内部 graph。
6. token binding 和 accessibility requirement 可以成为验证与导出的 contract。
7. Contract 变更必须产生跨 Workspace impact analysis 和可定位诊断。
8. 每个 contract member 的稳定 `id` 是绑定身份；name 只用于显示与目标代码命名。

### Component Instance

Component Instance 是 PIR graph 中独立于 native/adapted runtime component node 的节点类型：

```ts
type ComponentInstanceNode = {
  id: string;
  kind: 'component-instance';
  componentDocumentId: string;
  bindings: {
    props: Record<string, PirValueBinding>;
    events: Record<string, TriggerBinding>;
    variants: Record<string, string>;
  };
};
```

以上结构表达语义边界，不承诺最终字段命名。

规则：

1. Instance 通过 `componentDocumentId` 指向同一个 canonical Definition；修改 Definition 后所有实例自然消费新 revision。
2. Instance 只能通过 Contract 绑定 props、events、slots 和 variants。
   `bindings.props/events/variants` 的 key 分别是 stable prop/event/variant-axis member id，variant value 是 stable option id。
3. G1 不允许任意深层 node override；这会制造无法验证的派生组件分叉。
4. slot 内容归消费方 document 所有，保存在 `regionsById[instanceNodeId][slotMemberId]`；不在 Instance 上再复制一份 slots tree。
5. Instance 不复制 Definition graph，不保存内部节点快照。
6. missing component、missing contract member、type mismatch 和 inaccessible reference 必须产生稳定诊断。
7. detach 若后续提供，必须是显式 Transaction：物化独立子图并移除 instance identity；它不是静默 fallback。

Definition 内以 `component-slot-outlet` 节点声明 slot 投影位置，outlet 通过稳定 `slotMemberId` 指向自身 Contract；其后代只表示 fallback。G1 中同一个 slot member 最多有一个 outlet，避免一次消费方内容被隐式投影到多个位置。

### Component lexical scope

1. Instance 的 prop、event 和 variant bindings 在消费方 scope 求值。
2. Definition body 只能读取自身 state/data、Contract props 和显式 shared references；不能隐式读取某个消费方的局部 scope。
3. Instance slot region 中的节点保持消费方 lexical scope，包括外层 Collection 的 item/index symbols；它们不能隐式读取 Definition 的内部 state/data。
4. 若 Definition 要向 slot 内容提供值，必须由 slot contract 显式声明 slot props，并为每次投影创建受约束的 slot scope。
5. Renderer 与 Compiler 必须使用相同 scope composition，不能通过 React closure 或生成器局部变量引入额外可见性。

## 组件依赖图

每个 Component Definition 对其他 Component Definition 的 Instance 引用形成有向依赖图。

Workspace Validator / Command planner 负责阻断非法写入；Workspace Semantic Index 只表达依赖、影响与 semantic diagnostic：

1. Validator / planner 拒绝直接或间接循环实例依赖。
2. 区分 missing、inaccessible、ambiguous 和 cyclic reference。
3. 支持 definition、references 和 impact query。
4. 在删除或 contract breaking change 前列出所有 route、document、instance、collection 和 export target 影响。
5. 不以组件名称、path 或当前 UI tree 位置判断 identity。

## “抽取为组件”原子事务

### 输入约束

抽取 planner 接收一个 Workspace revision、源 PIR document、单一选中根和完整后代集合。非连续的任意节点集合不直接成为一个组件；多选若要支持，必须先被规范化为一个可验证 fragment。

### 分析阶段

Planner 生成可审阅 extraction plan：

1. 选中子图内部引用继续保留。
2. 指向外部 data/state/prop 的入站引用转为公开 prop 或明确的 shared reference。
3. 从组件内部发往外部的行为边界转为 event contract 或保留合法 route/graph/code reference。
4. 消费方提供的可变子内容转为显式 slot。
5. Token、Asset、CodeArtifact、Route、NodeGraph 和 Animation 等已有稳定对象保持类型化引用，不复制为局部字符串。
6. 跨 document target 变化生成 relocation map，供 animation、behavior、diagnostic target 和其他引用 owner 规划重写。
7. 从子图外部指向将被移动内部 node 的引用不能直接改写为 Definition internal node：其 owner 必须随子图迁入、目标必须提升为公开 Component part/contract member 并改写为 instance public target，二者都不成立时阻断抽取。
8. 选择源 document root 是合法输入；Transaction 用新 Component Instance 替换 source `rootId`。非 root selection 则保持原 parent、region 和 sibling index。
9. planner 必须在写入前报告 unsupported dependency、cycle 和无法安全重写的引用。

### 应用阶段

一个 extraction Transaction 必须原子完成：

```text
create pir-component document
  -> write ComponentContract
  -> move normalized subtree into definition graph
  -> replace source subtree with ComponentInstance
  -> rewrite every affected typed reference from relocation map
  -> validate Workspace + PIR + Component + domain references
  -> record one local History boundary
  -> WorkspaceOperation / Durable Outbox / Atomic Commit
```

任何一步失败都不得留下半个 component document、悬空 instance 或只在 UI store 中存在的补偿状态。Undo/redo 必须恢复整个跨文档事务。

## Collection 语义

### 一等 Collection Node

Collection 是控制重复与数据状态的结构节点，本身不承诺生成额外 DOM：

```ts
type CollectionNode = {
  id: string;
  kind: 'collection';
  source: CollectionSourceBinding;
  key: CollectionKeyBinding;
  symbols: {
    itemId: string;
    itemName: string;
    indexId: string;
    indexName: string;
    errorId?: string;
  };
};

regionsById[collectionNodeId] = {
  item: NodeId[];
  empty?: NodeId[];
  loading?: NodeId[];
  error?: NodeId[];
};

type CollectionSourceBinding =
  | { kind: 'literal'; value: JsonValue[] }
  | { kind: 'binding'; value: PirValueBinding };

type CollectionKeyBinding =
  | { kind: 'binding'; value: PirValueBinding }
  | { kind: 'index' };
```

规则：

1. `item` region 是模板；运行时每个 item 产生 scope 实例，不向保存态复制节点。
2. `empty`、`loading` 和 `error` 由互斥状态 region 显式拥有。
3. region children 的唯一保存位置是 `ui.graph.regionsById[collectionNodeId]`；Collection node 不再复制 region id 或 children 映射。
4. item/index/error 具有稳定 semantic symbol id；显示别名可以修改而不改变引用 identity。
5. reference key 在 item scope 中求值。使用 index 作为 key 必须通过 `{ kind: 'index' }` 明确选择，并在可重排数据上产生 warning；不得隐式 fallback。
6. item region 可包含普通 PIR node、Component Instance 或嵌套 Collection。
7. 嵌套 Collection 的子 scope 可以读取合法父 scope，并可以显式遮蔽显示名称；解析按 symbol identity 和 scope graph 完成。
8. Collection source、key、prop binding、event parameter、CodeSlot 和 animation target 必须使用同一 semantic scope。

### G1 与 G2 边界

G1 必须冻结完整四状态 region contract，并完成 literal/local reference source 的作者、预览和导出一致性。编辑器应允许手动切换 item/empty/loading/error 状态来编辑和验证各 region；该 preview state 只是 UI 偏好，不是领域保存态。

G2 再接入真实 Data/API provider，让 loading/error 由 query lifecycle 驱动，并增加 pagination、cache、retry、optimistic update 和 virtualization。G2 扩展数据来源，不重新定义 Collection、Region 或 item scope。

### Preview 与 Export 一致性

PIR React Renderer 与 Production Compiler 必须消费同一 Collection semantic contract 或通过同一 conformance scenarios。禁止 Inspector、Preview 和 Export 各自解释 source fallback、key、empty state 或 nested scope。

## Workspace Semantic Index 集成

Semantic schema 已建立 Component、Contract member、variant option、slot/slot-prop 与 Collection item/index/error 的 canonical identities。PIR-current Semantic Provider 发布：

- Component Definition、Contract member 与 Instance symbols；
- Definition -> Instance、Instance -> Contract member 引用边；
- component dependency edges；
- Collection、item/index/error scope 与 region ownership；
- source、key、binding 和 nested-scope reference edges；
- extraction、rename、delete、move 和 contract change 所需的 impact links。

保存态仍使用 Component、PIR 和 Collection 领域的类型化引用；索引只提供统一查询和影响图。

Provider、Workspace current read boundary 与 Component graph validator 使用同一 document identity，并已进入默认 WorkspaceSnapshot composition。

## G1 产品工作流

Blueprint 至少提供：

1. 创建空 Component Definition。
2. 从画布选择一个子图并预览 extraction plan。
3. 原子执行“抽取为组件”，然后在 Resources / Component 页面打开 Definition。
4. 从 palette/resources 插入 Component Instance。
5. 在 Inspector 编辑 contract-exposed props、events、slots 和 variants。
6. 从 Instance 跳转 Definition，从 Definition 查找所有 Instance。
7. 创建 Collection、选择 source/key、编辑四个状态 region，并在 item scope 中获得符号补全。
8. 在 Collection item region 中插入 Component Instance。
9. 从诊断跳转 missing reference、contract mismatch、cycle 和 scope error。

Component 页面承载 Definition、Contract、dependency 和 references 的正式作者表面。

## 阶段边界

### G1

- Workspace Semantic Index 与 Component/Collection provider；
- Component Definition、Contract、Instance、dependency validation；
- 原子 subtree extraction、undo/redo、save/reload 与 conflict-safe commit；
- contract-limited props/events/slots/variants；
- Collection source/key/item/empty/loading/error regions 与 nested scope；
- Component Instance 作为 item template；
- Preview/Export/SourceTrace/Diagnostics conformance。

### G2

- Data/API contract、query lifecycle、pagination、cache、retry、optimistic update；
- live/mock adapter 与 SecretRef/runtime-zone 边界；
- virtualization 和大数据集性能策略。

### G3

- instance/collection 上的行为 trace、动画生命周期、稳定 runtime target；
- Blueprint event -> NodeGraph -> Collection state -> Animation -> Code 的行为验证。

## 长期不变量

1. Component Definition 和消费方 document 都属于 Canonical Workspace VFS。
2. Instance 引用 stable `documentId`，不复制 Definition graph。
3. Contract 是唯一公开 override 边界，不允许任意内部 node patch。
4. 组件抽取是跨文档原子 Transaction，不是局部 JSON 复制。
5. Collection 是一等结构语义，item scope 和状态 region 显式建模。
6. Preview 与 Export 不分别猜测组件或列表语义。
7. 所有跨域引用进入 Workspace Semantic Index，但仍由 owner domain 类型化持久化。
8. 无版本 PIR-current 是 Component Instance 与 Collection 的唯一领域保存态和解释 contract；数字版本只属于 wire migration 边界。

## 非目标

1. G1 不支持任意深层 instance override、继承链或 mixin。
2. G1 不要求自动把所有外部引用猜成完美 Contract；无法安全分类时应阻止抽取并给出可操作诊断。
3. G1 不要求真实 API、pagination、cache 或 virtualization。
4. 本决策不把插件 runtime component descriptor 自动升级为用户 Component Definition。
5. 本决策不定义多框架 target 的最终组件文件布局。
