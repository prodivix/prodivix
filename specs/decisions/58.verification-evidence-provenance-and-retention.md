## 背景

G2 的 Execution Session、Console、Network、trace、test report 和 artifact 是可丢弃运行态。它们适合实时调试，
但缺少 Scenario/Plan identity、完整性、provenance、trust、retention 和跨运行比较，不能证明某个 Change 已达到
Verification closure。

反过来，把所有 runtime event 永久保存会复制 Execution store、扩大敏感数据面，并把不完整或被取消的 run
错误升级为证据。G3 需要明确区分原始运行输出、Evidence candidate、durable Evidence 与 Closure。

## 决策

`VerificationEvidence` 是 immutable、append-only、content-addressed 的验证事实。它不属于 Canonical Workspace，
不进入 Workspace replica、History、Outbox 或 publication projection；Baseline 和 Policy 是作者态，Evidence 是
对 exact revision 的外部事实。

### 生命周期

```text
ExecutionJob / ScenarioRun
  -> bounded normalized reports + artifact refs
  -> VerificationEvidenceCandidate
  -> completeness / schema / digest / secret / provenance validation
  -> append-only Evidence manifest + retained artifact blobs
  -> plan cell/check association
  -> derived VerificationClosure
```

Candidate 只存在于 coordinator 内存或短期 staging。只有完整通过 intake gate 的 candidate 才写 durable Evidence。
Malformed、oversized、digest mismatch、credential-bearing 或 identity drift candidate 被拒绝并形成 diagnostic，不产生
“partial evidence”。一次合法但 failed/cancelled/infrastructure-error 的 run 可以形成相应 outcome Evidence，但不能满足
required closure。

### Evidence manifest

Evidence current domain 至少包含：

- evidence id、schema/current codec identity 和 manifest digest；
- project/workspace、exact partition revisions、Executable Snapshot digest；
- ImpactSet、Plan、cell、check、Scenario/Policy/baseline revision/digest 引用；
- run/attempt/provider/job/session identity 与 parent attempt lineage；
- surface、framework target、runtime zone、browser/OS/toolchain/control profile；
- fixture、dependency lock、compiler/runtime/tool versions及输入 digest；
- started/completed/duration 与 `passed / failed / blocked / cancelled / infrastructure-error / unstable` outcome；
- normalized check results、SourceTrace、diagnostic codes、applied exemptions；
- content-addressed artifact references；
- producer、attestation/trust class、redaction policy 和 retention class。

自然语言日志、完整 DOM、任意 stack、request body、header、URL query、cookie、Secret、token 和环境值不进入 manifest。
大报告通过独立 artifact 保存；manifest 只保存 bounded summary、digest、media type、size 和授权引用。

### Artifact promotion

Execution artifact 默认按 G2 TTL 过期。Evidence intake 对 required artifact 执行：

1. 以 provider/job/artifact identity 通过授权 resolver 获取 bytes；
2. 校验 media type、size、digest、budget 和 redaction/scanner policy；
3. 按 exact digest 写入 Verification content-addressed store；
4. 在同一 intake transaction 中写 manifest/reference；
5. 完成后才允许原 execution artifact 自然过期。

Evidence store 可复用 `@prodivix/assets` 的 digest/byte verification primitive，但生命周期、授权、引用和 retention
由 Verification service 拥有；它不是 Workspace Asset，也不进入项目 runtime materialization。

### Provenance 与 trust

稳定 trust class：

- `local-unattested`：本机 Preview/Browser run，适合即时反馈，不满足默认 CI closure；
- `remote-attested`：受控 Remote provider/sandbox 产生，Control Plane attestation 已验证；
- `ci-attested`：受控 CI identity、plan digest 和 artifact chain 已验证；
- `imported-untrusted`：导入的外部报告，只可查看，除非专用 adapter 重新验证。

Producer 不能自报更高 trust。`EvidenceAttestor` port 由 Backend/Control Plane composition 提供，签名输入至少覆盖
manifest digest、artifact digest set、plan/cell/check、snapshot digest、producer identity、issuedAt/expiry 和 policy
generation。验证 key、algorithm 和 key id 可轮换；私钥、OIDC token 和 provider credential 不进入 Evidence。

本地开发不依赖付费云服务：测试使用 deterministic Ed25519 fixture 或 local attestor。真实云 attestation 是部署证据，
不是定义 G3 contract 的前置条件。

### Canonical serialization 与 compatibility

Evidence wire schema immutable/versioned；生产 domain 使用无版本 current model。签名与 digest 使用规范化 JSON：

- object key canonical ordering；
- semantically unordered set 排序、ordered step/result 保持顺序；
- UTF-8、finite number、exact key、duplicate identity 和 byte budget hard cut；
- unknown wire version 通过 migration 或 fail closed。

比较 key 至少包含 Scenario/check stable id、matrix cell、target、control profile、baseline compatibility、tool major/adapter
schema。不同 key 的结果标记 incompatible，不能强行计算 trend 或 visual diff。

### Retention

Retention 使用 policy-controlled class，而非 UI 任意天数：

- `session`：短期本地/调试证据，可自动过期；
- `change`：支撑一个 revision/change closure；
- `release`：被外部 release/review 引用时受保护；
- `legal-hold`：仅后续组织权限可管理。

Workspace Policy 可以请求 class/minimum，Backend/organization policy 决定允许的上下限。引用中的 Closure、release
或 hold 未解除时不得物理删除 artifact。清理按 manifest/reference/blob 三阶段执行，产生 durable tombstone 和 aggregate
audit；删除后 API 明确返回 `expired/deleted`，不能伪装 not-found。

G3 不提前实现 G5 的组织 retention UI、deployment promotion 或 production telemetry，但冻结相同 port。

### Privacy 与 Secret hard cut

Evidence intake 复用 G2 redaction，并增加 schema-level denylist/canary scan：

- Secret value、Authorization/Cookie、environment material、signed URL、private locator、raw request body 禁止持久化；
- screenshot/DOM/accessibility artifact 进入前执行尺寸、格式、metadata 和 sensitive-region policy；
- test failure message、console/network/security finding 受字段与 UTF-8 byte budget；
- canary 命中时整个 candidate fail closed，不能只删掉命中字段后继续标为 passed；
- diagnostic/meta 只保存 stable code、bounded context 和 SourceTrace。

### Immutability、supersession 与 baseline

Evidence 创建后不可编辑、重新签名或覆盖。重跑创建新 attempt；更高 attempt 不删除先前 failure。错误归类、exemption、
policy 或 baseline 改变后必须形成新 Plan/Evidence。

Visual baseline 不是 Evidence。Baseline digest/reference 存于 canonical `behavior-scenario` 或 `verification-policy`，bytes
复用 exact-byte blob primitive；“接受新 baseline”是显式可逆 Workspace Transaction。Evidence 只能建议 baseline update，
不能自动改写。

## Backend service boundary

Backend Verification service 需要独立表/port：

- evidence manifest/attempt；
- plan/check association；
- artifact blob/reference；
- attestation verification key metadata；
- retention protection/tombstone/audit。

所有 create/read/list/compare/delete-proposal API 进行 project/workspace authorization。Evidence create 必须幂等绑定
candidate id + digest；同 id 不同 digest hard conflict。客户端不能提交 arbitrary trust/retention escalation。

## 拒绝的方案

### 把 Execution Session 标记为 durable

拒绝。Session 包含调试噪声和 provider lifecycle，不具备 Plan completeness、trust 或长期 schema。

### 把 Evidence 保存进 Workspace document

拒绝。append-only report/blob 会污染作者态 revision、同步和 Git projection。

### 只保存一个 passed boolean

拒绝。缺少输入、toolchain、SourceTrace、artifact、policy 和 trust，无法审计或比较。

### 重跑成功后覆盖失败

拒绝。它破坏 provenance，并隐藏 flaky/unstable 行为。

## 后果

- G3 需要独立 Verification store、API、artifact retention 与 attestor port。
- Web 将 active Execution 与 durable Evidence 分成两个视图，但通过 job/plan identity关联。
- G4/G5 只能引用 immutable evidence/closure digest，不能上传自报 passed 状态。
- Evidence 的存储成本、隐私和删除成为一等 Gate。

## 验收

- [ ] Candidate intake 对 schema、identity、digest、budget、Secret 和 provenance fail closed。
- [ ] Evidence manifest/artifact append-only、幂等且 content-addressed。
- [ ] local/remote/CI trust 不可由 producer 提权。
- [ ] failed/retry/unstable attempt 全部保留 lineage。
- [ ] retention protection、sweep、tombstone、authorization 和跨副本并发 Gate 完成。
- [ ] Baseline 只能通过 Workspace Transaction 更新，Evidence 永不直接写作者态。
