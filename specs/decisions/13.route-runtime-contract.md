## 决策背景

Route Manifest 只描述路径和文档引用还不够。真实可用的路由系统需要统一承载：

1. 路由匹配和动态参数。
2. loader / action / guard 等 code-owned 能力。
3. error boundary、loading state、SEO、实验和监控。
4. 编辑器预览、生产导出和 AI patch 的同构上下文。

如果运行时能力散落在 `PdxRoute` props、事件 handler、页面局部状态或导出器私有逻辑中，后续会很难做一致诊断和跨框架导出。

## 决策内容

在 RouteGraph 上引入框架无关的 **Route Runtime Contract**。运行时契约由 route node 引用 code artifacts，不内嵌裸代码字符串。

```ts
type RouteRuntime = {
  loaderRef?: CodeReference;
  actionRef?: CodeReference;
  guardRef?: CodeReference;
  errorBoundaryDocId?: string;
  suspenseDocId?: string;
  seo?: {
    title?: string;
    description?: string;
    canonical?: string;
    noIndex?: boolean;
    openGraph?: Record<string, string>;
  };
  experiment?: {
    key: string;
    variantMap?: Record<string, string>;
  };
};
```

`loaderRef`、`actionRef`、`guardRef` 必须接入 Code Authoring Environment，并由 Code Semantic Provider 发布到 Workspace Semantic Index。Blueprint、NodeGraph、Animation 和 AI 都通过稳定引用与诊断目标操作这些代码能力。

## RouteRuntimeContext

编辑器、renderer、导航 action 和导出器共享以下上下文：

```ts
type RouteRuntimeContext = {
  currentPath: string;
  matchedPath: string;
  params: Record<string, string>;
  searchParams: Record<string, string | string[]>;
  hash?: string;
  matchChain: ResolvedRouteMatch[];
  activeRouteNodeId?: string;
  routeModuleScope?: string;
};
```

`PdxRoute`、`PdxOutlet` 和内置 `navigate` action 都消费这个 context。它们不自行定义项目级 path 解析规则。

## 执行顺序

1. Route Resolver 根据 URL 和 RouteGraph 得到 `matchChain`。
2. 从根到叶执行 `guardRef`。
3. 按策略执行 `loaderRef`。
4. 根据 `matchChain` 装配 layout + page + Outlet。
5. 提交类事件通过 `actionRef` 处理。
6. 应用 SEO、experiment 和监控 metadata。

异常处理：

- 任一节点异常优先命中最近的 `errorBoundaryDocId`。
- loader / guard / action 诊断落到对应 `CodeReference` 和 route node。

## 与 Blueprint 的关系

1. 基础用户只看到路由树、页面、布局和 Outlet。
2. runtime 配置放在 route inspector 的高级区。
3. loader / guard / action 使用 Code Authoring Environment 创建、编辑、引用和诊断。
4. AI 可以提出 route runtime patch，但必须经过 command / intent / validator。

## 与导出的关系

导出器读取 `ResolvedRouteGraph` 和 runtime refs，按目标框架生成对应路由产物：

1. React / Vite：生成 route objects、lazy boundaries 和 loader adapters。
2. Vue / Nuxt-like target：生成 route records 和 meta。
3. Web Components / native：生成轻量 route runtime。

导出器不能从画布组件树反推项目路由。

## 风险与缓解

1. 风险：运行时字段过多导致学习负担。
   - 缓解：基础/高级分层显示，默认只暴露 path、page、layout、outlet。
2. 风险：跨框架语义不一致。
   - 缓解：先定义框架无关 IR，再由 target adapter 降级或扩展。
3. 风险：code-owned 能力绕过符号环境。
   - 缓解：所有 runtime refs 必须是 `CodeReference`，由 Workspace Semantic Index 结合 Code Semantic Provider 提供补全和诊断。

## 验收标准

- [ ] 路由可声明并执行 loader / guard / action。
- [ ] 错误边界可按 route chain 层级兜底。
- [ ] SEO 字段可从 route runtime 输出到构建产物。
- [ ] `PdxRoute`、`PdxOutlet`、navigate action 使用同一个 `RouteRuntimeContext`。
