## 决策背景

Prodivix 需要支持组件库、动画库、图表库、代码编辑器、节点图引擎、导入导出器、AI 工具和社区扩展。若每接一个外部库都把库专属逻辑写进 core，会带来长期问题：

1. core 逐渐变成外部库兼容层集合，维护边界失控。
2. 不同库的运行时、渲染、代码生成、诊断和权限模型散落在业务代码里。
3. 插件系统上线后，已有内置外部库支持难以迁移，也难以交给社区扩展。
4. 动画编辑器如果直接内置 GSAP、react-spring 等库语义，会污染 PIR 的稳定动画事实源。

VS Code 有大量 extension points，是因为它把编辑器、语言服务、调试、菜单、主题、任务、终端、Notebook、文件系统等能力都拆成了可贡献面。Prodivix 不应在早期追求数量，而应先定义少量稳定的大类扩展点，让后续能力可以按同一模型继续细分。

## 决策目标

1. 明确 Prodivix core 只保留宿主协议、通用运行时和稳定扩展点，不长期内置具体外部库适配。
2. 让外部组件库、动画库、代码库、图表库和复杂领域库都通过插件贡献能力。
3. 保持 PIR、Workspace VFS、Code Authoring Environment、Diagnostics 和 Command/Intent 的边界稳定。
4. 支持官方插件和社区插件使用同一扩展机制。
5. 为现有外部库支持提供迁移方向：从 core 库专属实现迁移到插件。

## 核心决策

采用 **Extension Host + Contribution Points + Capability Sandbox** 架构。

```text
Plugin Manifest
  -> Contribution Points
  -> Capability Sandbox
  -> Host Gateway
  -> Workspace / PIR / Code Authoring / Renderer / Codegen / Diagnostics
```

规则：

1. core 只认识扩展点、能力权限、稳定 IR、诊断、命令网关和宿主 API。
2. core 不长期认识具体外部库，例如 Ant Design、Radix、GSAP、react-spring、Three.js、React Flow、CodeMirror。
3. 具体库支持应进入官方插件或社区插件，例如 `@prodivix/plugin-antd`、`@prodivix/plugin-gsap`。
4. 插件不得直接修改 workspace store、PIR graph、Animation 文档或 CodeArtifact；写入必须走 Command / Intent / Patch。
5. 插件不得绕过 Code Authoring Environment 保存任意裸代码字符串。
6. 插件扩展的诊断、Inspector、Palette、Render Policy、Codegen Policy 都必须能被宿主统一治理和降级。

## Core 与插件边界

### Core 保留

core 可以保留以下能力：

1. Extension Host 与插件生命周期。
2. Plugin Manifest 解析、版本兼容检查和签名校验。
3. Capability-based Sandbox 与权限授权。
4. Command / Intent / Patch 网关。
5. Workspace VFS、PIR、NodeGraph、Animation 的稳定文档模型。
6. Code Authoring Environment、CodeSlot、CodeReference、Authoring Symbol Environment。
7. Canonical External IR 的核心结构。
8. Render Policy、Codegen Policy、Inspector Contribution 等接口定义。
9. Diagnostics 注册、展示、定位和审计。
10. Native 内置能力，例如 Prodivix 自己的基础组件、基础 timeline、CSS filter、SVG filter 和稳定预览管线。

### 插件承载

插件应承载以下能力：

1. 外部组件库的组件 manifest、props schema、slot schema、分组、图标、预设。
2. 外部库的 Render Policy、Codegen Policy、Canvas 安全渲染策略。
3. 外部动画库的 runtime provider、track provider、easing provider、timeline script、preview adapter。
4. 外部领域库的 adapter，例如图表、地图、3D、富文本、节点图引擎、代码编辑器增强。
5. 插件专属 Inspector 面板、Palette 资产、Resource 面板入口。
6. 插件专属 diagnostics、code slots、symbol provider、completion provider、AI context provider。
7. 导入、导出、迁移和转换逻辑。

## 第一批扩展点

第一阶段不追求细碎扩展点数量，而是定义稳定大类。后续可以在这些大类下继续拆分子扩展点。

| 扩展点                   | 用途                                                                              |
| ------------------------ | --------------------------------------------------------------------------------- |
| `plugin.manifest`        | 声明插件 id、版本、入口、兼容范围、权限和贡献项                                   |
| `capability`             | 声明插件需要的 workspace、document、network、ui、codegen、ai 等权限               |
| `command.intent`         | 注册插件命令命名空间，并通过宿主网关写入 workspace                                |
| `externalLibrary`        | 注册外部库 descriptor、manifest、Canonical External IR 增强和依赖信息             |
| `renderPolicy`           | 注册画布和预览安全渲染策略                                                        |
| `codegenPolicy`          | 注册目标框架代码生成策略                                                          |
| `inspectorContribution`  | 注册 Inspector 字段、分组、面板或动作入口                                         |
| `paletteContribution`    | 注册组件、资源、动画 preset、节点图资产等 palette 项                              |
| `animationExtension`     | 注册动画 runtime、track、easing、preview adapter、timeline script slot            |
| `nodeGraphExtension`     | 注册节点类型、端口 schema、executor 绑定、调试视图                                |
| `codeAuthoringExtension` | 注册 CodeSlotProvider、CodeArtifactProvider、symbol provider、diagnostic provider |
| `diagnosticProvider`     | 注册插件诊断码、诊断生成器和定位信息                                              |
| `workspaceDocumentType`  | 注册新的 workspace document 或资源类型                                            |
| `importExportProvider`   | 注册导入、导出、迁移和格式转换                                                    |
| `aiContextProvider`      | 注册 AI 上下文、工具、patch 约束和解释能力                                        |

## Manifest 草案

```json
{
  "id": "@prodivix/plugin-gsap",
  "displayName": "GSAP Animation",
  "version": "0.1.0",
  "prodivix": {
    "engines": ">=0.1.0 <1.0.0"
  },
  "entry": "dist/index.js",
  "capabilities": [
    "workspace.read",
    "document.read:animation",
    "document.write:animation",
    "workspace.intent.dispatch",
    "code.slot.register",
    "codegen.policy.register",
    "ui.inspector.register"
  ],
  "contributes": {
    "animationExtension": [
      {
        "id": "gsap.timeline",
        "runtime": "gsap",
        "kind": "timeline-runtime"
      }
    ],
    "codegenPolicy": [
      {
        "id": "gsap.react.codegen",
        "target": "react",
        "runtime": "gsap"
      }
    ]
  }
}
```

Manifest 只声明贡献项和权限，不保存用户项目状态。用户项目状态应保存到 Workspace VFS、PIR、Animation 文档或插件配置文档中，并受 workspace revision 和 command history 管理。

## 外部库扩展模型

外部库统一通过 `externalLibrary` 扩展点接入。

```text
Plugin
  -> LibraryDescriptor
  -> Canonical External IR
  -> Palette / Inspector / Renderer / Codegen
```

规则：

1. core 可以提供通用 loader、export scanner、Canonical External IR 和兜底渲染。
2. 库专属 manifest、props 映射、slot 语义、render wrapper、codegen hints 不应长期留在 core。
3. 复杂组件的画布安全模式由插件的 Render Policy 提供。
4. 生产代码生成由插件的 Codegen Policy 提供。
5. 外部库诊断仍走统一 Diagnostics，不允许插件自建孤立错误系统。

官方外部库插件示例：

```text
@prodivix/plugin-antd
@prodivix/plugin-radix
@prodivix/plugin-headless-ui
@prodivix/plugin-three
@prodivix/plugin-react-flow
@prodivix/plugin-codemirror
```

## 动画扩展模型

Animation core 只保留 Prodivix native 动画语义，例如 timeline、track、binding、CSS filter、SVG filter、关键帧和基础预览。GSAP、react-spring、Motion 等外部动画库通过 `animationExtension` 接入。

可贡献项：

1. `animationRuntimeProvider`：声明运行时能力，例如 GSAP timeline、react-spring hook。
2. `animationTrackProvider`：声明自定义 track 类型和可编辑字段。
3. `easingProvider`：声明 easing preset 或 easing function。
4. `animationPreviewAdapter`：声明编辑态预览如何安全执行。
5. `animationCodegenPolicy`：声明目标框架代码生成方式。
6. `animationInspectorContribution`：声明 Inspector 字段和动作。
7. `animationCodeSlotProvider`：声明 timeline script、easing function、shader、runtime adapter 等 code slot。
8. `animationDiagnosticProvider`：声明签名、引用、运行时能力和兼容性诊断。

规则：

1. PIR / Animation 文档不保存 GSAP timeline 实例、react-spring hook 内部状态或库专属私有对象。
2. 复杂动画逻辑进入 Code Authoring Environment，通过 CodeReference 绑定。
3. Animation 文档只保存结构化引用、插件能力 id、目标节点、绑定关系和可迁移配置。
4. 代码生成时由插件读取结构化配置和 CodeReference，输出目标框架代码。

示例保存态方向：

```ts
type PluginAnimationBinding = {
  kind: 'plugin-animation';
  pluginId: string;
  capabilityId: string;
  targetNodeId: string;
  config?: Record<string, unknown>;
  reference?: CodeReference;
};
```

该类型是方向示例，不在本 ADR 中冻结 PIR schema。

## Code Authoring 扩展模型

任何 code-owned 能力都必须通过 Code Authoring Environment 接入。插件可以贡献：

1. `CodeSlotProvider`：告诉宿主哪里可以接代码能力。
2. `CodeArtifactProvider`：提供插件管理的代码 artifact 投影。
3. `CodeSymbolProvider`：提供符号、补全、定义跳转和引用查询。
4. `AuthoringDiagnosticProvider`：提供签名、引用、能力、运行时和兼容性诊断。
5. `AIContextProvider`：提供 AI 可理解的上下文和 patch 约束。

规则：

1. 插件不得把任意代码字符串保存到三编辑器局部状态。
2. CodeReference 以 artifact identity 为核心，不以 path 或临时 name 为主键。
3. owner 删除后，插件创建的 code artifact 默认进入 orphan 流程，不自动删除。
4. 诊断应能定位到 Code Editor、Inspector、Issues 或相关画布节点。

## 权限模型补充

在 ADR 14 的 capability 基础上，插件扩展点需要补充更细的能力域：

```text
extension.register:<point>
code.slot.register
codegen.policy.register
render.policy.register
diagnostic.provider.register
inspector.register
palette.register
ai.context.register
document.read:<type-or-scope>
document.write:<type-or-scope>
network.request:<allowlist-id>
```

规则：

1. 注册扩展点也需要权限，不是所有插件都能贡献所有能力。
2. 高风险能力，例如 `document.write:*`、`network.request:*`、`codegen.policy.register`，需要安装时或管理员策略授权。
3. 插件执行命令、网络请求、代码生成和 AI 工具调用都应进入审计日志。

## 迁移策略

现有 core 中已经存在或将要出现的外部库专属支持，统一按以下方向迁移：

1. 通用能力留在 core：loader、scanner、Canonical IR、diagnostic bridge、policy interface。
2. 库专属能力迁到插件：manifest、props schema、render wrapper、codegen hints、特殊上下文 provider。
3. 先标记为 official plugin candidate，再在插件宿主稳定后迁出 core。
4. 迁移期可以保留兼容 shim，但 shim 不应成为新的长期 API。
5. 新增外部库支持默认不得直接写进 core，除非它是 Prodivix native 能力的一部分。

## 当前实现状态（2026-06-11）

本 ADR 定义的是长期插件边界，不表示当前代码已经完成插件化迁移。

当前 MUI / Ant Design 等外部库的 adapter、manifest、profile 仍以内置实现保留在 Blueprint external libraries 相关模块中。这是插件宿主稳定前的过渡状态，不应被解释为 ADR 29 已经落地完成。

迁移顺序以插件宿主、contribution registry、capability sandbox 和宿主 API 稳定为前置条件。插件系统成熟前，已有 core-embedded 外部库支持可以继续维护，但新增能力应尽量按 future official plugin candidate 的边界组织，避免扩大长期迁移成本。

候选迁移对象：

```text
Ant Design adapter / policy
Radix adapter / policy
GSAP runtime / policy
react-spring runtime / policy
Three.js surface / policy
React Flow node shell / policy
CodeMirror extension / policy
Monaco extension / policy
```

## 非目标

1. 本 ADR 不冻结具体 TypeScript API。
2. 本 ADR 不要求一次性实现完整插件市场。
3. 本 ADR 不要求立即迁移所有现有外部库实现。
4. 本 ADR 不定义所有未来 extension points 的完整清单。
5. 本 ADR 不改变 PIR v1.3 当前 schema。
6. 本 ADR 不承诺任意外部库都能完整可视化编辑内部状态。
7. 本 ADR 不让插件绕过 Workspace VFS、Command/Intent 或 Code Authoring Environment。

## 分阶段落地

### Phase 1：扩展点契约

1. 定义 plugin manifest 草案。
2. 定义第一批 contribution points。
3. 定义扩展点注册权限和审计要求。
4. 更新外部库 ADR，明确库专属支持应迁移到插件。

### Phase 2：宿主与注册表

1. 实现 Extension Host 基础生命周期。
2. 实现 contribution registry。
3. 接入 capability sandbox 和 permission summary。
4. 让 Inspector、Palette、Diagnostics、Codegen 能消费插件贡献。

### Phase 3：官方插件迁移

1. 将现有外部组件库专属支持迁移为官方插件。
2. 将动画库支持以 GSAP 或 react-spring 作为标杆插件试点。
3. 将复杂领域库按 Embedded / Adapted / Code-owned 分层迁移。

### Phase 4：生态化

1. 提供插件 SDK 和模板。
2. 提供官方插件开发文档。
3. 建立插件签名、审核、版本兼容和市场分发机制。

## 长期不变量

1. Prodivix core 认识扩展点，不认识每个外部库。
2. 外部库支持优先由插件贡献。
3. 插件写入 workspace 必须走 Command / Intent / Patch。
4. code-owned 能力必须进入 Code Authoring Environment。
5. PIR 与 Animation 文档保存结构化引用，不保存复杂库私有运行时状态。
6. 插件诊断统一进入 Diagnostics。
7. 官方插件与社区插件使用同一宿主协议。
8. core 内的库专属 shim 只能作为迁移期兼容层。

## 验收标准（Draft）

- [ ] 第一批扩展点在文档中有稳定命名和职责边界。
- [ ] 外部库 ADR 明确 core 与插件的职责分工。
- [ ] 新增外部库支持默认可被归类到某个 contribution point。
- [ ] Animation 对 GSAP、react-spring 等库的支持路径不要求修改 core schema。
- [ ] 插件写入路径能被 Command / Intent / Patch 审计。
- [ ] 插件 code-owned 能力能通过 Code Authoring Environment 定位、诊断和生成代码。
- [ ] core 中的库专属实现有 official plugin candidate 迁移路径。
