## 背景

同一个 Scenario 在 Preview、Export 和 CI 重复执行，只有在时间、随机数、网络、数据、浏览器状态、动画偏好和
并发交错受控时才可比较。直接录制真实毫秒延迟、HTTP response、DOM event 或 mutation result，不但不可重复，
还可能重放 credential 和副作用。

G3 的 deterministic replay 不是浏览器录像回放，也不是把 Runtime state 持久化为 Workspace。它是用相同的
Scenario Program、fixture 和 control schedule 创建全新隔离运行，并比较规范化 observation。

## 决策

### Control profile

每个 Scenario Program 必须绑定完整、可摘要的 `BehaviorControlProfile`：

- virtual clock、timezone、locale；
- random seed、id factory seed；
- scheduler/interleaving plan；
- network mode、fixture set、fault schedule；
- viewport、device pixel ratio、color scheme；
- motion preference 与 animation clock；
- storage/cookie/session bootstrap；
- font/asset readiness policy；
- service worker/cache policy；
- total、step、settle、network 和 animation budget。

省略字段由 canonical default 补齐并进入 digest。adapter 的宿主默认值、当前系统时间、机器 locale 或用户浏览器
偏好不能隐式影响 required evidence。

### Clock、random 与 scheduler

Scenario runtime 通过 port 注入 monotonic virtual clock、wall-clock projection、timer queue、random 和 id factory。
领域代码在 verification profile 中使用这些 port；无法控制的 direct clock/random access 形成 diagnostic 或降低为
non-deterministic advisory evidence，不能满足 required deterministic closure。

等待使用 typed condition + bounded deadline：route ready、target visible、Data lifecycle、animation marker、network idle、
barrier。禁止持久化任意 `sleep(2000)` 作为稳定同步语义。用户录制的停顿只用于 draft suggestion。

并发由 lane、barrier 和 deterministic scheduler 表达。每个 runnable task 的选择顺序进入 replay schedule；用于验证
optimistic mutation、retry、cancel、route transition 和 stale response。真实线程时序可以作为 advisory stress run，不能
替代确定性 interleaving matrix。

### Network 与 Data

required G3 Scenario 默认 `fixture-only`：

- 未声明 request 一律拒绝，不回退 live；
- fixture 以 DataOperation/Server Function identity、canonical input digest、attempt 和 page/cursor 匹配；
- response、delay、disconnect、timeout、retry-after、stream frame 和 out-of-order delivery 由 fault schedule 控制；
- mutation 使用 per-run namespace、effect-before-replay fence 和 cleanup/rollback；
- Secret/environment resolver 在 test/fixture mode 不可用。

`isolated-live-read` 可产生明确标记的 advisory Evidence；production probe、真实用户数据和部署后 observation 属于 G5。
跨 execution durable stream replay 继续遵守 Data ADR 边界，不因 G3 Scenario 自动开放。

Network observation 只保留 G2 sanitized metadata、operation correlation 和 SourceTrace。header、query、body、cookie、
signed URL 和 credential 禁止进入 control log/evidence。

### Browser state 与 isolation

每个 attempt 使用新的 browser context、fixture namespace、storage partition 和 execution owner。可声明的初始状态通过
typed fixture/ref 注入；不得从作者当前浏览器复制 localStorage、cookie、IndexedDB 或 service worker cache。

Preview host 可以复用 immutable dependency/build cache，但不能复用 page、active process state、runtime store、network
interceptor 或 timer queue。Export/CI adapter遵守同一 isolation conformance。

### Motion、render 与 settle

motion axis 至少支持 `full` 与 `reduced`。override 只存在于 verification context，并同时投影标准
`prefers-reduced-motion` 与 Prodivix Animation policy；应用看见的两者必须一致。

Visual checkpoint 只在以下条件全部满足后 capture：

1. Scenario 指定 typed checkpoint；
2. required target/route ready；
3. font、asset 和 layout readiness 满足；
4. pending controlled network/timer/animation 状态符合 checkpoint policy；
5. viewport/DPR/color/motion 与 baseline compatibility key 一致。

“network idle”不能等待未知 background stream 永久结束；stream、infinite animation 和 polling 必须通过显式
checkpoint condition 或 freeze policy。

### Replay record

每次 run 产生 bounded `BehaviorReplayRecord`：

- Scenario/Program/control/fixture/snapshot digest；
- ordered step transition、barrier 和 deterministic scheduler decision；
- normalized observation/result digest；
- domain trace correlation 和 SourceTrace；
- fault injection、retry、cancel 和 cleanup outcome；
- non-determinism flags。

Record 不保存 raw DOM tree、screenshot bytes、network payload 或 runtime closure；这些只作为受控 artifact reference。
事件数、深度、字符串和总字节有硬预算。超限 run 为 infrastructure-error，不截断后宣称 passed。

### Replay semantics

Replay 创建新的 attempt：

1. 重新 materialize exact snapshot/bundle；
2. 新建隔离 browser/runtime/fixture namespace；
3. 使用相同 Scenario Program 和 control schedule；
4. 重新执行 effect，不复用旧 mutation response 或 runtime state；
5. 比较 normalized step/assertion/trace digest；
6. 差异形成 deterministic divergence diagnostic。

旧 operation、mutation 或 Server Function effect 不自动重放。若 fixture 无法重新创建，Replay 标记 blocked，而不是
从旧 Evidence 注入结果。

### Debugger

交互式 breakpoint、step、continue、pause、seek 和 variable inspection 是同一 deterministic scheduler 的控制面。
debug pause 不修改 Scenario Program 或 Evidence；从 paused run 继续形成同一 attempt，若用户改变 input/control 则创建
新的 exploratory attempt，不能满足原 Plan。

Debugger state、breakpoints、selected frame 和 variable expansion 是可丢弃 UI/runtime state。若用户要保存 breakpoint
或新增 assertion，必须通过 Scenario/NodeGraph Command。

## Provider conformance

Browser、Remote、Export 与 CI adapter 必须证明：

- 支持/拒绝相同 control capability；
- exact Program + controls + fixtures 得到相同 semantic outcomes；
- unsupported control 在执行前 fail closed；
- cancellation/dispose 清理 timer、network interception、context、fixture 和 effect lease；
- trace/replay record 没有 Secret；
- instrumentation 不进入 production profile 或获得生产 write authority。

视觉像素允许按 compatibility key 使用不同 baseline，语义 step/assertion outcome 不允许因 adapter 私有等待规则漂移。

## 拒绝的方案

### 录制并重放真实 HTTP

拒绝。它不可重复，可能包含 credential/个人数据，也可能重复副作用。

### 以固定 sleep 解决 flaky

拒绝。等待必须绑定 typed condition 和 deadline。

### 将 browser storage/runtime snapshot 写入 Workspace

拒绝。运行态不是作者态，且可能包含 Secret。

### 只比较最终截图

拒绝。截图无法证明 Route/Data/NodeGraph/Animation 中间语义，也不能定位 divergence。

## 后果

- Runtime/Compiler 需要 verification-only control ports 和 direct nondeterminism boundary checks。
- Data/Auth/Server test adapters需要共享 fixture/fault contract。
- Scenario runner与 debugger共享 scheduler/replay kernel，不再各自实现等待和单步。
- Evidence 明确记录 non-determinism，不把 exploratory/live run 冒充 closure。

## 验收

- [ ] clock/random/id/network/motion/storage/viewport controls 全部进入 Program digest。
- [ ] condition wait、lane/barrier 和 scheduler interleaving 可重复且有硬预算。
- [ ] fixture-only 未声明网络、Secret 和旧 mutation replay 多层 fail closed。
- [ ] replay 使用新隔离 attempt，能检测并定位 deterministic divergence。
- [ ] Browser/Remote/Export/CI 通过 control/isolation/cancel conformance。
- [ ] debugger state 不进入 Workspace，探索性修改不满足原 Plan。
