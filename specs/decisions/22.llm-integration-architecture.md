## 当前实现状态（2026-06-11）

本 ADR 记录 LLM 集成的目标架构，不表示当前产品中的 LLM 能力已经可用。

当前代码库已经存在部分 foundation：Gateway、Context Builder、Tool Registry、Trace Store、Mock Provider、OpenAI-compatible provider 和结构化输出校验基础。但真实 LLM 运行链路、工具执行闭环、repair loop、可用 UI 工作流、权限治理和验收测试仍未达到产品可用状态。

因此，本 ADR 保持 Draft。后续若要升为 Accepted，需要先确认真实 provider 配置、任务执行、dry-run/apply、trace/replay、错误修复和用户确认链路已经形成可验证闭环。

## 决策背景

Prodivix 的定位不是传统低代码工具，也不是单纯代码生成器。README 中已经明确：

- MFE 使用 PIR 作为唯一真相源。
- 蓝图编辑器、节点图编辑器、动画编辑器三层统一于 PIR。
- 用户可以像搭积木一样开发前端，也可以在任意环节混入真实代码。
- 复杂逻辑可以直接写真实 TypeScript。
- AI 助手需要深度集成，用于生成节点、补全逻辑、优化 PIR。

因此，MFE 接入 LLM 不应退化成“聊天框 + 文件改写”，也不应把所有能力强行压成 PIR patch。当前主流 LLM 工具链已经形成若干稳定能力：

1. 结构化输出与 tool calling。
2. Agent workflow 与多步任务执行。
3. MCP 等外部工具 / 资源协议。
4. 代码 agent 对真实工程文件、测试、类型检查的调用能力。
5. RAG / context retrieval。
6. trace / eval / replay。

MFE 的 LLM 架构应主动吸收这些工具链能力，同时保持 PIR、节点图、真实代码三者的统一。

## 决策目标

1. 让 LLM 成为 MFE 的受约束开发协作者，而不是不受控的文本生成器。
2. 同时支持可视化编辑、节点图逻辑编辑、真实代码生成。
3. 所有写入操作都必须可验证、可 dry-run、可撤销。
4. 对内提供 MFE 语义级 tools，对外提供 MCP 接入能力。
5. 支持大意图分解为 plan / task / command / code artifact。
6. 让 LLM 的代码能力进入真实工程链路：类型检查、测试、编译、预览。

## 核心决策

### 1) LLM 输出分为三种一等通道

MFE 不采用单一输出通道。LLM 的结果必须归入以下三类之一：

```text
Intent -> PIR commands
Intent -> Node graph operations
Intent -> Code artifacts
```

#### PIR Commands

适用于：

- 组件树修改
- 布局、样式、主题 token 应用
- 数据绑定
- 触发器配置
- 动画挂载
- 路由和 Outlet 绑定

要求：

- 不直接要求 LLM 输出整份 PIR。
- 优先输出 intent command 或受限 PIR patch。
- 每个 command 必须有 schema。
- 每个 command 必须支持 dry-run。
- 可逆操作应进入 command history。

#### Node Graph Operations

适用于：

- 数据流
- 控制流
- 条件分支
- 节点图连线
- 内置节点组合
- 自定义代码节点挂载

要求：

- LLM 只在当前节点图能力范围内生成操作。
- port 类型、control-flow edge、data-flow edge 必须由系统验证。
- 不允许 LLM 编造不可用节点类型或端口。

#### Code Artifacts

适用于：

- 自定义 JS/TS 代码节点
- 外部执行代码
- adapter / props mapping
- shader / worker / utility code
- 单元测试、Playwright 流程、视觉回归用例

要求：

- 代码生成是 MFE 的正式能力，不是 fallback。
- 代码必须进入真实工程链路。
- 写入后必须经过 format、typecheck、test 或 compiler dry-run。
- 代码 artifact 应与 PIR / Node Graph 有明确绑定关系。

### 2) MFE 内部建立 LLM Gateway

前端功能不直接散落调用模型 API。所有 LLM 请求经过统一 Gateway。

```text
MFE UI
  -> LLM Gateway
    -> Provider Adapter
    -> Context Builder
    -> Tool Registry
    -> Structured Output Validator
    -> Repair Loop
    -> Trace / Eval Store
```

LLM Gateway 职责：

- 统一模型供应商。
- 管理系统提示词和 tool schema。
- 控制上下文预算。
- 构造最小上下文。
- 验证结构化输出。
- 接收 validator 错误并触发 repair loop。
- 记录 trace，便于复现和评估。

### 3) Context Builder 必须最小化上下文

不要把整个项目直接塞给模型。每次请求根据任务构造最小上下文。

可选上下文：

- 当前选中节点。
- 当前 subtree。
- 当前路由。
- 当前 data model 摘要。
- 当前 Node Graph 子图。
- 当前动画 timeline 摘要。
- 当前主题 token。
- 可用组件和 capability schema。
- 相关代码 artifact 摘要。
- 最近 diagnostics。
- 本次允许调用的 tools。

原则：

- 默认不给全量 PIR。
- 默认不给全仓库文件。
- 大意图先生成 plan，不直接执行。
- 上下文必须标明哪些信息是权威状态，哪些只是摘要。

### 4) Tool Registry 提供 MFE 语义级工具

LLM 使用的工具不应是裸文件读写为主，而应优先是 MFE 语义工具。

首批内部 tools：

```text
readSelectedNode
readPirSubtree
readNodeCapabilities
listAvailableComponents
listThemeTokens
listDataModelPaths
listNodeGraphCapabilities
dryRunPirCommands
applyPirCommands
dryRunNodeGraphOperations
applyNodeGraphOperations
createCodeArtifact
updateCodeArtifact
bindCodeArtifactToNode
runTypecheck
runUnitTests
runCompilerDryRun
runVisualRegression
explainDiagnostics
```

写操作规则：

- 任何写操作必须有 dry-run 对应工具。
- 默认先 dry-run，再 apply。
- 高风险操作必须要求用户确认。
- apply 后必须进入 undo/redo 或 revision 记录。

高风险操作包括：

- 删除大量节点。
- 替换页面结构。
- 修改全局主题。
- 引入外部依赖。
- 写入外部执行代码。
- 修改路由根结构。
- 批量重排节点图。

### 5) 对外提供 MFE MCP Server

MFE 应作为 MCP Server 暴露给外部 LLM 工具，使 Claude、Cursor、Codex、VSCode 插件等工具能够理解 MFE 项目。

MCP resources 示例：

```text
pir://project/current
pir://node/{id}
pir://subtree/{id}
schema://pir/current
schema://component/{type}
schema://command/{type}
theme://tokens
node-graph://graph/{id}
code-artifact://artifact/{id}
compiler://diagnostics
```

MCP tools 示例：

```text
dryRunCommands
applyCommands
inspectNodeCapabilities
dryRunNodeGraphOperations
applyNodeGraphOperations
createCodeArtifact
runCompiler
runTests
runVisualRegression
```

约束：

- MCP Server 不暴露无限制文件系统写权限。
- MCP 写操作仍走 MFE validator、capability、dry-run、history。
- MCP Server 复用内部 Tool Registry，不维护第二套规则。

### 6) 大意图必须进入计划模式

当用户意图较大，例如“做一个完整电商首页”或“重构整个页面视觉风格”，LLM 不允许直接生成一大批改动。

流程：

```text
Big Intent
  -> Plan
  -> Milestones
  -> Tasks
  -> Small command / code batches
  -> Dry-run
  -> Preview
  -> Apply
  -> Test
```

Plan artifact 示例：

```json
{
  "goal": "Build ecommerce homepage",
  "assumptions": ["Use existing product card component"],
  "milestones": [
    { "id": "structure", "title": "Create page structure" },
    { "id": "hero", "title": "Build hero section" },
    { "id": "products", "title": "Add product grid" },
    { "id": "responsive", "title": "Tune responsive layout" },
    { "id": "tests", "title": "Generate tests" }
  ]
}
```

每个执行批次应限制：

- 修改节点数量。
- 新增节点数量。
- 跨编辑器范围。
- 外部依赖写入。
- 是否需要用户确认。

### 7) Repair Loop 是可靠性交互的基础

LLM 面对 MFE 私有协议时不能依赖“一次生成就完全正确”。系统必须支持修复闭环。

```text
generate
  -> schema validate
  -> capability validate
  -> dry-run
  -> diagnostics
  -> repair
  -> dry-run
  -> apply
```

诊断信息必须结构化：

```json
{
  "code": "UNKNOWN_THEME_TOKEN",
  "path": "commands[2].style.color",
  "message": "Theme token does not exist.",
  "allowedValues": ["--text-primary", "--text-muted", "--danger-color"]
}
```

要求：

- error code 稳定。
- path 指向具体字段。
- 尽量给出 allowed values 或 repair hints。
- repair loop 有次数上限。
- 多次失败后返回用户可理解的解释。

### 8) 代码生成必须被工程链路接住

MFE 不排斥 LLM 写代码。相反，真实代码是 MFE 融合式开发模型的一等能力。

代码生成流程：

```text
generate code
  -> write artifact
  -> format
  -> typecheck
  -> unit test
  -> compiler dry-run
  -> bind to PIR / Node Graph
```

代码 artifact 类型：

- `node-code`
- `external-runtime-code`
- `adapter-code`
- `test-code`
- `shader-code`
- `worker-code`
- `utility-code`

约束：

- 代码 artifact 必须有 owner 或绑定目标。
- 外部依赖必须经过 license / security 检查。
- 执行型代码必须经过 sandbox / capability 约束。
- 失败 diagnostics 可以回给 LLM 修复。

### 9) 测试生成是一等能力

LLM 不只用于生成业务功能，也用于生成和维护测试。

测试生成输入：

- PIR subtree。
- 节点能力。
- 路由信息。
- 数据模型。
- 用户交互路径。
- 视觉关键区域。
- Node Graph 行为。
- 动画 timeline。

测试类型：

- Vitest 单元测试。
- Playwright 交互流程。
- 视觉回归测试。
- Node Graph 数据流测试。
- 动画关键帧测试。
- 编译输出 smoke test。

规则：

- LLM 负责提出测试和生成草案。
- runner 负责判断通过与否。
- 测试失败 diagnostics 可进入 repair loop。

### 10) Trace 与 Eval 必须内建

LLM 功能上线后，如果没有 trace 和 eval，可靠性无法迭代。

每次 LLM 任务记录：

- user intent。
- context 摘要。
- model。
- tools schema version。
- generated artifacts。
- validation errors。
- repair attempts。
- dry-run diff。
- final apply result。
- test result。

Eval 用例来源：

- 固定 fixtures。
- 用户常见任务。
- 失败任务回放。
- 社区主题 / 外部组件 / 节点图复杂案例。

## 直接影响

### PIR Command Protocol

现有 intent command 需要继续扩展：

- 增加 LLM-safe schema。
- 增加 dry-run result。
- 增加 structured diagnostics。
- 增加 risk level。
- 增加 undo metadata。

### Capability Registry

组件、节点图、动画、代码 artifact 都需要暴露给 LLM 的 capability 摘要。

Capability 必须回答：

- 当前对象支持什么。
- 哪些字段可编辑。
- 哪些 command 可用。
- 哪些值合法。
- 哪些操作高风险。

### Workspace / VFS

代码 artifact 需要与 workspace 文件体系打通，但 LLM 不应默认获得裸写权限。

### Security

LLM 可能生成执行型代码，因此必须沿用插件 sandbox、依赖 license、外部库安全策略。

## 取舍说明

### 为什么不是简单聊天框

聊天框只能解释和生成文本，无法保证状态一致、可撤销、可验证。MFE 的核心价值是可视化和代码共同收敛到 PIR，LLM 必须进入这个工程系统。

### 为什么不是只生成 PIR

MFE 本来支持真实代码。复杂业务逻辑、shader、worker、adapter、测试让 LLM 直接写代码更合适。关键不是禁止写代码，而是让代码被 MFE 的验证链路接住。

### 为什么需要 MCP

MFE 不应只服务内置 AI 助手。通过 MCP，外部 LLM 工具可以读取 PIR、检查 schema、调用 dry-run、运行测试，从而让 MFE 项目进入主流 AI coding 工具链。

### 为什么需要 repair loop

LLM 面对私有协议时不可能长期一次成功。可靠性来自小动作空间、结构化错误、dry-run 和修复闭环，而不是相信模型“足够聪明”。

## 非目标

1. 本决策不要求立刻实现完整 agent 平台。
2. 本决策不指定唯一模型供应商。
3. 本决策不要求放弃人工编辑。
4. 本决策不允许绕过 PIR / workspace / validator 直接修改核心状态。
5. 本决策不定义具体 UI 细节，例如聊天面板布局。

## 验收标准（Draft）

- [ ] 存在统一 LLM Gateway 设计，不在业务组件中散落模型调用。
- [ ] LLM 输出至少区分 PIR commands、Node Graph operations、Code artifacts。
- [ ] 写操作具备 dry-run 和 apply 两阶段。
- [ ] validator diagnostics 可被 LLM repair loop 消费。
- [ ] Code artifact 生成进入 format / typecheck / test / compiler dry-run 链路。
- [ ] 大意图先生成 plan artifact，再分批执行。
- [ ] MFE MCP Server 有明确 resources / tools 边界。
- [ ] trace / eval 数据模型可以复现 LLM 任务。
