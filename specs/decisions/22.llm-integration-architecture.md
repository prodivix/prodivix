## 当前实现状态（2026-06-29）

本 ADR 记录 LLM 集成的目标架构，不表示当前产品中的 LLM 能力已经可用。

当前代码库已经存在 foundation：Gateway、Context Builder、Tool Registry、Trace Store、Mock Provider、OpenAI-compatible provider、streaming runtime 和结构化输出校验基础。Web 侧已有 Blueprint assistant，但它主要是 plan-only，且上下文仍偏当前 route / selected node。

当前最大的架构缺口不是“模型还不够强”，而是 **AI action 的作用域过窄**：

1. 现有协议主要表达 `pir-command`、`node-graph-operation`、`code-artifact`。
2. 这会让 AI 看起来只能修改编辑器内部内容。
3. 真实产品里 AI 还必须能改 workspace 级对象：路由、页面/布局文档、资源、i18n、外部依赖、项目设置、导出配置、诊断修复计划、测试与发布配置等。
4. 这些对象不能靠聊天文本或裸文件改写完成，必须进入受约束的 action / command / validator 链路。

因此，本 ADR 保持 Draft。后续若要升为 Accepted，需要先确认真实 provider 配置、workspace action registry、dry-run/apply、trace/replay、错误修复和用户确认链路已经形成可验证闭环。

## 决策背景

Prodivix 的定位不是传统低代码工具，也不是单纯代码生成器。Prodivix 使用 Workspace VFS、PIR、Code Authoring Environment、NodeGraph、Animation、Resources、Export Planner 和 RouteGraph 共同表达一个前端项目。

AI 助手不应退化成“聊天框 + 文件改写”，也不应把所有能力强行压成 PIR patch。它应该是受约束的 workspace 协作者：能理解当前 UI 触发点，也能修改触发点背后的 workspace 对象。

这意味着：

1. 画布节点、Inspector、Class Protocol、NodeGraph 和 Animation 是重要入口。
2. RouteGraph、Workspace documents、Assets、Settings、Export、Diagnostics、Tests 也是一等 target。
3. UI 触发点只决定“用户此刻从哪里发起”，不决定“AI 只能改这个局部 DOM 或 PIR 节点”。
4. 每个 target 都必须声明能力、风险、dry-run、apply 和 validator。

## 决策目标

1. 让 LLM 成为受约束的 workspace 开发协作者，而不是不受控的文本生成器。
2. 同时支持可视化编辑、路由编辑、资源管理、节点图逻辑、动画、真实代码、测试和导出配置。
3. 所有写入操作都必须可验证、可 dry-run、可撤销或可追踪。
4. 对内提供 Prodivix 语义级 tools，对外提供 MCP 接入能力。
5. 支持大意图分解为 plan / milestone / task / action proposal / command batch。
6. 让 AI 的代码能力进入真实工程链路：format、typecheck、test、compiler dry-run、preview。

## 核心决策

### 1) AI 输出收敛为 Workspace Action Proposal

旧的三通道模型：

```text
Intent -> PIR commands
Intent -> Node graph operations
Intent -> Code artifacts
```

只能覆盖编辑器局部修改。新的目标模型是：

```text
Intent
  -> Plan
  -> Workspace Action Proposal
  -> Dry-run
  -> Review
  -> Apply through command / intent / code authoring path
```

`WorkspaceActionProposal` 是 AI 写操作的统一外壳。它必须声明 target、scope、operation、risk、required validators 和 preview kind。

```ts
type AiActionTargetKind =
  | 'workspace'
  | 'route'
  | 'document'
  | 'pir-node'
  | 'code-artifact'
  | 'node-graph'
  | 'animation'
  | 'asset'
  | 'i18n'
  | 'external-library'
  | 'theme'
  | 'settings'
  | 'export'
  | 'test'
  | 'diagnostic';

type WorkspaceActionProposal = {
  id: string;
  target: {
    kind: AiActionTargetKind;
    id?: string;
    documentId?: string;
    routeNodeId?: string;
  };
  operation: string;
  payload: unknown;
  riskLevel: LlmRiskLevel;
  requiresConfirmation?: boolean;
  reason?: string;
};
```

PIR command、NodeGraph operation 和 CodeArtifact 仍是一等能力，但它们只是 Workspace Action Proposal 的具体 operation family。

### 2) AI Action Target 范围

AI 可写目标按 domain 划分：

| Domain              | 示例能力                                     | 写入路径                                   |
| ------------------- | -------------------------------------------- | ------------------------------------------ |
| Blueprint / PIR     | 修改节点文本、props、layout、data binding    | PIR command / document patch               |
| RouteGraph          | 创建页面、改 route segment、绑定 Outlet      | route intent / route manifest command      |
| Code Authoring      | 生成 handler、executor、mounted CSS、adapter | CodeArtifact / CodeReference               |
| NodeGraph           | 创建节点、连线、executor 绑定                | NodeGraph operation validator              |
| Animation           | 生成 timeline、keyframe、easing、shader ref  | Animation document operation               |
| Resources           | 添加 public asset、i18n entry、外部库声明    | workspace resource command                 |
| Theme / Settings    | 调整 token、preview 设置、workspace policy   | settings command / theme validator         |
| Export / Deploy     | 修复导出配置、解释 build error、生成 target  | Export Planner config / diagnostics action |
| Tests / Diagnostics | 生成测试、解释 issue、修复诊断               | test artifact / diagnostic repair action   |

AI 不应默认拥有全 workspace 写权限。每个 UI trigger 或 plan task 都必须给出明确 target scope 和 allowed operations。

### 3) Prodivix 内部建立 LLM Gateway

前端功能不直接散落调用模型 API。所有 LLM 请求经过统一 Gateway。

```text
Prodivix UI Trigger
  -> AI Action Registry
  -> Context Builder
  -> LLM Gateway
    -> Provider Adapter
    -> Tool Registry
    -> Structured Output Validator
    -> Repair Loop
    -> Trace / Eval Store
  -> Dry-run / Review / Apply
```

LLM Gateway 职责：

1. 统一模型供应商。
2. 管理系统提示词和 tool schema。
3. 控制上下文预算。
4. 构造最小上下文。
5. 验证结构化输出。
6. 接收 validator 错误并触发 repair loop。
7. 记录 trace，便于复现和评估。

### 4) Context Builder 必须最小化上下文

不要把整个项目直接塞给模型。每次请求根据 action target 构造最小上下文。

可选上下文：

1. 当前 route / matchChain / route node。
2. 当前选中 PIR node 或 subtree。
3. 当前 workspace document 摘要。
4. 当前 NodeGraph 子图。
5. 当前 animation timeline 摘要。
6. 当前资源、i18n、external library 或 export config 摘要。
7. 当前主题 token。
8. 可用组件、action capability 和 validator schema。
9. 相关 CodeArtifact 摘要。
10. 最近 diagnostics。
11. 本次允许调用的 tools。

原则：

1. 默认不给全量 PIR。
2. 默认不给全仓库文件。
3. 大意图先生成 plan，不直接执行。
4. 上下文必须标明哪些信息是权威状态，哪些只是摘要。
5. 上下文必须记录 omitted context，避免模型误以为它看到了全部事实。

### 5) Tool Registry 提供 Prodivix 语义级工具

LLM 使用的工具不应以裸文件读写为主，而应优先是 Prodivix 语义工具。

首批内部 read / inspect tools：

```text
readCurrentRoute
readRouteManifestSummary
readSelectedNode
readPirSubtree
readWorkspaceDocumentSummary
readResourceSummary
readExportConfigSummary
readNodeCapabilities
listAvailableComponents
listThemeTokens
listDataModelPaths
listNodeGraphCapabilities
listAnimationCapabilities
explainDiagnostics
```

首批 dry-run / apply tools：

```text
dryRunWorkspaceActions
applyWorkspaceActions
dryRunPirCommands
applyPirCommands
dryRunRouteIntents
applyRouteIntents
dryRunNodeGraphOperations
applyNodeGraphOperations
dryRunAnimationOperations
applyAnimationOperations
createCodeArtifact
updateCodeArtifact
bindCodeArtifact
runTypecheck
runUnitTests
runCompilerDryRun
runVisualRegression
```

写操作规则：

1. 任何写操作必须有 dry-run 对应工具。
2. 默认先 dry-run，再 apply。
3. 高风险操作必须要求用户确认。
4. apply 后必须进入 undo/redo、workspace revision 或 trace 记录。

高风险操作包括：

1. 删除大量节点或文档。
2. 替换页面结构。
3. 修改全局主题、workspace 设置或导出配置。
4. 引入外部依赖或远程资源。
5. 写入执行型代码。
6. 修改路由根结构。
7. 批量重排节点图、动画轨道或资源树。

### 6) 对外提供 Prodivix MCP Server

Prodivix 应作为 MCP Server 暴露给外部 LLM 工具，使 Claude、Cursor、Codex、VSCode 插件等工具能够理解 Prodivix workspace。

MCP resources 示例：

```text
workspace://current
route://manifest/current
pir://document/{id}
pir://node/{id}
pir://subtree/{id}
schema://pir/current
schema://action/{type}
theme://tokens
node-graph://graph/{id}
animation://timeline/{id}
code-artifact://artifact/{id}
resources://summary
compiler://diagnostics
```

MCP tools 示例：

```text
dryRunWorkspaceActions
applyWorkspaceActions
inspectTargetCapabilities
dryRunRouteIntents
applyRouteIntents
dryRunNodeGraphOperations
applyNodeGraphOperations
createCodeArtifact
runCompiler
runTests
runVisualRegression
```

约束：

1. MCP Server 不暴露无限制文件系统写权限。
2. MCP 写操作仍走 Prodivix validator、capability、dry-run、history。
3. MCP Server 复用内部 Tool Registry，不维护第二套规则。

### 7) 大意图必须进入计划模式

当用户意图较大，例如“做一个完整电商首页”“给项目接登录态”“把文档页导出成 React Vite 项目”，LLM 不允许直接生成一大批改动。

流程：

```text
Big Intent
  -> Plan
  -> Milestones
  -> Tasks
  -> Scoped workspace actions
  -> Dry-run
  -> Preview
  -> Apply
  -> Verify
```

每个执行批次应限制：

1. 修改对象数量。
2. 跨 domain 范围。
3. 外部依赖写入。
4. 是否需要用户确认。
5. 验证命令和 preview 要求。

### 8) Repair Loop 是可靠性交互的基础

LLM 面对 Prodivix 私有协议时不能依赖“一次生成就完全正确”。系统必须支持修复闭环。

```text
generate
  -> schema validate
  -> capability validate
  -> dry-run
  -> diagnostics
  -> repair
  -> dry-run
  -> apply
```

诊断信息必须结构化：

```json
{
  "code": "UNKNOWN_THEME_TOKEN",
  "path": "actions[2].payload.style.color",
  "message": "Theme token does not exist.",
  "allowedValues": ["--text-primary", "--text-muted", "--danger-color"]
}
```

要求：

1. error code 稳定。
2. path 指向具体字段。
3. 尽量给出 allowed values 或 repair hints。
4. repair loop 有次数上限。
5. 多次失败后返回用户可理解的 blocked result。

### 9) 代码生成必须被工程链路接住

Prodivix 不排斥 LLM 写代码。真实代码是融合式开发模型的一等能力。

代码生成流程：

```text
generate code
  -> write artifact
  -> format
  -> typecheck
  -> unit test
  -> compiler dry-run
  -> bind to PIR / RouteGraph / NodeGraph / Animation
```

代码 artifact 类型：

1. `node-code`
2. `external-runtime-code`
3. `adapter-code`
4. `test-code`
5. `shader-code`
6. `worker-code`
7. `utility-code`
8. `route-loader-code`
9. `route-action-code`
10. `route-guard-code`

约束：

1. 代码 artifact 必须有 owner 或绑定目标。
2. 外部依赖必须经过 license / security 检查。
3. 执行型代码必须经过 sandbox / capability 约束。
4. 失败 diagnostics 可以回给 LLM 修复。

### 10) Trace 与 Eval 必须内建

每次 LLM 任务记录：

1. user intent。
2. trigger surface。
3. action target 和 scope。
4. context 摘要与 omitted context。
5. model 和 provider。
6. tools schema version。
7. generated action proposals。
8. validation errors。
9. repair attempts。
10. dry-run diff。
11. final apply result。
12. verification result。

Eval 用例来源：

1. 固定 fixtures。
2. 用户常见任务。
3. 失败任务回放。
4. 社区主题 / 外部组件 / 节点图复杂案例。
5. route / export / resource repair cases。

## 直接影响

### LLM Protocol

现有 `LlmOutputChannel` 需要扩展或被 `WorkspaceActionProposal` 包装。短期可以保留 `pir-command`、`node-graph-operation`、`code-artifact`，但 Web editor 侧必须增加 target-scoped validator，不能把这三类当成 AI 的全部能力边界。

### Workspace Command / Intent

Route、resource、settings、export、test 等写操作应进入 workspace command / intent 系统。AI 不应绕过这些路径直接改 React state 或 VFS 文件。

### Capability Registry

组件、路由、节点图、动画、代码 artifact、资源、导出配置都需要暴露给 AI 的 capability 摘要。

Capability 必须回答：

1. 当前对象支持什么。
2. 哪些字段可编辑。
3. 哪些 command / intent / operation 可用。
4. 哪些值合法。
5. 哪些操作高风险。

### Workspace / VFS

代码 artifact、资源、i18n、external library 和 export config 需要与 workspace 文件体系打通，但 AI 不应默认获得裸写权限。

### Security

LLM 可能生成执行型代码、依赖声明和远程资源配置，因此必须沿用插件 sandbox、依赖 license、外部库安全策略和 workspace policy。

## 取舍说明

### 为什么不是简单聊天框

聊天框只能解释和生成文本，无法保证状态一致、可撤销、可验证。Prodivix 的核心价值是可视化、代码和 workspace 项目模型共同收敛，LLM 必须进入这个工程系统。

### 为什么不是只生成 PIR

Prodivix 不是单 PIR 文档工具。真实项目还包含 RouteGraph、Workspace VFS、Code Authoring、资源、设置、导出、测试和诊断。把所有 AI 能力压成 PIR patch 会把架构带偏。

### 为什么需要 MCP

Prodivix 不应只服务内置 AI 助手。通过 MCP，外部 LLM 工具可以读取 PIR、检查 schema、调用 dry-run、运行测试，从而让 Prodivix 项目进入主流 AI coding 工具链。

### 为什么需要 repair loop

LLM 面对私有协议时不可能长期一次成功。可靠性来自小动作空间、结构化错误、dry-run 和修复闭环，而不是相信模型“足够聪明”。

## 非目标

1. 本决策不要求立刻实现完整 agent 平台。
2. 本决策不指定唯一模型供应商。
3. 本决策不要求放弃人工编辑。
4. 本决策不允许绕过 PIR / workspace / validator 直接修改核心状态。
5. 本决策不定义具体 UI 细节，例如聊天面板布局。

## 验收标准

- [ ] 存在统一 LLM Gateway 设计，不在业务组件中散落模型调用。
- [ ] AI 写操作收敛为 target-scoped Workspace Action Proposal。
- [ ] AI target 覆盖 RouteGraph、Workspace documents、resources、settings、export、diagnostics、PIR、NodeGraph、Animation 和 CodeArtifact。
- [ ] 写操作具备 dry-run 和 apply 两阶段。
- [ ] validator diagnostics 可被 LLM repair loop 消费。
- [ ] Code artifact 生成进入 format / typecheck / test / compiler dry-run 链路。
- [ ] 大意图先生成 plan artifact，再分批执行。
- [ ] Prodivix MCP Server 有明确 resources / tools 边界。
- [ ] trace / eval 数据模型可以复现 LLM 任务。
