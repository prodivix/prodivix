## 决策背景

Prodivix 的诊断不只是日志或 toast。一个稳定错误码最终可能同时出现在：

1. Code Editor inline diagnostic。
2. Inspector 字段或 Panel。
3. Blueprint canvas 节点标记。
4. NodeGraph 节点、端口或连线。
5. Animation timeline 或 track。
6. Preview overlay、viewport badge、导出前 gate。
7. Issues 面板、API 错误详情、问题上报和 AI 修复上下文。

如果每个入口直接拼接 `message`、`hint` 和临时按钮，会出现以下问题：

1. 同一错误码在不同 UI 中含义不一致。
2. TypeScript、ESLint、CSS、shader 等上游错误码被直接暴露为产品主错误。
3. 用户看得到错误，但不知道错误对象、源码位置和可跳转入口。
4. AI 修复和问题上报只能读取自然语言，缺少稳定 evidence 和 action。
5. 本地化、文档链接、快速修复和发布 gate 会重复实现。

因此需要一个长期稳定的诊断展示契约。它不定义具体 React 组件样式，而定义诊断从结构化事实到 UI 展示模型的稳定桥接。

## 决策目标

1. 保持 `ProdivixDiagnostic` 作为跨层诊断事实对象，不把 UI 状态塞入诊断事件。
2. 为每个稳定错误码提供可复用的展示模板，包括标题、摘要、证据、定位和动作。
3. 让 Issues、Code Editor、Inspector、Canvas、NodeGraph、Animation、Preview 和导出 gate 使用同一展示模型。
4. 支持 `targetRef`、`sourceSpan`、`meta.upstream`、`meta.evidence` 和 `docsUrl` 的统一呈现。
5. 支持快速跳转、打开文档、重试、应用 quick fix、上报问题等动作，但动作本身保持声明式。
6. 允许 UI 文案本地化，保证 `code`、定位协议、action id 和 evidence key 长期稳定。

## 核心决策

### 1) 区分诊断事实、模板和展示模型

Prodivix 诊断展示链路分三层：

```ts
type ProdivixDiagnostic = {
  code: string;
  severity: ProdivixDiagnosticSeverity;
  domain: ProdivixDiagnosticDomain;
  message: string;
  hint?: string;
  docsUrl?: string;
  retryable?: boolean;
  cause?: unknown;
  meta?: Record<string, unknown>;
  targetRef?: DiagnosticTargetRef;
  sourceSpan?: SourceSpan;
  quickFixes?: readonly DiagnosticQuickFixReference[];
};

type DiagnosticPresentationTemplate = {
  code: string;
  titleKey: string;
  summaryTemplate: DiagnosticMessageTemplate;
  detailTemplate?: DiagnosticMessageTemplate;
  primaryLocation?: DiagnosticLocationPreference;
  evidence?: DiagnosticEvidenceTemplate[];
  sections?: DiagnosticDetailSectionTemplate[];
  actions?: DiagnosticActionTemplate[];
};

type DiagnosticPresentation = {
  code: string;
  title: string;
  summary: string;
  detail?: string;
  severity: ProdivixDiagnosticSeverity;
  domain: ProdivixDiagnosticDomain;
  locations: DiagnosticLocationPresentation[];
  evidence: DiagnosticEvidencePresentation[];
  sections: DiagnosticDetailSectionPresentation[];
  actions: DiagnosticActionPresentation[];
  docsUrl?: string;
};
```

规则：

1. `ProdivixDiagnostic` 是事实对象，可以跨 worker、API、日志和 telemetry 传递。
2. `DiagnosticPresentationTemplate` 属于 registry，是每个错误码的展示意图。
3. `DiagnosticPresentation` 是 UI 可直接渲染的结果，由 presentation builder 生成。
4. 模板和展示模型不包含 React component、callback、closure 或不可序列化对象。
5. 展示文案可以本地化；定位、证据和动作使用稳定结构化字段。

### 2) 诊断模板只描述展示意图

模板不负责计算错误是否存在，也不负责执行跳转。

```ts
type DiagnosticMessageTemplate = {
  defaultText: string;
  i18nKey?: string;
  variables?: DiagnosticTemplateVariable[];
};

type DiagnosticTemplateVariable = {
  name: string;
  source:
    | { kind: 'diagnostic'; path: string }
    | { kind: 'definition'; path: string }
    | { kind: 'location'; path: string }
    | { kind: 'meta'; path: string };
  fallback?: string;
  format?: 'plain' | 'code' | 'number' | 'ratio' | 'color' | 'path';
};
```

规则：

1. 模板只能引用诊断对象、registry definition、location resolver 和 meta 中的字段。
2. 模板变量缺失时必须有 fallback 或安全省略策略。
3. 模板不得读取完整源码、Token、API key、完整 prompt 或未裁剪 stack。
4. 同一错误码的标题必须来自模板或本地化资源，不由 provider 每次临时拼接。
5. Provider 可以提供 `message` 和 `hint`，但 registry 模板是跨 UI 的默认展示来源。

### 3) Location 是长期跳转协议

展示层必须把 `targetRef` 和 `sourceSpan` 转成可读位置和可执行跳转。

```ts
type DiagnosticLocationPreference =
  | 'target'
  | 'source'
  | 'target-then-source'
  | 'source-then-target'
  | 'operation';

type DiagnosticLocationPresentation = {
  id: string;
  role: 'primary' | 'secondary' | 'related';
  label: string;
  kind:
    | 'workspace'
    | 'document'
    | 'pir-node'
    | 'inspector-field'
    | 'route'
    | 'nodegraph-node'
    | 'nodegraph-port'
    | 'animation-track'
    | 'code-artifact'
    | 'theme-token'
    | 'viewport'
    | 'runtime-dom'
    | 'component-slot'
    | 'operation';
  ref?: DiagnosticTargetRef;
  sourceSpan?: SourceSpan;
  canNavigate: boolean;
};
```

定位规则：

1. `sourceSpan` 用于 Code Editor inline diagnostic 和代码跳转。
2. `targetRef` 用于产品对象定位，例如节点、字段、端口、track、theme token 或 viewport。
3. 一个诊断可以同时有 `targetRef` 和 `sourceSpan`，展示层必须保留两者。
4. 默认主位置由错误码模板决定；例如 `COD-2003` 通常 source 优先，`UX-1001` 通常 target 优先。
5. 无法解析的 ref 仍应显示原始稳定标识，并把 `canNavigate` 设为 false。
6. Location resolver 可以依赖当前打开文档和编辑器状态，但 resolver 输出必须是展示模型，不回写诊断事实。

### 4) Action 是声明式能力，不是 UI 回调

快速跳转、打开文档、重试和 quick fix 都应通过 action descriptor 表达。

```ts
type DiagnosticActionKind =
  | 'navigate'
  | 'open-docs'
  | 'retry'
  | 'apply-fix'
  | 'acknowledge'
  | 'create-exemption'
  | 'copy-report'
  | 'open-related';

type DiagnosticActionTemplate = {
  id: string;
  kind: DiagnosticActionKind;
  labelKey: string;
  labelFallback: string;
  requires?: DiagnosticActionRequirement[];
  placement?: 'primary' | 'secondary' | 'overflow';
};

type DiagnosticActionRequirement =
  | 'targetRef'
  | 'sourceSpan'
  | 'docsUrl'
  | 'retryable'
  | 'quickFix'
  | 'exemptable'
  | 'relatedDiagnostics';

type DiagnosticActionPresentation = {
  id: string;
  kind: DiagnosticActionKind;
  label: string;
  enabled: boolean;
  disabledReason?: string;
  placement: 'primary' | 'secondary' | 'overflow';
  payload: Record<string, unknown>;
};
```

规则：

1. Template 定义“这个错误码可能有哪些动作”。
2. Presentation builder 根据诊断事实、当前 surface 和 resolver 结果决定 action 是否 enabled。
3. Action runner 是 UI 层能力，负责执行 `navigate`、`retry`、`apply-fix` 等动作。
4. 诊断对象不得携带函数。
5. `apply-fix` 必须引用可审计的 quick fix id 或 command id，不允许直接携带任意脚本。
6. 高风险动作必须进入确认或 review flow，不能因为模板定义了 action 就直接执行。

### 5) Evidence 是结构化证据，不是长文本详情

展示层应把 `meta.evidence`、`meta.upstream`、`cause` 摘要和相关诊断转成证据区。

```ts
type DiagnosticEvidenceTemplate = {
  id: string;
  labelKey: string;
  labelFallback: string;
  source:
    | { kind: 'meta'; path: string }
    | { kind: 'upstream'; path: string }
    | { kind: 'diagnostic'; path: string }
    | { kind: 'location'; path: string };
  format?: 'plain' | 'code' | 'number' | 'ratio' | 'color' | 'list' | 'json';
  redaction?: 'none' | 'summary' | 'hash' | 'hidden';
};

type DiagnosticEvidencePresentation = {
  id: string;
  label: string;
  value: string;
  format: 'plain' | 'code' | 'number' | 'ratio' | 'color' | 'list' | 'json';
  sensitive?: boolean;
};
```

规则：

1. Evidence 必须短、可扫描、可复制到问题上报中。
2. 上游 TypeScript、ESLint、CSS、SCSS、GLSL、WGSL 错误码进入 `meta.upstream`，展示为 evidence 或 detail section。
3. `cause` 默认不展示给普通用户，只允许在开发者详情或报告包中使用裁剪摘要。
4. UX 证据优先展示标准编号、viewport、颜色、对比度、DOM path、focus order 和 bounding box。
5. COD 证据优先展示上游错误码、期望/实际类型、语言模式、artifactId、host contract 和 source span。

### 6) Detail sections 按稳定类别组织

不同诊断域需要不同详情区，但 section id 应长期稳定。

```ts
type DiagnosticDetailSectionKind =
  | 'what-happened'
  | 'how-to-fix'
  | 'location'
  | 'evidence'
  | 'upstream'
  | 'related-diagnostics'
  | 'host-contract'
  | 'standard-reference'
  | 'developer-details'
  | 'reporting';

type DiagnosticDetailSectionTemplate = {
  id: string;
  kind: DiagnosticDetailSectionKind;
  titleKey: string;
  titleFallback: string;
  visibleWhen?: DiagnosticActionRequirement[];
};
```

规则：

1. Issues 面板可以展示所有 sections。
2. Inline tooltip 只展示 `what-happened`、`how-to-fix` 和最关键 evidence。
3. Inspector 字段提示优先展示 `location`、`how-to-fix` 和 action。
4. Preview overlay 优先展示目标位置、viewport、UX evidence 和打开 Issues 的动作。
5. `developer-details` 默认折叠，且必须过滤敏感信息。

### 7) Surface 决定展示密度，不改变诊断语义

同一 `DiagnosticPresentation` 可以在不同 surface 中裁剪展示。

```ts
type DiagnosticSurface =
  | 'issues-panel'
  | 'code-editor-inline'
  | 'inspector-field'
  | 'blueprint-canvas'
  | 'nodegraph'
  | 'animation-timeline'
  | 'preview-overlay'
  | 'operation-status'
  | 'export-gate'
  | 'api-error-detail';
```

Surface 规则：

| Surface              | 展示重点                                     |
| -------------------- | -------------------------------------------- |
| `issues-panel`       | 完整标题、位置、证据、动作、文档和相关诊断   |
| `code-editor-inline` | 代码位置、短说明、上游错误、打开 docs/target |
| `inspector-field`    | 字段位置、修复建议、跳到代码或相关节点       |
| `blueprint-canvas`   | 节点标记、短说明、打开 Inspector 或 Issues   |
| `nodegraph`          | 节点/端口定位、类型/连接证据、打开 Issues    |
| `animation-timeline` | track/keyframe 定位、绑定证据、打开 Issues   |
| `preview-overlay`    | DOM/viewport 证据、节点/字段反向跳转         |
| `operation-status`   | 操作失败摘要、重试、打开详情                 |
| `export-gate`        | 阻断原因、严重程度、批量修复/豁免            |
| `api-error-detail`   | requestId、后端 envelope、相关 diagnostics   |

规则：

1. Surface 可以隐藏 section 或 action，但不能改写 `code`、severity、domain 和 primary meaning。
2. Toast 只能作为轻量入口，不应成为诊断唯一展示位置。
3. `fatal` 和 `error` 默认在相关 surface 可见；`warning` 进入相关面板；`info` 可折叠。
4. 聚合展示必须保留原始 code 列表和每条诊断的定位。

### 8) Presentation builder 是唯一组装入口

所有 UI surface 应通过同一个 builder 获得展示模型。

```ts
type BuildDiagnosticPresentationInput = {
  diagnostic: ProdivixDiagnostic;
  definition?: DiagnosticDefinition;
  template?: DiagnosticPresentationTemplate;
  surface: DiagnosticSurface;
  locale: string;
  resolver: DiagnosticPresentationResolver;
};

type DiagnosticPresentationResolver = {
  resolveLocation(
    targetRef?: DiagnosticTargetRef,
    sourceSpan?: SourceSpan
  ): DiagnosticLocationPresentation[];
  resolveActions(input: {
    diagnostic: ProdivixDiagnostic;
    template: DiagnosticPresentationTemplate;
    locations: DiagnosticLocationPresentation[];
    surface: DiagnosticSurface;
  }): DiagnosticActionPresentation[];
  formatEvidence(input: {
    diagnostic: ProdivixDiagnostic;
    template: DiagnosticPresentationTemplate;
    locations: DiagnosticLocationPresentation[];
    locale: string;
  }): DiagnosticEvidencePresentation[];
};
```

规则：

1. Provider 只产生诊断事实，不直接产生 UI 卡片。
2. Registry 提供 definition 和 presentation template。
3. Builder 负责插值、证据裁剪、位置解析、action enablement 和 surface 裁剪。
4. UI components 只渲染 `DiagnosticPresentation`，不直接读取复杂 `meta`。
5. API、worker 或日志环境无法解析 UI 位置时，可以只生成无导航 presentation。

### 9) Registry 需要从码位定义扩展到展示模板

现有 `DiagnosticDefinition` 已包含 `code`、`domain`、`severity`、`stage`、`retryable`、`docsPath` 和 `defaultPlacement`。后续应扩展为：

```ts
type DiagnosticDefinition = {
  code: string;
  domain: ProdivixDiagnosticDomain;
  severity: ProdivixDiagnosticSeverity;
  stage: string;
  retryable: boolean;
  docsPath: string;
  docsUrl?: string;
  defaultPlacement?: DiagnosticPlacement[];
  presentation?: DiagnosticPresentationTemplate;
};
```

规则：

1. 新增稳定错误码时必须同步定义 presentation template。
2. 没有专用模板的错误码使用域级默认模板。
3. 域级默认模板必须至少显示 code、message、hint、docs、location 和 report action。
4. `COD`、`UX`、`PIR` 等域可以共享 action 和 section，但 evidence 模板按域扩展。
5. 文档生成器可以读取 presentation template 生成“UI 展示建议”章节，但文档生成不是运行时依赖。

### 10) COD 与上游错误码的展示规则

`COD-xxxx` 是 Prodivix 主错误码，上游错误码只作为详情。

```ts
type UpstreamDiagnostic = {
  source: 'typescript' | 'eslint' | 'css' | 'scss' | 'glsl' | 'wgsl';
  code?: string | number;
  severity?: 'info' | 'warning' | 'error';
  message: string;
  sourceSpan?: SourceSpan;
  docsUrl?: string;
};
```

展示规则：

1. 列表标题显示 `COD-xxxx` 和 Prodivix 诊断名称。
2. 详情中显示上游来源、原始 code、原始 message 和 docsUrl。
3. 上游 code 可以作为筛选条件，但不能替代 Prodivix `code`。
4. 多个上游错误可聚合在同一 `COD-xxxx` 下，但必须保留各自 sourceSpan。
5. 如果上游错误无法映射到稳定 Prodivix 语义，使用 `COD-9002` 或更具体的证据不足码。

### 11) UX 与外部标准编号的展示规则

`UX-xxxx` 是 Prodivix 主错误码，WCAG、axe、Lighthouse、WAI-ARIA 等编号进入 evidence。

展示规则：

1. 标题显示 `UX-xxxx` 和 Prodivix 体验问题名称。
2. 详情显示 WCAG criterion、level、rule id、viewport、computed style、contrast ratio 等证据。
3. 如果 evidence 不足，使用 `UX-9004`，不能伪装成通过或失败。
4. 对用户代码导致的 UX 问题，可以同时提供 code sourceSpan 和 UX targetRef。
5. UX 豁免通过 action 或 export gate 流程创建，不直接删除诊断。

## Prodivix 默认模板策略

### 域级默认模板

每个 domain 至少需要一个默认模板：

| Domain      | 默认 primary location | 默认 actions                             |
| ----------- | --------------------- | ---------------------------------------- |
| `code`      | `source-then-target`  | open source、open target、open docs      |
| `ux`        | `target-then-source`  | open target、open docs、create exemption |
| `pir`       | `target`              | open target、open docs、copy report      |
| `editor`    | `target`              | open target、retry、open docs            |
| `nodegraph` | `target`              | open nodegraph、open docs                |
| `animation` | `target`              | open timeline、open docs                 |
| `codegen`   | `operation`           | open export detail、copy report          |
| `backend`   | `operation`           | retry、copy requestId、open docs         |
| `ai`        | `operation`           | retry、open prompt detail、open docs     |

### 错误码专用模板

需要专用模板的码位包括：

1. 有上游错误的 `COD-xxxx`。
2. 有标准证据的 `UX-xxxx`。
3. 有 requestId 和 backend envelope 的 `API-xxxx`。
4. 有 graph、port、track、viewport、theme token 等特殊定位的码位。
5. 有 quick fix 或豁免流程的码位。

## 2026-07-13 实现基线

当前实现已形成以下纵向链路：

1. `@prodivix/diagnostics` 持有 framework-neutral 的 provider snapshot、单调 revision、active/stale/resolved 生命周期、跨 provider fingerprint 去重、稳定排序、查询和汇总。
2. provider 的晚到旧 revision 会被拒绝，revision 切换期间尚未重新发布的 provider 结果标记为 stale，不会用旧扫描结果复活已解决问题。
3. Web composition root 已接入 Workspace VFS、Route Manifest、PIR、NodeGraph、Animation、Code parser、Durable Outbox 和 Revision Conflict provider，并提供全局 Issues 产品面、严重程度计数、筛选、来源说明和 `Alt+0` 入口。
4. `targetRef` 已支持 Workspace、VFS node、Document、Route、PIR node、NodeGraph、Animation、CodeArtifact 和 Operation 等定位；Web runner 已打通 Workspace、Route、PIR node、NodeGraph、Animation track、CodeArtifact/SourceSpan 与操作记录入口。
5. Quick Fix 只携带 typed command/transaction reference。Web action runner 必须先通过受信 registry 将 reference 解析为 `WorkspaceCommandEnvelope` 或 `WorkspaceTransactionEnvelope`，再进入 History/Operation 写路径；诊断对象不能携带 callback 或脚本。

仍未完成的范围：

1. 真实 TypeScript/JavaScript/CSS/GLSL/WGSL Language Service 与 Plugin/UX/Export 等 broader provider 需要继续发布同一 snapshot contract。
2. 当前 Code provider 对持久化 CodeArtifact 做 parser-level diagnostics 并支持精确 SourceSpan 聚焦，但它不等于 G1 的完整 Language Service、definition/reference 或增量诊断。
3. 当前 Animation provider 已覆盖持久化 Animation 文档的结构与时间轴诊断；Animation lifecycle、composition、runtime parity 和完整行为验证继续归 G3。
4. G0 所需 Issues provider、稳定 target、Code SourceSpan、编辑器回跳、Golden Conformance 与全域写入审计已闭环并通过 `pnpm run verify:g0`；broader provider 不回退 G0 Product Gate。

## 分阶段落地

### Phase 1：文档和类型契约

1. 建立本决策文档。
2. 在 diagnostics 类型中补充 `DiagnosticPresentationTemplate`、`DiagnosticPresentation`、location、action 和 evidence 类型。
3. 不实现复杂 UI，只保证类型边界和 registry 可承载模板。

### Phase 2：Registry 模板

1. 为 `COD` 和 `UX` 建立域级默认模板。
2. 为高价值码位补专用模板，例如 `COD-2003`、`COD-3011`、`UX-1001`、`UX-2001`、`UX-9004`。
3. 保持旧 `message`、`hint` 兼容。

### Phase 3：Presentation builder

1. 实现纯函数 builder。
2. 接入 location resolver 和 action resolver。
3. 先让 Issues 面板消费展示模型。
4. 再逐步接入 Code Editor、Inspector、Canvas、Preview 和 export gate。

### Phase 4：Quick actions 和修复

1. 接入 open source、open target、open docs、retry、copy report。
2. 为有审计需求的 quick fix 引入 command id。
3. 为 UX 豁免引入结构化 create exemption action。
4. AI 修复只消费 presentation/evidence/targetRef，不直接解析自然语言 message。

## 长期不变量

1. `ProdivixDiagnostic` 是事实对象，presentation 是派生对象。
2. 展示模板声明结构和意图，不包含 React 组件或可执行回调。
3. 所有快速动作都是 action descriptor，由 surface 的 action runner 执行。
4. `code`、`domain`、`severity`、`targetRef`、`sourceSpan` 和 action id 是稳定协议。
5. UI 文案可以本地化，不能作为程序判断依据。
6. 上游错误码和外部标准编号只作为 evidence，不替代 Prodivix 主错误码。
7. Issues、inline tooltip、Inspector、Canvas、Preview 和 export gate 可以裁剪展示密度，但不能改变诊断语义。
8. 无法定位或证据不足必须显式表达，不能假装有跳转或假装检查通过。
9. 展示层必须过滤敏感信息，不直接暴露 `cause`、完整源码、Token、完整 prompt 或未裁剪 stack。

## 非目标

1. 本决策不定义具体 React 组件层级、CSS class 或视觉样式。
2. 本决策不要求立即实现完整 Issues 面板。
3. 本决策不把每个错误码绑定到独立 UI 组件。
4. 本决策不改变后端 Diagnostic Envelope。
5. 本决策不要求 provider 立即提供所有 evidence；缺失证据应通过 fallback 或证据不足诊断表达。
