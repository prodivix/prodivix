# 独立组件路由模块与宿主路由融合

## 状态

- ImplementationStatus：G1 Frontend Composition Implemented / G2 Export Integration Planned
- ProductGateStatus：G1 Contribution Passed / G2 In Progress
- Global Phase：G1-G2
- 创建日期：2026-02-08

> 本 ADR 只定义 Route Module Composition。“独立组件”在本文中表示可挂载相对 RouteGraph 的 route module，不定义 Blueprint 的 `pir-component` Public Contract、Component Instance、subtree extraction 或 Collection reuse；这些语义由 `specs/decisions/38.blueprint-component-instance-and-collection.md` 定义。

## 决策背景

Prodivix 同时支持页面项目和可复用组件。组件内部可能需要多个状态页、子视图或局部导航，例如 account center、settings panel、dashboard widget。但如果组件直接声明绝对路径，会带来三个问题：

1. 组件复用到不同宿主路径时会产生冲突。
2. 宿主无法统一治理 guard、loader、SEO、监控和实验策略。
3. 画布中的 `PdxRoute` 会变成第二套项目路由系统。

因此，组件路由必须与 workspace RouteGraph 使用同一套结构，但保存为 **相对路由模块**。

## 决策内容

引入 **Route Module Composition**：

1. 独立组件可以导出相对 RouteGraph，称为 `RouteModule`。
2. 宿主项目通过 route node 挂载该 module，并提供 `mountPath` 或父级 route context。
3. 编译、预览和调试阶段将宿主 RouteGraph 与组件 RouteModule 合成 `ResolvedRouteGraph`。
4. 组件运行时能力受宿主策略裁剪；组件不得声明全局导航守卫。
5. `PdxRoute` 作为 route module 的可视化投影组件存在，不直接保存绝对路径。

## 数据结构

```ts
type RouteModule = {
  moduleId: string;
  version: '1';
  root: RouteNode;
};

type RouteModuleMount = {
  mountId: string;
  moduleRef: string;
  mountPath?: string;
  parentRouteNodeId?: string;
  runtimePolicy?: {
    allowLoader?: boolean;
    allowAction?: boolean;
    allowGuard?: boolean;
  };
};
```

组件模块中的 `segment` 是相对片段，不允许以 `/` 开头。模块根可以有 index route，但不能声明 workspace 级绝对 path。

## 合成规则

1. 宿主 RouteGraph 先解析自身 route tree。
2. 遇到 module mount 时，将模块 root 挂入宿主节点。
3. 合成 path 由宿主 path + module relative segment 得到。
4. 参数名冲突在编辑期报错，或由宿主显式提供别名策略。
5. guard、loader、action、SEO 的最终治理权属于宿主。

## 示例

### 组件模块导出

```json
{
  "moduleId": "account-center",
  "version": "1",
  "root": {
    "id": "account-root",
    "children": [
      { "id": "account-home", "index": true, "pageDocId": "page-account-home" },
      {
        "id": "account-profile",
        "segment": "profile",
        "pageDocId": "page-account-profile"
      }
    ]
  }
}
```

### 宿主挂载

```json
{
  "mountId": "mount-account-center",
  "mountPath": "/user",
  "moduleRef": "account-center"
}
```

### 合成结果

- `/user` -> `page-account-home`
- `/user/profile` -> `page-account-profile`

## 与 PdxRoute 的关系

`PdxRoute` 不再通过 children 上的任意 `data-route-path` 形成项目级路由。它的职责收敛为：

1. 在组件画布中预览当前 route module 的匹配结果。
2. 在宿主预览中消费 `RouteRuntimeContext`，展示当前 module scope 下命中的分支。
3. 为 Inspector 提供 route module 调试入口。

项目级导航、发布和导出必须以合成后的 `ResolvedRouteGraph` 为准。

## 为什么不直接共享绝对路由

1. 组件复用要求可被挂载到多个宿主路径。
2. 绝对路径共享会引入冲突与依赖顺序问题。
3. 组件与宿主应保持松耦合。
4. 宿主必须拥有最终运行时治理权。

## 实施顺序

1. 让 `pir-component` document 支持单模块路由预览。
2. 让 `PdxRoute` 局部 children 匹配通过 route module projection 运行。
3. 在 Workspace RouteManifest 中建立 module mount 能力。
4. 导出时输出合成后的 route graph 与 source trace。

## 风险与缓解

1. 风险：模块循环引用。
   - 缓解：module graph 构建阶段做 DAG 校验。
2. 风险：参数命名冲突。
   - 缓解：编辑期诊断并要求宿主提供别名。
3. 风险：调试复杂度提升。
   - 缓解：路由调试面板展示 source module -> resolved path 映射。

## 验收标准

- [ ] `component` 项目可独立预览相对 route module。
- [ ] `project` 可挂载至少两个组件 route module。
- [ ] 合成路由可追踪到宿主 route node、module id 和源文档。
- [ ] 路径冲突在编辑阶段可被阻止。
