## 背景

G2 需要让图片、字体和下载文件从 Canonical Workspace 进入 Browser/Remote Preview、Test、Build
与 standalone Export。ADR 生效前，`asset` Workspace document 仍可保存 `dataUrl`、`text` 和 size/mime，
Compiler 再从作者态 JSON 解码字节。这会把大二进制复制进 Workspace Operation、local replica、
revision diff、日志和 Git text projection，并且无法证明 digest、size、MIME、授权与 Remote materialization
使用的是同一个对象。

现有 `ExportAssetContribution`、`ExecutableProjectFile`、Remote codec 和 rootless Worker 已支持
`Uint8Array`。缺失的是原始 blob 的稳定 owner、引用 contract、授权 materialization 与安全策略，
而不是另一套 project filesystem。

## 决策

### 1. Transport-neutral owner

`@prodivix/assets` 是 Binary Asset current contract 的唯一 transport-neutral owner，负责：

- content-addressed blob reference、digest、byte length 与 media type normalization；
- upload/materialization port 与 byte-level digest/size verification；
- deterministic transform recipe identity；
- media policy、size budget 与稳定 validation issue。

该包不依赖 React、Browser、Gin、PostgreSQL、Remote transport、Workspace persistence 或图片处理 SDK。
`@prodivix/workspace` 只把类型化 reference 保存到 `asset` document，并继续拥有 document identity、path、
revision、History 与 Atomic Commit。

### 2. Canonical Workspace 只保存引用

current `WorkspaceAssetDocumentContent` 形状为：

```ts
{
  kind: 'asset';
  mime: 'image/png';
  size: 1234;
  blob: {
    kind: 'workspace-blob';
    digest: 'sha256-...';
    byteLength: 1234;
    mediaType: 'image/png';
  };
  category?: 'image';
  metadata?: { width?: 640; height?: 480 };
}
```

`mime`/`size` 必须与 blob reference 完全一致。`dataUrl`、base64、`text`、filesystem path、signed URL、
download token、bucket key 和 provider object 不得进入 Workspace document。文本 public file 也按 UTF-8 bytes
进入 blob store；Code/Config 应继续使用各自类型化 document，不借 Asset 绕过领域 validator。

### 3. Blob identity 与 tenant isolation

- digest 使用 canonical `sha256-` + 64 位小写十六进制；size 是非负 safe integer；media type 必须是
  去参数、lowercase 的 `type/subtype`。
- digest 绑定 exact bytes，而不是 filename、MIME 或 Workspace path。reference 同时保存 expected size/media type，
  materialization 必须三者全部匹配。
- 生产 blob store 按 Workspace/owner 授权分区；即使底层物理去重，也不得通过 exists/timing 暴露跨 tenant
  对象存在性。
- upload 是幂等的：相同 Workspace/digest/bytes 返回 existing；digest 相同但 metadata/bytes 不一致 fail closed。
- blob 先验证并持久化，再提交引用它的 Workspace Operation。操作失败产生的 orphan 由 retention/mark-sweep
  清理，不把 blob upload 伪装成 Workspace 作者事务。
- production blob 以 nullable `unreferenced_since` 记录 reference mark：current Asset document reference 对应
  `NULL`，orphan 对应首次观察或 durable dereference 时间。新引用与 dereference 在持有 canonical Workspace row lock
  的同一 authoring transaction 内更新 mark；sweep 也先取得该锁并在锁内重检 exact Workspace-local digest reference。
  因此 sweep 与 commit 串行化，locked Workspace 被 `SKIP LOCKED` 跳过，且只有严格早于 cutoff 的 orphan 才能删除。
- delete/replace 只开始新的 retention window，不同步删除 bytes；相同 exact-byte orphan 的授权 retry 可以刷新 grace，
  但 current reference 不得被 retry 改成 orphan。sweep 必须按 Workspace/blob 数量有界，且只记录 aggregate telemetry。
- local-only adapter 使用独立 IndexedDB blob database，并以 Workspace/digest 分区；它实现相同 uploader/reader
  port、大小限制、幂等 existing/conflict 与 read-time exact-byte verification，不把 bytes 并入 local project record、
  Workspace wire 或 Outbox。IndexedDB 不可用、缺 blob 或 identity drift 均 fail closed。
- local -> cloud project import 是新 Workspace 的单次原子迁移例外：Web 在发送前 materialize/verify 全部引用，Backend
  通过 bounded multipart 接收一个 reference-only JSON manifest 与 digest-named raw byte parts，并在同一 PostgreSQL
  transaction 内创建 Project、Workspace、blobs 与 documents。missing、unreferenced、duplicate、digest/size/media drift
  或 budget overflow 在任何写入前 fail closed；无 Asset 的 JSON import 保持兼容，旧 JSON-only Asset import 固定
  `AST-2004`。

### 4. Materialization hard cut

Compiler 不直接读取网络、数据库、signed URL 或 Browser cache。composition root 通过
`BinaryAssetBlobReader` 取得 bytes，形成已经验证的 `BinaryAssetMaterialization`，再把它传给 Compiler。
Compiler 按 `(workspaceId, assetDocumentId, digest)` 精确匹配并生成 `ExportAssetContribution`：

```text
Workspace asset reference
  -> authorized blob reader
  -> digest/size/media verification
  -> Compiler ExportAssetContribution + SourceTrace
  -> ExecutableProjectFile Uint8Array
  -> Browser/Remote filesystem
```

缺失、重复、未引用、digest drift、size drift 或 media type drift 都产生 blocking diagnostic；不得回退到
`dataUrl`、空文件、远程 URL 或运行时下载。

### 5. Executable Snapshot 与 Remote

Executable Snapshot 只携带已经 materialize 的 bounded bytes。blob credential/reference store locator 不进入
snapshot；Snapshot content digest 覆盖 binary bytes。Remote strict codec、Control Plane、Worker 和 rootless sandbox
继续使用同一 file budget、digest 与 network hard cut。安装期 allowlist 不能被 Asset materialization 复用为项目
运行出站通道。

### 6. Transform 与 delivery

derived asset 不是新的 Workspace 真相。transform identity 绑定 source digest、transformer id/version、canonical
parameters 与 output media type；结果进入可丢弃的 content-addressed derived cache。crop/resize/format conversion
等作者动作只保存显式 recipe/reference，不保存临时 preview bytes。

G2 first vertical 只允许经过 policy 的静态 media inline。HTML、JavaScript、SVG 等 active content 永远禁止
inline；只有配置的 scanner 判 clean 后，才允许从独立 capability origin 以
`application/octet-stream`/attachment、deny-all CSP、sandbox、`nosniff` 与 `noopen` 交付。public/CDN URL 只能由
独立 delivery projection 生成，不能写回 Workspace。

当前 PNG first vertical 使用 deterministic structural sanitizer 剥离非渲染 metadata，并验证 chunk/CRC、
critical chunk、颜色 profile、dimensions/pixels。baseline JPEG second-format vertical 只接受 8-bit baseline
Huffman、1/3 component、合法 DQT/DHT/SOF0/SOS/EOI 与 bounded segment/scan/sampling/dimensions；剥离 APP/COM，
只保留渲染所需 Adobe APP14，并保持已验证 entropy bytes exact。progressive/arithmetic/CMYK、非默认 EXIF
orientation、table redefinition、trailing/truncated/malformed 或超预算输入全部 fail closed。两种格式的最终 clean
attestation 都由 versioned scanner chain 在对应 structural scanner 与 ClamAV `INSTREAM` malware adapter 全部 clean
后颁发。ClamAV adapter 在建立连接前验证 exact
digest/length/media，以有界 frame、response 与 timeout 执行，并把 daemon signature 收敛为固定 finding code。
daemon error、connection、timeout 或 protocol drift 均 fail closed。

scanner composition 区分 required engine 与同 engine replica：required engine 必须全部 clean；replica 只在 bounded
scanner infrastructure unavailable 时按固定顺序 failover，quarantine 不得回退到另一 replica。Host fleet runtime
以 bounded `PING`/`VERSIONCOMMANDS` 验证每个 replica 的 engine、signature database version/time 与 required command
set；过期或未来时间拒绝 readiness。每个 required engine 只接受 freshest converged cohort，downgrade、相同
freshness 的不同 digest、不可比较 frontier 或全部 replica unavailable 均 fail closed。

effective policy version 由 canonical composition base identity 与全部 required engine policy digest 共同派生；只有
所有 engine 都通过，才原子发布新的 immutable scanner generation。FreshClam metadata 更新可由 readiness/delivery
refresh 接纳，不要求重启。Host 在 upload 前和 capability session 签发前验证同一 generation；观察到新 generation
时撤销旧短期 session，old-generation in-flight scan 不得签发。derived cache 继续保存 exact bytes，但 scanner version
变化必须在命中时重新扫描并替换 attestation。更多格式的 transform、完整 raster re-encode、第二 malware vendor 与
public CDN operation Gate 仍待完成。

### 7. Git 与 local replica

Canonical local replica 保存 reference/metadata，不复制 base64；local blob database 是按引用 materialize 的独立
byte owner。local duplicate 必须先复制全部 exact referenced blobs 再提交新 Workspace record；project delete 先删除
作者态再清理对应 blob partition，document delete/replace 不做同步 destructive delete。local -> cloud 已使用
upload-aware import：JSON manifest 继续只保存 reference，raw parts 按 digest 去重并受单对象/数量/总字节预算约束，
Backend 在一个 transaction 内提交新 cloud Workspace。Git projection 生成稳定 manifest，并在目标明确支持时投影
binary file 或 LFS pointer；不得把 provider bucket key 或 signed URL 提交到 Git。runtime filesystem 中的 asset 修改
仍是可丢弃运行态，只有显式 import/replace planner 才能上传新 blob 并形成可逆 Workspace Transaction。

## First Vertical 范围

首个纵切完成：

1. `@prodivix/assets` strict contract、digest 与 materialization verification；
2. Workspace asset current contract hard cut，拒绝 `dataUrl`/`text`；
3. Backend Workspace-scoped content-addressed upload/read adapter；
4. Compiler 从 verified materialization 生成 binary Export/Executable files；
5. Browser/Remote codec/rootless 路径保持 exact bytes 与 SourceTrace；
6. digest/size/media drift、active content、oversize 与 reference leak Gate；
7. PNG 与 baseline JPEG deterministic sanitizer、derived cache、各自 structural + ClamAV scan/quarantine、Backend
   owner gateway 与独立 capability Asset Delivery Host；allowlisted 其他 attachment 同样必须通过 ClamAV。
8. Workspace-scoped IndexedDB local blob adapter、Resources upload/preview、Run/Test/Export materialization，以及
   duplicate-before-reference/delete-after-author-state lifecycle。
9. local-to-cloud upload-aware bounded multipart、strict manifest/raw-part decoder，以及 Project/Workspace/blob/document
   single-transaction rollback Gate。
10. transport-neutral required scanner chain/replica failover、ClamAV multi-engine fleet freshness selection、atomic
    generation refresh、old-session revocation 与 in-flight signing fence。
11. Workspace-locked PostgreSQL reference/orphan mark、durable dereference grace、bounded `SKIP LOCKED` sweep、
    aggregate-only maintenance telemetry 与真实 PostgreSQL isolation/concurrency Gate。
12. deterministic Git binary/LFS manifest/pointer/object projection、managed attributes lifecycle、missing/drift hard cut，
    以及 upload-receipt-fenced runtime filesystem Asset import/replace 单事务采纳。

第二 malware vendor、完整 raster re-encode/更多格式 transform、public/CDN delivery，以及
Remote/Test/Build/Export 与第二 target 的真实 product image closure 在后续阶段完成；Browser JPEG
upload/reload/sanitize/isolated-preview 已由独立产品 Gate 固定。

## 停止条件

1. 任一实现要求把 bytes/base64、signed URL、token 或 bucket key 写入 Workspace 时停止。
2. Compiler 需要自行访问 blob store 或网络时停止，改由 composition root materialize。
3. 未验证 digest/size/media type 时不得把 bytes 放入 Export Bundle 或 Executable Snapshot。
4. active content 未经 sanitizer/独立 origin policy 时不得作为同源 public asset 交付。
5. blob upload 不得绕过 Workspace owner authorization，也不得用跨 tenant dedup 暴露对象存在性。

## 后果

- Workspace revision、History、Outbox 与 Git text diff 保持小而确定。
- Browser、Remote 与 Export 复用同一 materialized bytes，不需要第二套 Asset runtime truth。
- 既有 Workspace 的常规 blob upload 与后续 document commit 是有序但独立的 durable 操作；reference-aware retention
  已保证失败 upload 最终回收、current reference 永不被 sweep 删除，并为 durable dereference 提供完整 grace window。
  新 local-to-cloud import 因 Project、Workspace、blob 与 document 同属一个 PostgreSQL 边界而使用真实单事务提交。
- 旧 `dataUrl` asset 是不符合 current contract 的历史形状，必须显式迁移，不能静默兼容。
