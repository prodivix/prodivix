## 决策背景

MFE 的后端错误不能长期停留在普通字符串或仅面向 HTTP 状态码的表达上。Workspace 同步、PIR 校验、Intent/Patch、Git 集成、AI Provider、社区系统都会跨越前端、后端、文档、日志和用户上报。

如果后端只返回自然语言错误，前端无法稳定地把失败挂到对象、字段或操作入口上，文档也无法长期维护可检索的排障入口。

因此后端需要稳定的 Diagnostic Envelope，作为所有 API 错误响应和后端诊断回传的长期协议。

## 决策目标

1. 让后端错误响应具有稳定 `code`，可被文档、测试、日志和前端消费。
2. 明确 HTTP status 与 diagnostic code 的关系，避免用 status 承载业务语义。
3. 支持单个错误和多个 diagnostics 的长期扩展。
4. 不保留旧错误响应兼容层，直接切换到稳定 envelope。
5. 避免泄露隐私数据、Token、SQL、堆栈和完整用户输入。
6. 让前端能把后端错误挂到就地提示、操作入口或调试详情，而不是直接 toast。

## 核心决策

### 1) 所有新后端错误响应使用统一 Envelope

统一响应结构：

```ts
type BackendErrorEnvelope = {
  error: {
    code: string;
    message: string;
    severity?: 'info' | 'warning' | 'error' | 'fatal';
    domain?: string;
    retryable?: boolean;
    requestId?: string;
    docsUrl?: string;
    details?: Record<string, unknown>;
    diagnostics?: BackendDiagnostic[];
  };
};

type BackendDiagnostic = {
  code: string;
  message: string;
  severity?: 'info' | 'warning' | 'error' | 'fatal';
  domain?: string;
  retryable?: boolean;
  path?: string;
  targetRef?: DiagnosticTargetRef;
  docsUrl?: string;
  details?: Record<string, unknown>;
};
```

JSON 示例：

```json
{
  "error": {
    "code": "WKS-4003",
    "message": "Content revision conflict.",
    "severity": "warning",
    "domain": "workspace",
    "retryable": true,
    "requestId": "req_01J...",
    "docsUrl": "/reference/diagnostic-codes#workspace",
    "details": {
      "serverRev": 42,
      "clientRev": 41,
      "revision": "contentRev"
    }
  }
}
```

规则：

1. `error.code` 是该响应的主错误码，必须稳定。
2. `error.message` 是兜底文案，不作为测试和业务分支依据。
3. `error.diagnostics` 用于返回多个具体诊断；主 `error.code` 表示整体失败原因。
4. `requestId` 用于关联日志和用户上报。
5. 所有 API 错误响应只返回该 envelope，不再返回裸 `{ error: string, message: string }`。

### 2) 不保留旧响应兼容字段

以下旧响应形态不再作为服务端输出契约：

```json
{
  "error": "SOME_CODE",
  "message": "..."
}
```

以下混合响应也不允许出现：

```json
{
  "error": {
    "code": "API-4001",
    "message": "Validation failed."
  },
  "message": "Validation failed."
}
```

规则：

1. 前端新代码只读取 `error.code`。
2. 后端不得返回顶层 `message`、字符串型 `error`、`legacyError` 或旧 `code` 字段。
3. 不符合 envelope 的错误响应视为协议违规；前端只能降级为 `API-9001`，不能解析旧字段。

### 3) HTTP status 只表达传输层和粗粒度类别

HTTP status 与 diagnostic code 是正交关系：

| HTTP status | 用途                       | 示例 code                          |
| ----------- | -------------------------- | ---------------------------------- |
| `400`       | 请求形状错误或业务校验失败 | `API-1001`、`API-4001`             |
| `401`       | 未认证或会话失效           | `API-2001`、`API-2002`             |
| `403`       | 已认证但权限不足           | `API-3001`                         |
| `404`       | 请求资源不存在或不可见     | `WKS-1001`、`WKS-3001`             |
| `409`       | 并发冲突、revision 冲突    | `WKS-4001`、`WKS-4002`、`WKS-4003` |
| `422`       | 语义可解析但领域校验失败   | `PIR-1001`、`PIR-2001`             |
| `429`       | 限流                       | `API-1020`                         |
| `500`       | 服务端未知异常             | `API-9001`                         |
| `502/503`   | 外部服务或依赖不可用       | `API-6001`、`AI-1002`              |

规则：

1. 前端不能只根据 HTTP status 判断业务错误。
2. 同一个 HTTP status 下可以有多个不同 diagnostic code。
3. 同一个 diagnostic code 的 HTTP status 应保持稳定，除非 ADR 修订。
4. `5xx` 不应承载可预期的业务校验失败。

### 4) 主 code 与 diagnostics 数组的关系

单一错误：

```json
{
  "error": {
    "code": "WKS-3001",
    "message": "Document not found."
  }
}
```

多个领域诊断：

```json
{
  "error": {
    "code": "PIR-4001",
    "message": "PIR materialize failed.",
    "diagnostics": [
      {
        "code": "PIR-2003",
        "message": "Child node id does not exist.",
        "path": "/ui/graph/childIdsById/root/0"
      },
      {
        "code": "PIR-2006",
        "message": "Node is not reachable from root.",
        "path": "/ui/graph/nodesById/card"
      }
    ]
  }
}
```

规则：

1. `error.code` 是操作失败的摘要码。
2. `error.diagnostics[]` 是可定位的下游诊断。
3. 如果只有一个具体诊断，允许不返回数组。
4. 如果返回数组，前端应优先使用数组中的 `targetRef` / `path` 做就地提示。

### 5) details 是机器可读上下文，不是日志倾倒口

允许进入 `details` 的内容：

1. revision 数字、docId、workspaceId、routeId、nodeId 等稳定 ID。
2. provider 名称、operation 名称、阶段名。
3. 受控枚举状态，例如 `expectedRev`、`actualRev`、`capability`。
4. 可展示的短字符串，例如 route path、component type。

禁止进入 `details` 的内容：

1. Token、API key、cookie、authorization header。
2. SQL、连接串、文件系统绝对敏感路径。
3. 完整用户输入、完整 Prompt、完整模型响应。
4. 原始堆栈 trace。
5. 大型 payload、完整 PIR 文档、完整源码。

后端日志可以记录更丰富的内部上下文，但必须通过 `requestId` 与用户可见 envelope 关联。

### 6) targetRef 是可选的对象定位协议

后端可以返回 `targetRef`，帮助前端把诊断挂到就地 UI：

```ts
type DiagnosticTargetRef =
  | { kind: 'workspace'; workspaceId: string }
  | { kind: 'document'; workspaceId: string; documentId: string }
  | { kind: 'pir-node'; documentId: string; nodeId: string }
  | { kind: 'route'; routeId: string }
  | { kind: 'inspector-field'; nodeId: string; fieldPath: string }
  | { kind: 'operation'; operation: string };
```

规则：

1. `targetRef` 必须指向前端可理解的稳定对象。
2. 没有明确对象时，使用触发操作作为 target，例如保存、导出、AI apply。
3. 后端不能假设前端一定有对应 UI；前端可以降级到操作入口提示。
4. `path` 用于协议/文档路径，`targetRef` 用于产品对象定位，两者可以同时存在。

### 7) requestId 必须贯穿日志和响应

每个错误 envelope 应包含 `requestId`。如果入口层已有 trace id，应复用或映射。

规则：

1. HTTP middleware 生成或读取 `X-Request-Id`。
2. 响应 header 返回同一个 `X-Request-Id`。
3. 错误日志必须包含 requestId、code、status、route、method。
4. 用户上报问题时，前端应尽量附带 requestId。

### 8) 前端消费原则

前端 API client 应把 envelope 解析为远端诊断对象：

```ts
type RemoteDiagnosticError = {
  status: number;
  code: string;
  message: string;
  requestId?: string;
  retryable?: boolean;
  diagnostics: ProdivixDiagnostic[];
};
```

规则：

1. API client 只解析，不直接 toast。
2. 调用点决定把错误挂到对象、字段、操作入口或阻断摘要。
3. 前端本地化优先使用 `code`，后端 `message` 作为兜底。
4. 前端测试断言 `code`，不依赖自然语言文案。
5. 远端 `details` 只能进入调试详情或受控 UI，不直接拼进用户文案。

### 9) Workspace 链路优先接入

首批必须稳定的后端 code：

| Code       | HTTP status | 场景                 |
| ---------- | ----------- | -------------------- |
| `WKS-1001` | `404`       | 工作区不存在         |
| `WKS-2001` | `409`       | Capability 不支持    |
| `WKS-3001` | `404`       | 文档不存在           |
| `WKS-3002` | `422`       | 文档类型不支持该操作 |
| `WKS-4001` | `409`       | Workspace rev 冲突   |
| `WKS-4002` | `409`       | Route rev 冲突       |
| `WKS-4003` | `409`       | Content rev 冲突     |
| `WKS-5001` | `422`       | Intent 类型不支持    |
| `WKS-5002` | `422`       | Patch 应用失败       |

Workspace 相关 handler 不应继续返回 generic `WORKSPACE_ERROR` 作为最终错误码。

### 10) PIR validator 返回 PIR diagnostics

后端 PIR 校验不应只返回 `PIR validation failed`。它应返回主 code 和具体 diagnostics。

建议：

```ts
type PirValidationError = {
  code: 'PIR-4001';
  diagnostics: BackendDiagnostic[];
};
```

对于后端已能识别的稳定场景，直接返回对应 `PIR-xxxx`：

| Code       | 场景                  |
| ---------- | --------------------- |
| `PIR-1001` | 保存态包含 `ui.root`  |
| `PIR-1002` | 缺少 `ui.graph`       |
| `PIR-1003` | 节点字段非法          |
| `PIR-2001` | 根节点不存在          |
| `PIR-2002` | 节点 key 与 ID 不一致 |
| `PIR-2003` | 子节点引用不存在      |
| `PIR-2004` | graph 存在环          |
| `PIR-2005` | 节点存在多个结构父级  |
| `PIR-2006` | 孤儿节点              |

规则：

1. 后端和前端 PIR validator 应共享同一语义码位。
2. 短期可以只返回第一个诊断，长期应支持多个 diagnostics。
3. 不能把所有 PIR 失败都折叠成 `API-4001`。

### 11) 后端内部错误类型

Go 后端建议引入内部错误类型：

```go
type DiagnosticError struct {
	Code        string
	Message     string
	Severity    string
	Domain      string
	Status      int
	Retryable   bool
	Details     map[string]any
	Diagnostics []Diagnostic
	Cause       error
}
```

规则：

1. handler 可以直接返回 `DiagnosticError`。
2. middleware 或 response helper 负责序列化 envelope。
3. `Cause` 只进日志，不进响应。
4. `Details` 进入响应前必须经过安全审查。
5. 未分类错误统一映射为 `API-9001`，并记录 cause。

### 12) OpenAPI 必须表达 envelope

所有新增或修订 API 的错误响应必须引用统一 schema：

```yaml
components:
  schemas:
    ErrorEnvelope:
      type: object
      required: [error]
      properties:
        error:
          $ref: '#/components/schemas/ErrorPayload'
```

Workspace OpenAPI 后续需要补齐：

1. `409` response 对应 `WKS-4001/4002/4003`。
2. `422` response 对应 `PIR-xxxx` 和 `WKS-500x`。
3. 每个错误示例必须包含稳定 code。

## 迁移计划

### Phase 1：Response helper

1. 扩展 `apps/backend/internal/platform/http/response/error.go`。
2. 支持新 envelope。
3. 删除旧字段输出，不提供兼容层。
4. 增加 requestId middleware 或复用已有 trace id。

### Phase 2：Workspace 优先映射

1. Workspace not found -> `WKS-1001`。
2. Document not found -> `WKS-3001`。
3. Revision conflict -> `WKS-4001/4002/4003`。
4. Intent unsupported -> `WKS-5001`。
5. Patch failed -> `WKS-5002`。

### Phase 3：PIR 后端 validator

1. 将 `pir_v13_validator.go` 从普通 error string 升级为 diagnostic error。
2. 返回 `PIR-xxxx`。
3. 与前端 validator 语义对齐。

### Phase 4：前端 API client

1. 解析 `BackendErrorEnvelope`。
2. 映射为远端 diagnostic。
3. 调用点决定就地提示或操作入口提示。
4. 停止在 API 层直接 toast。

### Phase 5：OpenAPI 与文档

1. 更新 `workspace-sync.openapi.yaml`。
2. 文档站错误码参考加入后端响应说明。
3. 新增后端错误码测试与契约测试。

## 长期不变量

1. `error.code` 是错误语义的稳定主键。
2. HTTP status 不替代 diagnostic code。
3. `message` 可改写、可本地化，不作为程序分支依据。
4. `details` 永远不包含 secret、完整源码、完整 Prompt、堆栈或大 payload。
5. 已发布 code 不复用；废弃 code 必须保留文档说明。
6. 前端 API 层不直接决定 UI 呈现。
7. 后端未知异常统一映射为 `API-9001`，但日志保留 requestId 和 cause。
8. Workspace 与 PIR 的稳定业务失败不能折叠为 generic API error。
9. 后端错误响应不得包含顶层 `message`、字符串型 `error`、`legacyError` 或旧顶层 `code`。

## 非目标

1. 本决策不定义 MFE 诊断 UI 形态。
2. 本决策不定义 telemetry 存储方案。
3. 本决策不要求后端返回完整多语言文案。
4. 本决策不允许为了排查方便泄露敏感内部错误。

明确不是非目标：本决策要求当前已维护 API 一次性迁移到 envelope，不保留旧响应兼容层。

## 验收标准（Draft）

- [ ] 后端 response helper 能输出统一 `BackendErrorEnvelope`。
- [ ] 新错误响应包含 `error.code` 和 `error.message`。
- [ ] 错误响应不包含顶层 `message`、字符串型 `error`、`legacyError` 或旧顶层 `code`。
- [ ] 错误响应 header 与 body 都能关联 requestId。
- [ ] Workspace revision conflict 返回 `WKS-400x`。
- [ ] Workspace document not found 返回 `WKS-3001`。
- [ ] 后端 PIR validator 至少返回首个 `PIR-xxxx`。
- [ ] 前端 API client 能解析新 envelope。
- [ ] OpenAPI 中存在统一 ErrorEnvelope schema。
- [ ] 新增后端错误路径测试断言 code，而不是自然语言 message。
