- DecisionStatus：Accepted

## 决策背景

Workspace、PIR、Diagnostics 与 Code Authoring 已经成为 Web Editor、Compiler、CLI、VS Code、后端协议和未来 Agent Runtime 的共同基础。这些领域实现继续位于 `apps/web/src` 会造成三个问题：

1. Web 应用成为领域核心的隐式 owner，其他入口只能复制实现或反向依赖应用。
2. React、Zustand、DOM 与 Vite 依赖容易渗入可序列化、可重放的作者态内核。
3. Preview、Compiler 与 Generated Runtime 会各自实现相同语义；此前 Compiler 与 Web 已经存在两份 `materializeUiTree`。

因此，`apps/web` 必须收敛为 Web 产品的 composition root 与交互适配层；稳定领域语义必须由独立 package 持有。

## Phase 1 包边界

### `@prodivix/diagnostics`

拥有：

- 可序列化诊断 envelope；
- definition/presentation contract；
- 通用 action、evidence 与类型守卫。

不拥有：

- PIR、Code、UX 等所有领域 catalog 的全局聚合；
- React Issues UI、Web navigation 或文档路由解析。

PIR catalog 归 `@prodivix/pir`，Code catalog 归 `@prodivix/authoring`。现有 UX catalog 仅通过 `@prodivix/diagnostics/ux` opt-in subpath 暴露，待 Quality domain 建立后再迁移。Web composition root 负责把领域 provider 聚合为产品 Issues 视图。

### `@prodivix/authoring`

拥有：

- `WorkspaceSymbol`、`WorkspaceScope`、`WorkspaceReferenceEdge` 与 revision-aware semantic query contract；
- semantic contribution、统一 identity/visibility/resolution/impact policy 与 provider composition contract；
- code-owned `CodeArtifact`、`CodeReference`、Code Semantic Provider mapping；
- `CodeSlot` 与 Trigger binding contract；
- artifact/semantic/diagnostic/slot provider registries。

不拥有 Canonical Workspace snapshot、各领域保存态解释、CodeMirror、Monaco 或 React 状态。Route、PIR、Component、Collection、NodeGraph、Animation、Code、Token 与 Asset owner 发布完整 semantic contribution；`@prodivix/authoring` 不扫描其内部结构。

### `@prodivix/code-language`

拥有：

- TypeScript / JavaScript 的无 DOM Language Service project 与 revision-bound capability session；
- definition、references、completion、hover、diagnostics 与 rename edit proposal adapter；
- code module、durable export、import/reference dependency 与 code diagnostic semantic contribution。

不拥有 Canonical Workspace code document、Workspace Command/Transaction、CodeMirror 状态、React UI 或 Issues presentation。它只消费 `@prodivix/authoring` 的稳定 language/semantic contract；Web composition root 将 provider、Workspace snapshot、Code Editor 与 Issues 组合起来。

### `@prodivix/tokens`

拥有：

- DTCG Format/Resolver 外部格式 profile 与 codec 边界；
- 无版本号的 Design Token 与 Resolver current model；
- group、root token、alias、JSON Pointer、group extension、cycle 与 effective type validation；
- set、modifier、context、resolution order、input/default 与 ordered source resolution plan；
- revision-bound Token/Group 与 DesignSystem/Set/Modifier/Context Semantic Contribution Provider。

不拥有 Canonical Workspace snapshot、Resources/Inspector React UI、产品主题 CSS 变量或导出 target。`@prodivix/workspace` 保存原始 DTCG Token/Resolver JSON 并通过 Command/Transaction 修改；`@prodivix/themes` 继续只拥有 Prodivix 产品自身的主题投影。

### `@prodivix/pir`

拥有：

- 无版本号的 PIR-current domain contract、factory、normalization 与 validation；
- Component、Collection、Binding、extraction、mutation、projection 与 semantic contribution；
- 显式 `./wire` 边界中的 generated contract、codec 与 deterministic migration；
- PIR diagnostic catalog。

不拥有 React Renderer、组件注册、图标加载、DOM selection、浏览器 navigation 或 route debug。

Compiler 必须消费该包的 materializer，不再维护第二份实现。

### `@prodivix/workspace`

拥有：

- Workspace VFS 与 snapshot contract；
- Command/Intent、Patch policy 与 History；
- validation、selectors 与 `.prodivix/**` projection；
- Workspace code document 到 CodeArtifact 的 authoring bridge；
- transport、replica 与 Git/VFS projection 共用的 Workspace document wire codec；

不拥有 Zustand、fetch、IndexedDB、React hook 或 Web API client。

### `@prodivix/workspace-sync`

拥有：

- Workspace/Route/Document revision capture 与 canonical 409 decoder；
- transport-neutral Workspace semantic diff、code text diff 与三方合并；
- conflict session、显式 resolution 与 fresh WorkspaceOperation 构建；
- 从 WorkspaceOperation 写集推导 exact Atomic Commit revision vector；
- `entry.operation`、History 与 local apply 保留无版本 domain operation；只有
  `workspaceOperationCommitPirWire.ts` 可通过 `@prodivix/pir/wire` 把
  `entry.request.operation` 投影为持久化 wire value。该依赖是精确 persistence boundary，
  不授权 Workspace Sync 的其他模块绕过 Workspace 读取 PIR 领域状态；
- durable outbox entry、exact request、causal head、lease、retry/backoff、conflict blocking 与 ACK identity 状态机。

不拥有 fetch、认证 token、Zustand、React conflict UI 或具体后端 endpoint adapter。Web 负责 transport 与 presentation composition，但不得在 `apps/web` 复制 diff3/rebase 语义。

### `apps/web`

继续拥有：

- React 页面、画布、Inspector 与 Store adapter；
- Web navigation、DOM、localStorage/IndexedDB 与 API transport；
- Route runtime CodeSlot 的 composition adapter。

## 强制约束

Phase 1 core package 必须满足：

1. 生产源码不得导入 `@/`、`apps/web`、React、Zustand、React Flow 或 `@prodivix/ui`。
2. TypeScript 使用 `lib: ["ES2022"]`，不通过 DOM lib 隐式获得浏览器全局。
3. package 可以独立 build 和 test；测试从公共 barrel 或包内稳定入口消费。
4. Domain catalog 由领域 owner 持有，基础 Diagnostics 不反向聚合领域。
5. Web、Compiler 与未来 Generated Runtime 的相同语义应复用同一实现或同一 conformance suite。
6. Web 通过 `@prodivix/*` public API 消费领域核心，composition root 保持为唯一应用装配入口。

边界由 `scripts/check-core-package-boundaries.mjs` 检查，并接入根 `pnpm lint`。

## 当前 Core 状态

Store 持有一个 canonical snapshot，wire DTO 只存在于 API/IndexedDB 边界。`@prodivix/workspace-sync` 持有 semantic diff3、conflict session、resolution operation、Atomic Commit planner、Durable Outbox 与 local replica 语义。

下一步聚焦 Language Service worker 生命周期、ExecutionProvider 与 Animation lifecycle/composition/runtime ports。专用 Asset provider 已从 generic Workspace facts 硬切为 `core.assets`，独立导出项目 browser smoke 已由 `@prodivix/golden-conformance` 建立可重复 Gate。

## Runtime / NodeGraph owner

1. `@prodivix/runtime-core` 已建立 transport-neutral execution request、trace、state patch 与 instance-owned executor registry。
2. `@prodivix/nodegraph` 已拥有不依赖 PIR、React Flow、Window 或 DOM 的 graph document、strict decoder、selection、executor registry、step budget 与 deterministic trace。
3. Web 通过 `@prodivix/runtime-browser` 与显式 `executeNodeGraphAction` composition adapter 注入 document-qualified standalone NodeGraph document。
4. NodeGraph executor 消费独立 graph document 与显式 runtime ports。
5. Compiler 将 standalone NodeGraph 编译为带显式 runtime requirement 与 SourceTrace 的独立模块。
6. Runtime Core 与 NodeGraph 使用轻示例测试和属性测试验证 Registry 隔离、Codec fail-closed、执行确定性与 step budget。

## Animation / Browser Runtime owner

1. `@prodivix/animation` 已拥有 Animation contract、确定性 normalizer、作者态工厂、keyframe easing、timeline direction/fill/iteration 求值和中立 frame projection。
2. Animation Core 使用 `lib: ["ES2022"]`，不依赖 React、DOM、WAAPI、完整 PIR 文档或 Web 状态。
3. `@prodivix/runtime-browser` 已承接 Animation CSS/SVG preview projection、浏览器 ID port 与 NodeGraph 浏览器 action adapter。
4. Web 通过 `@prodivix/animation` 与 `@prodivix/runtime-browser` 组合作者态、求值和预览能力。
5. Compiler 的 keyframe interpolation 与 Web preview 复用同一 `@prodivix/animation` evaluator。
6. 原 Web preview 示例单测由 package 属性测试替代，不重复保留同语义测试。

## Router / PIR React Renderer owner

1. `@prodivix/router` 已接管 RouteManifest contract、closed codec、path normalization、matcher、mutation、module composition、validation 与 navigation target classification。
2. Workspace 持有 Route intent 到 Workspace Command/Transaction 的集成职责。
3. `@prodivix/pir-react-renderer` 已接管 PIRRenderer、PIRNode、scope、registry、capability、route context、route debug、rich text 与 icon provider。
4. Web、Plugin Platform 和编辑器通过 package public API 消费 Renderer。
5. navigation、NodeGraph 与 Animation 行为通过 document-qualified reference 和显式 host/runtime port 组合。
6. Router 使用 `lib: ["ES2022"]` 独立 build/test；Renderer 使用 DOM/React adapter 边界独立 build/test。

## Workspace Local Replica owner

1. `@prodivix/workspace-sync` 已拥有无 DOM replica contract、canonicalization、snapshot/settings 独立 watermark、non-regression advance 与 Outbox materialization。
2. IndexedDB、ProjectSummary、capabilities、认证失败策略和 Editor hydration 留在 Web adapter；Core 不反向依赖浏览器或 UI。
3. Replica 保存 server-confirmed base，并通过同一个跨 store readonly transaction 读取 base 与双 Outbox 后 materialize pending changes。
4. Operation/Settings ACK 都先持久化 confirmed snapshot 与 ACK bridge id，再删除 queue entry，关闭 ACK 后崩溃造成的丢失或重复应用窗口。
5. 仅 transport/retryable failure 允许离线恢复；权威认证、授权、not-found 与 schema failure 不读取缓存。
6. multi-device sync、presence、review 与按领域选择的 CRDT 由 Collaboration & Delivery plane 承接。

## 后续抽取批次

1. Runtime 后续：建立 Browser/Remote ExecutionProvider，并让 Preview、Export 与 Test 共享显式执行 port。
2. `@prodivix/animation` 后续：补齐 lifecycle、composition、conflict、reduced-motion、diagnostic/trace 与 playback port。
3. 继续缩小 `@prodivix/shared`，把剩余领域 contract 与 LLM runtime 分别归还对应 owner。

## Phase 1 验收

- 无 DOM 核心 packages（含 `@prodivix/code-language`、`@prodivix/tokens`、`@prodivix/router`、`@prodivix/workspace-sync`、`@prodivix/runtime-core`、`@prodivix/nodegraph` 与 `@prodivix/animation`）可独立 build/test；`@prodivix/runtime-browser` 与 `@prodivix/pir-react-renderer` 可独立 build/test。
- Web 只通过 `@prodivix/*` public API 消费已迁移核心。
- Compiler 已删除重复 PIR materializer。
- 根 typecheck、unit tests、lint 与 production build 全部通过。
