## 决策摘要

人类手势、AI proposal、插件 action 与 importer input 先由领域 planner 解释，再形成可逆 `WorkspaceCommandEnvelope` 或原子 `WorkspaceTransactionEnvelope`。Command / Transaction 通过本地校验和 History 后形成 `WorkspaceOperation`，进入 Durable Outbox 与 Atomic Commit。

Intent 是 planner 的本地输入模型。每个入口可以使用最适合自身交互的 typed intent，但生产作者态写入统一收敛到 Command / Transaction。

## Planner 边界

```text
Human gesture / AI proposal / Plugin action / Importer input
  -> domain planner
  -> impact / capability / revision checks
  -> reversible Command or atomic Transaction
  -> final-state validation
  -> local History
  -> WorkspaceOperation / Durable Outbox / Atomic Commit
```

Planner 负责：

1. 读取明确的 Canonical Workspace revision 与领域上下文。
2. 将交互输入解析为领域合法动作。
3. 生成确定性的 forward/reverse operations、target、domain hint 与用户可读 label。
4. 对 rename、move、delete、extract 等跨文档动作查询 impact，并组合为一个 Transaction。
5. 在 apply 前返回可定位、可修复的领域诊断。

## Command contract

当前稳定 contract 由 `@prodivix/workspace` 持有：

```ts
type WorkspaceCommandEnvelope = {
  id: string;
  namespace: string;
  type: string;
  version: string;
  issuedAt: string;
  forwardOps: WorkspacePatchOperation[];
  reverseOps: WorkspacePatchOperation[];
  target: {
    workspaceId: WorkspaceId;
    documentId?: WorkspaceDocumentId;
    routeNodeId?: string;
  };
  mergeKey?: string;
  label?: string;
  domainHint?: WorkspaceCommandDomain;
};
```

规则：

1. `namespace/type/version` 为领域与插件提供稳定命名空间。
2. `forwardOps` 与 `reverseOps` 描述同一可逆修改。
3. Patch operation 只存在于 Command 内部，并受 path policy、codec 与 validator 约束。
4. `target`、`domainHint` 与 operation metadata 为 History、Issues、Diff 和审计提供稳定落点。
5. `mergeKey` 只控制本地 History 合并语义，不改变 durable operation identity。
6. Command apply 从输入 snapshot 产生新的候选 snapshot，不直接调用远端 transport。

## Transaction contract

```ts
type WorkspaceTransactionEnvelope = {
  id: string;
  workspaceId: WorkspaceId;
  issuedAt: string;
  commands: WorkspaceCommandEnvelope[];
  label?: string;
  mergeKey?: string;
};
```

Transaction 是一个原子作者态边界：

1. commands 按确定顺序应用到同一个候选 Workspace。
2. Workspace 与所有相关领域 validator 检查最终候选状态。
3. 任一 command 或最终态校验失败时，Transaction 整体保持未应用。
4. History、undo、redo、replay、Outbox 与 Atomic Commit 以同一个 transaction identity 处理该修改。
5. 跨文档 component extraction、route composition、code binding 与 reference relocation 使用 Transaction。

## WorkspaceOperation

`WorkspaceOperation` 是持久化与远端提交的 exact authoring request，payload 是已经规划和校验的 Command 或 Transaction。它携带 actor、base revision vector、operation identity 与 metadata，并由 Durable Outbox 在发送前持久化。

Atomic Commit 以强幂等方式提交 exact operation。ACK 推进 confirmed revision 与 local replica watermark；pending operation materialization 继续提供离线作者态读取。

## 扩展规则

1. 新领域先定义 typed planner input、Command factory、validator 与 diagnostic catalog。
2. 插件通过受权限约束的 Host Gateway 请求 planner 能力，Host 生成 Workspace Command / Transaction。
3. AI 输出 Intent、Action Proposal 或 Code Patch Proposal，由 planner 转换为领域 Command / Transaction。
4. 新的 wire 字段通过 WorkspaceOperation contract 与 Atomic Commit ADR 演进。
5. Settings 使用独立的 durable Settings Outbox / Commit contract。

## 长期不变量

1. Canonical Workspace VFS 是领域写入的唯一作者态目标。
2. 所有生产作者态修改都具有可逆 Command 或原子 Transaction 表达。
3. Patch operation 只属于 Command 内部实现。
4. WorkspaceOperation 是 durable authoring write request。
5. History、Outbox、Atomic Commit、Diff、Issues 与审计共享稳定 operation identity。
6. Planner、validator 与 domain owner 共同决定动作是否合法。
