# 插件沙箱与能力权限模型

## 状态

- Draft
- 日期：2026-02-08
- 实现状态更新：2026-07-11
- 执行计划：`specs/implementation/plugin-browser-sandbox-phase4.md`

## 决策背景

未来规划涉及插件市场、自定义节点、第三方 API 集成。若插件可直接访问宿主全量能力，将引入安全与稳定性风险：

1. 非授权数据访问（项目内容、用户信息）
2. 任意网络请求与隐私泄露
3. 插件崩溃拖垮编辑器主线程

## 决策内容

采用 **Capability-based Sandbox**：

1. 插件运行在宿主选择的隔离上下文；Manifest 不指定具体 transport
2. 插件通过声明能力（capabilities）申请权限
3. 宿主按最小权限原则授权
4. 所有敏感操作经宿主网关执行并审计

`document.write` 等写能力仅代表可以调用受校验的 Host Gateway patch / intent，不授予对 Workspace store、PIR graph、编辑器局部状态或文件系统的直接写权限。

普通 same-origin Web Worker 仍具有 `fetch`、WebSocket、IndexedDB 等平台能力，不能单独作为安全边界。community plugin 默认需要隔离 origin / opaque origin、严格 CSP（网络默认拒绝）和消息网关；official plugin 可以采用优化 transport，但必须使用同一 Host API、capability 和审计协议。

## 能力清单（初版）

```txt
workspace.read
workspace.intent.dispatch
document.read + scope
document.write + scope
route.read
route.intent.dispatch
network.request + allowlist scope
secrets.read + secret scope
extension.register + contribution point scope
telemetry.emit
```

Manifest v1 不使用 `network.request:<allowlist-id>` 一类拼接字符串，而使用 `{ id, scope, reason, optional }` 对象。宿主可以在授权缓存或审计索引中生成内部 canonical key，但不能要求插件作者自行拼接和解析权限字符串。

完整结构以 `specs/plugins/plugin-manifest-v1.schema.json` 为准。JSON Schema 负责结构校验，semantic validator 负责 capability 与 contribution 对应关系、资源路径、SemVer range 和宿主策略。

## 授权模型

1. 安装时显示权限摘要
2. 高风险能力（`network.request`、`secrets.read`）需显式确认
3. 企业模式支持管理员策略（allow/deny list）
4. 运行时可撤销授权

## 运行时隔离

1. 插件不得直接访问宿主 DOM、Workspace store、PIR graph 或宿主凭据。
2. 插件不得直接发起网络请求或使用持久化存储；敏感能力只经 Host Gateway。
3. 插件与宿主通过版本化消息协议通信，输入输出均执行结构与权限校验。
4. 每个插件有可执行的 artifact、消息、pending request、wall-time 与 heartbeat 预算；浏览器不支持可靠 hard heap cap 时不得伪称精确内存隔离。
5. 超限、失联或协议违规触发熔断、owner 清理与降级。

## 审计与可观测

1. 记录插件触发的 intent 与网络请求摘要
2. 支持审计导出（时间、插件、能力、结果）
3. 异常率与超时率进入健康评分

## 当前实现状态（更新至 2026-07-11）

`@prodivix/plugin-host` 已实现本 ADR 的 transport-neutral Host Core 部分：

1. Capability identity 使用完整 `(id, scope)`，未请求能力不能被授权。
2. PermissionSnapshot 不可变且带单调 revision；required deny 进入 `blocked`，optional deny 可按 point 降级。
3. Registry commit 会重新检查 owner generation、permission revision、operation token 与 capability grant。
4. Required revoke 会 abort activation、停止 runtime 并清理 installation/activation lease；optional point revoke 会按 capability 原子移除 contribution。
5. Runtime adapter、package reader、policy、audit sink、clock 与 ID factory 均为注入 port，Host Core 不创建 Worker 或 iframe。
6. Phase 4.0 已实现 runtime artifact 独立限额、实际 SHA-256、Manifest integrity 与 package digest 绑定；校验失败不会调用 runtime adapter。
7. Host `shutdown()` 会 abort Manifest read/activation、以 `host-shutdown` reason 停止 runtime、清理 owner lease 并关闭 subscriber。

Phase 4.1-4.3 已完成 Browser 安全底座与 Gateway contract：

1. `@prodivix/plugin-protocol` 从四份 JSON Schema 生成类型、runtime Schema 与无 `eval`/`new Function`/CommonJS runtime helper 的 standalone validator；strict codec、exact contract registry、sequence/correlation/cancel/timeout 均由公开行为测试覆盖。
2. `@prodivix/plugin-browser` 通过 nonce-bound opaque iframe broker、私有 MessagePort 和 Dedicated Worker 执行 verified runtime bytes。Chromium 不允许 opaque frame 中的 blob module Worker，因此 Host bootstrap 明确构建为 self-contained IIFE classic Worker，再在 Worker 内导入 verified self-contained ESM runtime；broker 在 transfer 前校验 bootstrap/runtime digest。
3. `@prodivix/plugin-sandbox` 作为独立静态 origin 生成 CSP、Permissions Policy、Cloudflare `_headers` 与 Nginx 配置；production Playwright 已证明 runtime/UI 无法访问 parent DOM、network、storage、popup、top navigation、browser permission 或 nested Worker，hang 不阻塞 editor main loop。
4. Browser Gateway 已实现 frozen exact contract registry、Manifest-requested capability、live permission subscription、owner/generation/session guard、rate/concurrency/timeout/response limits，以及 `runtime.health/ping`、telemetry、workspace、document 与 network 最小 handler contract。
5. `network/request` 只通过 Host adapter 执行，逐跳校验 exact HTTPS origin、method、normalized path、headers、private target、redirect 与响应上限；`secrets.read` 继续无 handler。
6. Host-owned IndexedDB audit store 使用 record count + UTF-8 bytes 双上限，required-before-effect 在事务提交失败时 fail closed，持久记录只保留受限脱敏 metadata 与 diagnostic code。
7. Phase 4.4 已将 Browser adapter、Gateway contracts 与 Palette 收敛进 workspace-scoped Web Plugin Platform。editor shell 创建单一 Host session，Provider 在 workspace switch/unmount/HMR/test teardown 时先等待旧 cleanup/shutdown，再创建下一 Host。
8. 产品 composition root 已注入 Workspace summary 与 Document read service ports；未冻结 exact intent/patch contract 前，write ports 返回 unavailable。未配置独立 sandbox URL 时 runtime activation fail closed。

Phase 4.5-4.8 已完成 External Library、Render Policy、Codegen Policy、Icon Provider 与 Blueprint Template exact contract，以及 Ant Design、MUI、Radix bundled official plugin 迁移。三个 official package 使用 build-attested React Host Module，不把 React/DOM implementation 暴露给 community runtime。

Phase 4.9 已完成最终 hardening：strict protocol 增加 deterministic property/fuzz，production sandbox conformance 在 Chromium、Firefox、WebKit 上覆盖 verified runtime、异常终止和 UI authority 隔离；sandbox policy 由单一来源生成 CSP、Permissions Policy、Cloudflare 与 Nginx 配置。独立 GHCR image、Compose service 和 deploy smoke 会验证真实 production 响应的 script hash、无 Cookie 与未知路径 404。

本 ADR 仍保持 Draft：activation-time sandbox RPC implementation proxy、完整 Workspace/Document write ports、broader extension points、public SDK 与 community package 分发尚未完成。不能把 bundled official package 的受信任主线程投影误写成任意 community plugin 已可生产安装。

## 风险与缓解

1. **风险：权限弹窗过多影响体验**
   - 缓解：分组授权 + 模板策略
2. **风险：沙箱通信性能损耗**
   - 缓解：批量消息与只读快照缓存
3. **风险：恶意插件绕过限制**
   - 缓解：签名校验、内容哈希、策略拦截与市场审核

## 验收标准（Draft）

- [x] Host Core 不会授予 Manifest 未请求的 capability。
- [x] Required capability deny/revoke 会阻止 activation 并清理 owner lease。
- [x] 敏感调用可通过 live permission guard 读取当前 revision，而不是永久缓存 activation-time allow。
- [x] Chromium、Firefox、WebKit production sandbox conformance 证明 opaque runtime/UI 无直接 DOM、network、storage 或 browser authority。
- [x] Host network adapter 对 HTTPS allowlist、redirect、header、private target 与 response limit fail closed。
- [x] required Gateway audit 不可用时 service effect 不执行，持久 audit 有界且脱敏。
- [ ] 插件无法越权读写工作区数据
- [x] 插件 runtime hang/crash 不影响主编辑器事件循环可用性
- [ ] 审计日志可追踪插件关键行为
