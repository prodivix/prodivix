## 决策背景

使用单一 `workspaceRev` 会让高频文档编辑、Workspace tree 修改与 RouteManifest 修改互相制造无意义冲突，也无法为用户说明究竟哪个作者态分区已过期。

Revision 必须反映并发控制边界，而不是把所有文件变化压成一个全局版本号。

## 决策内容

Canonical Workspace 使用以下 revision：

1. `workspaceRev`：Workspace tree、文档挂载关系与 Workspace 结构层版本。
2. `document.contentRev`：单个 WorkspaceDocument 内容版本。
3. `document.metaRev`：单个 WorkspaceDocument 的 path、name、type、capabilities 等 metadata 版本。
4. `routeRev`：RouteManifest 逻辑版本。
5. `opSeq`：服务端 Workspace 操作的全局单调序号，用于确认、审计与后续增量读取；它不是独立 compare-and-set 输入。

## 写入规则

### 文档内容

- 请求校验目标文档的 `expectedContentRev`。
- 成功后仅推进目标文档 `contentRev` 及 Workspace `opSeq`。
- 不因纯内容变化推进 `workspaceRev` 或 `routeRev`。
- 文档内容写入不得通过 Workspace revision 绕过 document compare-and-set。

### Workspace 结构与文档 metadata

- 请求校验 `expectedWorkspaceRev`。
- Workspace tree、文档创建/删除/移动/重命名等结构变化推进 `workspaceRev`。
- Atomic Commit 修改已有文档 name/path/capabilities 时还必须校验对应 `metaRev`，成功后推进该 `metaRev`。
- 新增文档以显式 `contentRev: null, metaRev: null` 声明 identity 必须不存在；删除文档校验当前 contentRev 与 metaRev，避免删除并发内容或 metadata 修改。
- 纯 Workspace 结构写入不推进无关文档的 `contentRev`。

### RouteManifest

- 请求同时校验 `expectedWorkspaceRev` 与 `expectedRouteRev`。
- 成功后推进 `workspaceRev`、`routeRev` 与 `opSeq`。
- Route 写入不推进无关文档的 `contentRev`。

### 多分区操作

多分区用户动作使用一条原子 WorkspaceTransaction：所有 expected revisions 在同一事务内校验，全部成功后共同推进受影响 revisions；任一校验或 validator 失败都不产生 partial state。

`@prodivix/workspace-sync` 已由 Operation 写集推导 exact expected vector，Web 已统一提交 `/operations/commit`；后端按 ADR 36 在单一数据库事务内完成全部 CAS、Apply、validation、revision、operation log 与幂等结果。旧 `/batch` 的逐条 Store 提交已 Hard Cut。

## 冲突分类

Wire contract 只允许三种冲突：

| 错误码     | 类型                 | 必需 expected 基线                                 | current 摘要                                                               |
| ---------- | -------------------- | -------------------------------------------------- | -------------------------------------------------------------------------- |
| `WKS-4001` | `WORKSPACE_CONFLICT` | `workspaceRev`                                     | `workspaceRev`、`routeRev`、`opSeq`                                        |
| `WKS-4002` | `ROUTE_CONFLICT`     | `workspaceRev`、`routeRev`                         | `workspaceRev`、`routeRev`、`opSeq`                                        |
| `WKS-4003` | `DOCUMENT_CONFLICT`  | `document.id` 及 `contentRev` / `metaRev` baseline | Workspace revisions、`opSeq` 与安全 metadata；远端已删除时 `document:null` |

`HYBRID_CONFLICT` 已 Hard Cut 移除：

1. 它会把多个 compare-and-set 失败折叠成客户端无法稳定处理的第四种语义。
2. 一次请求只返回实际检测到的具体分区冲突。
3. 客户端获取最新 canonical snapshot 后，用 semantic diff3 分析所有重叠变化；不依赖 409 枚举全部差异。

## Revision 与作者态 Diff 的边界

Revision 用于确认“基线是否仍然有效”，不用于描述“内容改变了什么”。

- 409 `current` 只提供安全的 revision 摘要。
- 完整内容只通过已授权的 Workspace snapshot 读取。
- Semantic diff 忽略 `contentRev`、`metaRev`、`updatedAt` 与 `opSeq` 的单纯变化。
- 冲突解决后必须相对于最新 remote snapshot 构建新 Operation；不得把旧 expected revisions 或旧 reverse ops 带入新写入。

## 安全与健壮性不变量

1. 所有 revision 必须是正整数，并由服务端权威推进。
2. expected 与 current document id 必须一致。
3. 非 document conflict 不得夹带 document 属性；document conflict 的 expected document 至少包含 contentRev/metaRev 之一，current.document 属性必需且为安全 metadata 或表示远端已删除的 `null`。
4. 409 不返回 document content。
5. 不存在与无权访问使用同一 404 行为，避免通过 revision details 探测 Workspace。
6. 任何成功 mutation 都返回最新 `workspaceRev`、`routeRev` 与 `opSeq`；受影响文档返回最新 document revisions。

## 对未来能力的扩展价值

1. 不同文档可以独立保存，减少无关冲突。
2. Durable outbox 可以按受影响分区保持因果顺序。
3. Git history、审计时间轴与冲突 UI 可以用 revision 定位基线，再用 semantic diff 解释变化。
4. 后续 CRDT 可以局部进入 document content，而 Workspace/Route 仍保留服务端 compare-and-set。

## 验收标准

- [x] 文档内容、Workspace 结构与 RouteManifest 使用独立 revision 基线。
- [x] Document 修改不因无关 Route revision 变化而失败。
- [x] Canonical 409 明确指出 Workspace、Route 或 Document 分区。
- [x] `HYBRID_CONFLICT` 从 wire schema、decoder 与实现中移除。
- [x] 409 不返回文档正文，且未授权请求不暴露 current revision。
- [x] 客户端从 WorkspaceOperation 精确推导多分区 expected vector，不再构造旧 batch 链式 revisions。
- [x] 后端跨文档、多分区 Operation 具备单事务原子提交与强幂等 replay。
