# Inspector Panel 架构决策

## 状态

- Draft
- 日期：2026-04-19
- 执行清单：
  - `specs/implementation/inspector-section-to-panel-refactor.md`

## 决策背景

当前 Blueprint Inspector 处于 Section 向 Panel 迁移的中间态，存在以下问题：

1. 运行时渲染树已经主要按 Tab 拆分，但仍残留旧 `Section` 时代的命名与折叠状态模型。
2. Style Tab 与 Code Tab 的容器模式不一致，Style 已部分接入 panel registry，Code 仍是 tab 内部硬编码 section。
3. Panel 缺少统一父组件，标题栏、展开折叠按钮和 header actions 无统一约束。
4. 部分能力在迁移过程中丢失渲染路径，例如 className 编辑、external props 编辑。
5. Style Tab 当前只覆盖部分样式域，难以承载后续 typography、background、border 等独立能力扩展。
6. Code Tab 除了 triggers 之外，还需要为未来的外部执行代码挂载预留独立入口。

## 决策目标

1. 将 Inspector 统一到稳定层级：`Tab -> Panel -> Group -> Field`。
2. 让 `Panel` 成为 Inspector 中唯一合法的一等折叠单元。
3. 为所有 Panel 建立统一的标题栏、actions 和展开折叠交互。
4. 用 capability-based visibility 约束 Panel / Group 的出现条件。
5. 为 Style / Code 两个 tab 的继续扩展提供一致的架构基座。

## 决策内容

### 1) 统一采用 `Tab -> Panel -> Group -> Field`

- `Tab` 负责大域切分：`basic / style / data / code`
- `Panel` 负责一等语义域与折叠
- `Group` 负责 Panel 内部子能力组织
- `Field` 负责具体编辑控件

约束：

- 不再允许旧 `Section` 继续存在于运行时渲染树
- 不允许使用 `Group` 替代 `Panel`

### 2) 所有 Panel 必须通过统一 Panel Frame 渲染

新增统一父组件，例如 `InspectorPanelFrame`，至少负责：

- 整行标题栏
- 左侧标题
- 右侧 actions 区
- 固定包含展开折叠按钮
- body 容器样式和 spacing

约束：

- 不允许 panel 自己各自拼 header
- 不允许把 action button 放在 tab 顶部独立行

### 3) Header Actions 是 Panel 的一等组成

标题栏右侧结构固定为：

```text
custom actions... + expand/collapse
```

首批明确归属：

- `ClassNamePanel`：Mounted CSS action button
- `TriggersPanel`：Add Trigger action button
- `ExternalCodePanel`：后续若有“打开代码编辑器”等动作，也应挂在 header actions

### 4) Panel 按 Tab 通过 registry 统一解析

保留 `INSPECTOR_PANELS` 机制，但扩展为按 `tab` 过滤：

- `resolveInspectorPanels(node, 'style')`
- `resolveInspectorPanels(node, 'code')`

结论：

- Style Tab 和 Code Tab 不再手写各自的 panel 结构
- panel 的出现与顺序统一由 registry 决定

### 5) Style Tab 中样式域优先拆成独立 Panel

Style Tab 首批语义域：

- `ClassNamePanel`
- `LayoutPatternPanel`
- `LayoutPanel`
- `TypographyPanel`
- `BackgroundPanel`
- `BorderPanel`
- `AnimationPanel`

决策理由：

1. 各样式域出现条件不同
2. 各样式域折叠状态应独立
3. 避免把 LayoutPanel 膨胀成“通用大样式容器”

结论：

- `Panel` 负责语义域
- `Group` 只负责 panel 内局部组织

### 6) Code Tab 至少包含两个一等 Panel

Code Tab 首批必须容纳：

- `TriggersPanel`
- `ExternalCodePanel`

其中：

- `TriggersPanel` 负责事件到动作的绑定
- `ExternalCodePanel` 负责节点外部执行代码的挂载入口

结论：

- 两者不可合并
- 即使 `ExternalCodePanel` 第一阶段只做壳层，也必须先在架构中占位

### 7) Panel / Group 必须按 capability 条件出现

统一规则：

- `panel.match(node)` 决定 panel 是否出现
- `group.match?(node, context)` 决定 group 是否出现

示例约束：

- `TypographyPanel` 仅在节点有文本能力时出现
- `LayoutPanel` 仅在普通布局节点出现
- layout pattern root 只显示 `LayoutPatternPanel`，不显示 `LayoutPanel`
- `ExternalCodePanel` 仅在支持外部代码挂载的节点出现

## 直接影响

### 目录结构

运行时活跃代码需从 `sections/` 迁移到：

- `inspector/InspectorContext*.ts(x)`
- `inspector/fields/*`
- `inspector/panels/*`
- `inspector/components/InspectorPanelFrame.tsx`

### 状态模型

折叠状态统一为：

- `expandedPanels: Record<string, boolean>`

删除：

- `ExpandedSectionsState`
- `expandedSections`
- `toggleSection`

### 运行时结构

- Basic Tab 先保持 field 直出
- Style / Code Tab 统一改为 panel loop
- Data Tab 暂时维持现状，但内部折叠也应统一复用 `expandedPanels`

## 取舍说明

### 为什么不把所有样式继续塞进 LayoutPanel

不采用该方案，原因：

1. 语义域混杂
2. 可见性条件复杂
3. 折叠粒度过粗
4. 后续扩展成本更高

### 为什么 Basic Tab 暂不强制改为 Panel

当前 Basic Tab 的主要问题是能力丢失和命名迁移，不是容器结构不足。优先修复主干架构，不在这一轮强行把所有 field 包成 panel。

### 为什么现在就要引入 ExternalCodePanel

即使能力尚未完整实现，也需要先占住架构位置，否则未来 Code Tab 会再次经历一次结构重排。

## 非目标

1. 本决策不要求本轮补齐全部 Inspector 测试。
2. 本决策不要求一次性做完 Typography / Background / Border 的完整字段能力。
3. 本决策不改变 PIR 文档结构。
4. 本决策不改变 `@prodivix/ui` 组件运行时协议。
