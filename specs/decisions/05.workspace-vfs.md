## 决策摘要

Prodivix 使用 Canonical Workspace VFS 作为项目作者态的唯一 aggregate。Workspace 同时保存逻辑文件树、领域文档、RouteManifest、revision 分区与当前作者上下文。

PIR page/layout/component、NodeGraph、Animation、Code、Asset 与 Project Config 是 Workspace 内由各领域 owner 管理的独立文档。PIR 负责 UI graph，不承担整个项目的单一巨型数据结构。

## Canonical aggregate

`@prodivix/workspace` 持有稳定模型、codec、validator、selector、Command、Transaction 与 History。完整 TypeScript contract 以 `specs/workspace/workspace-model.md` 为准。

```text
WorkspaceSnapshot
  ├─ workspaceRev / routeRev / opSeq
  ├─ treeRootId / treeById
  ├─ docsById
  │    ├─ pir-page / pir-layout / pir-component
  │    ├─ pir-graph / pir-animation
  │    └─ code / asset / project-config
  ├─ routeManifest
  └─ activeDocumentId / activeRouteNodeId
```

## Identity 与 path

1. `WorkspaceDocument.id` 是跨 rename、move、route、component instance、CodeReference 与 diagnostic target 的稳定身份。
2. `WorkspaceDocument.path` 是用户可理解的 VFS path，用于 Resources、文件树、Git projection、Export 与错误文案。
3. VFS node identity 与 document identity 分离；一个 document 由一个 document node 挂载。
4. 文档内部实体通过 `{ documentId, localEntityId }` 构成稳定 target。
5. RouteManifest 使用 document id 连接 page/layout，不使用 path 或显示名称作为主引用。

## VFS 语义

1. directory 拥有有序、去重的 children；child 与 parent 必须双向一致。
2. document node 引用一个存在的 Workspace document。
3. root、orphan、cycle、duplicate sibling name 与 document path 一致性由 Workspace Validator 统一检查。
4. Tree、Canvas、Timeline、Preview、Export、Git、Issues 与 Semantic Index 都从同一 snapshot 派生读投影。
5. 用户通过 Resources、Blueprint、Route、NodeGraph、Animation 与 Code 作者表面创建、移动、重命名和编辑对象；这些动作统一形成领域 Command 或原子 Transaction。

## 保存与同步

```text
Domain planner
  -> reversible Command / atomic Transaction
  -> Workspace + domain validation
  -> local History
  -> WorkspaceOperation
  -> Durable Outbox
  -> strong-idempotent Atomic Commit
  -> confirmed local replica
```

Backend Workspace 是 canonical persistence owner。Git、Project publication、Preview、Export 与 local replica 都是明确用途的投影。

## 领域扩展

1. 新 document type 必须声明 owner package、codec、validator、projection 与 diagnostic catalog。
2. 跨文档引用使用领域类型化 contract，并由 Workspace Semantic Index 提供统一 definition/reference/impact 查询。
3. 跨文档修改先构造最终候选 Workspace，再由所有相关 validator 检查并作为一个 Transaction 应用。
4. 浏览器离线能力通过正式 local replica 与双 Durable Outbox materialize canonical base + pending changes。

## 边界

1. Workspace VFS 是逻辑作者态文件系统，不实现 chmod、symlink 等完整 POSIX API。
2. 二进制大资产由独立 Asset storage/adapter 承载，Workspace 保存稳定 identity 与 metadata。
3. multi-device collaboration、presence、review 与按领域选择的 CRDT 属于 G5 Collaboration & Delivery。
