## 决策背景

当前三个可视化编辑器在 `apps/web/src/editor/features/` 下的目录层级不对称：

1. `animation/` 与 `development/` 都是顶层 feature，根组件直接位于 feature 根。
2. Blueprint 编辑器被压在 `design/` 之下：`features/design/blueprint/...`，比另外两个编辑器深一层。
3. `design/` 容器目前只承载一个 feature 子树（`blueprint/`）加一个 inspector 子树（`design/inspector/`），属于 layer-of-one。

Inspector 的位置进一步放大了这个不对称：

1. Inspector 内容库（74 个文件）位于 `features/design/inspector/`，与 `features/design/blueprint/` 平级。
2. 但 Inspector 在产品语义上是 Blueprint 编辑器的一块面板（与 Canvas、Sidebar、ComponentTree 同级），不是一个独立 feature。
3. 实际消费关系也证实了这点：Inspector 的外部引用方只有 Blueprint（`design/blueprint/editor/components/Inspector/BlueprintEditorInspector.tsx` 薄壳与 `useBlueprintEditorInspectorController`），NodeGraph 和 Animation 编辑器都不引用它。
4. 这种"内容库与 feature 平级"的摆法原本隐含"可被多编辑器复用"的意图，但当前没有任何复用规划。

此外，Blueprint 编辑器内部存在一层纯分组的 `editor/components/` 目录：

1. `features/design/blueprint/editor/components/` 下全是 UI 组件文件夹（Canvas / Sidebar / ComponentTree / Inspector / ViewportBar / AddressBar / Assistant / SaveIndicator），不承担任何领域区分职责。
2. 这导致核心组件路径过深，例如画布主组件位于 `features/design/blueprint/editor/components/Canvas/BlueprintEditorCanvas.tsx`，距 feature 根 5 层。

## 决策目标

1. 让 Blueprint、NodeGraph、Animation 三个编辑器在 `features/` 下真正平级。
2. 让 Inspector 回归"Blueprint 编辑器的一块面板"语义，与 Canvas 同层。
3. 消除 Blueprint 编辑器内部纯分组的目录层级，缩短核心组件路径。
4. 不引入运行时逻辑变更，目录调整应为纯机械的路径迁移。

## 决策内容

### 1) 移除 `design/` 层，Blueprint 提升为顶层 feature

- `features/design/blueprint/` → `features/blueprint/`
- 迁移完成后 `features/design/` 目录不再存在

理由：Blueprint 编辑器自身就是一个 feature，`design/` 容器只有一个 feature 子树，是冗余包装层。三个编辑器平级后，与 `animation/`、`development/` 的目录约定一致（根组件位于 feature 根下）。

### 2) 保留 `editor/` 子层，拍平 `editor/components/`

保留 `features/blueprint/editor/`。Blueprint 体量大（180+ 文件），`editor/`（编辑器 UI 与控制逻辑）与 `data/` / `external/` / `layoutPatterns/`（非 UI 领域库）的区分在该体量下有导航价值。`animation/` 不需要此层是因为它体量小且没有独立领域库。

拍平纯分组层 `editor/components/`：

- `editor/components/Canvas/` → `editor/canvas/`
- `editor/components/Sidebar/` → `editor/sidebar/`
- `editor/components/ComponentTree/` → `editor/componentTree/`
- `editor/components/ViewportBar/` → `editor/viewportBar/`
- `editor/components/AddressBar/` → `editor/addressBar/`
- `editor/components/Assistant/` → `editor/assistant/`
- `editor/components/SaveIndicator/` → `editor/saveIndicator/`

`editor/components/Inspector/`（薄壳）按下一节合并进 inspector 子系统。

### 3) Inspector 迁入 Blueprint，与 Canvas 平级

- `features/design/inspector/`（74 文件内容库）→ `features/blueprint/editor/inspector/`
- `features/design/blueprint/editor/components/Inspector/BlueprintEditorInspector.tsx`（薄壳）→ `features/blueprint/editor/inspector/BlueprintEditorInspector.tsx`（与内容库合并到同一目录）

理由：Inspector 是 Blueprint 编辑器的一块面板，不是独立 feature。薄壳（1 个文件）与内容库（panels / fields / tabs / classProtocol 等）合并到同一 `inspector/` 目录，避免重建"壳 vs 内容"的二级拆分。

## 直接影响

### 目录迁移

使用 `git mv` 保留 rename 历史。推荐顺序（避免路径冲突）：

1. 先拍平 blueprint 内部：`git mv features/design/blueprint/editor/components/<X> features/design/blueprint/editor/<x>`，逐个组件迁移。
2. 合并 inspector 薄壳：`BlueprintEditorInspector.tsx` 暂留在原位，待第 3 步统一安置。
3. `git mv features/design/inspector features/design/blueprint/editor/inspector`，再把薄壳 `git mv` 进同一目录。
4. `git mv features/design/blueprint features/blueprint`。
5. 删除空的 `features/design/`（含 `__tests__`，先迁出 `BlueprintGraphExecutor.test.ts` 到 `features/blueprint/__tests__/`）。

### import 路径更新

- 所有 `@/editor/features/design/inspector/...` → `@/editor/features/blueprint/editor/inspector/...`
- 所有 `@/editor/features/design/blueprint/...` → `@/editor/features/blueprint/...`
- barrel re-export（原 `editor/components/index.ts`、各组件 `index.ts`）随目录调整同步

影响文件量大（180+ 文件中的 import），但纯机械替换，无语义变更。

### 文档同步

- `CLAUDE.md` "Repository Map" 与 "Web Editor Areas" 段：`features/design` → `features/blueprint`，描述里"Blueprint editor, palette, tree, canvas, inspector, resources, external libraries"需更新路径。
- `AGENTS.md` 中对应描述同步。
- `.trellis/spec/` 若有 design feature 描述需同步（执行前需 `python ./.trellis/scripts/get_context.py --mode packages` 核对）。

### 测试位置

- `features/design/__tests__/BlueprintGraphExecutor.test.ts` → `features/blueprint/__tests__/BlueprintGraphExecutor.test.ts`
- inspector 内部各 `__tests__/` 随目录迁移

## 取舍说明

### 为什么不保留 `design/` 作为未来"第二个设计器"的容器

不采用，原因：

1. 目录容器应反映当前现实，不为假设的未来预留。当前 `design/` 只有一个 feature 子树。
2. 若未来真出现第二个设计器：
   - 复用 inspector → 把 inspector hoist 到 `features/shared/` 或 `editor/` 根（一次机械改动）。
   - 不复用 inspector → 新设计器独立 feature 文件夹，无需动现有结构。
3. 两种未来都不需要现在这个 `design/` 容器。保留它的代价是持续的路径深度与语义模糊，而 hoist-later 是一次性低成本操作。

### 为什么不把 `development/` / `animation/` 重命名为 `graphEditor/` / `animationEditor/`

不采用，原因：

1. 这两个名字是既定约定，AGENTS.md / CLAUDE.md / 路由 / git 历史都认这俩名。
2. 重命名收益仅为命名整齐，代价是大量 import 改动 + 文档同步 + git history 漂移。
3. 本决策的核心目标是修不对称，而修不对称可以通过"把 Blueprint 提上来"达成，不需要"把另外两个改名字"。

### 为什么保留 `editor/` 子层而不完全拍平对齐 `animation/`

Blueprint 有 `data/` / `external/` / `layoutPatterns/` 等非 UI 领域子系统，`editor/` 区分"编辑器 UI 与控制"和"领域库"，在该体量下有真实导航价值。`animation/` 没有这层是因为它没有同规模的独立领域库。结构对齐应以"各自体量下的合理分层"为准，不以字面层数相同为准。

### 为什么把 inspector 薄壳与内容库合并到同一目录

薄壳（`BlueprintEditorInspector.tsx`）只有 1 个文件，是 inspector 子系统的入口；内容库（panels / fields / tabs / classProtocol）是子系统的主体。分开放会重建"壳 vs 内容"的二级拆分，增加无谓的目录跳转。合并后 `BlueprintEditorInspector.tsx` 作为 `inspector/` 目录的入口文件，与内部各子目录平级，是最自然的摆法。

## 非目标

1. 本决策不改任何运行时逻辑、状态模型或组件 props。
2. 本决策不重命名 `animation/` / `development/`。
3. 本决策不调整 inspector 内部的 panels / fields / tabs / classProtocol 子结构（那是 ADR 21 的范畴）。
4. 本决策不拆分 `ViewportBar`（当前一个组件承担模式切换 / 视口尺寸 / zoom / 设备预设四件事），留作后续单独决策。
5. 本决策不合并 viewport 配置数据（当前散在 `editor/model/data.ts` 与 `data/viewport.ts`），留作后续清理。
