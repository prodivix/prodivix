## 决策背景

NodeGraph 已从协议占位进入正式编辑器实现。当前代码库已经基于 React Flow（`@xyflow/react`）建立稳定的 NodeGraph 编辑器主链路：

1. 画布与交互入口位于 `apps/web/src/editor/features/development/reactflow/`。
2. NodeGraph 编辑器已包含节点类型、连线动作、画布状态、分组布局、项目快照、PIR 序列化和测试。
3. 执行链路已经通过 `apps/web/src/core/executor/nodeGraph/` 接入运行时。

因此，NodeGraph 不再处于 Rete.js Parking Lot 阶段。React Flow 是当前和可预见阶段的长期技术选型。

## 决策内容

采用 React Flow 作为 Prodivix NodeGraph 编辑器的长期画布与交互底座。

核心原则：

1. React Flow 负责节点图画布、节点/边交互、视口、选择、拖拽和连接 UI。
2. Prodivix 自己维护 NodeGraph 的稳定语义模型、PIR 序列化、port 约束、执行器绑定和诊断。
3. React Flow 的内部状态不得成为 Workspace VFS 或 PIR 的权威事实源。
4. 写入工作区的变更必须通过 Workspace Command / Intent / Patch 链路。
5. code-owned 能力必须通过 Code Authoring Environment 和 Authoring Symbol Environment 接入，不在节点局部状态保存裸代码字符串。

## 架构边界

### React Flow 承载

1. 节点图画布渲染。
2. 节点与边的视觉交互。
3. viewport、selection、drag、connect、context menu 等编辑体验。
4. 与 React 组件体系内的编辑器 UI 组合。

### Prodivix Core 承载

1. NodeGraph 文档模型与 Workspace VFS 投影。
2. control-flow / data-flow port 语义与连线约束。
3. PIR 序列化和反序列化。
4. NodeGraph 执行器和运行时诊断。
5. 与 Code Authoring Environment 的 CodeReference / CodeArtifact 绑定。
6. 与 LLM、Inspector、Issues、Git Projection 的稳定接口。

## 与插件系统的关系

React Flow 是 core 当前内置的 NodeGraph 编辑器底座。按照 ADR 29 的长期方向，未来插件系统可以提供 `nodeGraphExtension` 扩展点，让插件贡献节点类型、port schema、executor 绑定和调试视图。

这并不意味着 React Flow 立即迁出 core。插件宿主稳定前，React Flow 继续作为官方内置实现存在。

若未来需要把 React Flow 相关能力拆为官方插件，应满足：

1. Extension Host 能支持编辑器级 contribution。
2. NodeGraph 模型、命令、诊断和执行器接口已稳定。
3. 拆分不会破坏现有项目的 PIR / Workspace 兼容性。

## 验收标准

- [x] NodeGraph 编辑器基于 React Flow 建立正式实现入口。
- [x] React Flow 状态可投影到 Prodivix NodeGraph / PIR 语义模型。
- [x] 连线和节点操作受 Prodivix port 语义约束。
- [x] NodeGraph 执行器与运行时链路存在。
- [ ] 所有 NodeGraph 写入统一进入 Workspace Command / Intent / Patch。
- [ ] React Flow 相关实现与插件扩展点边界在 ADR 29 后续阶段继续收敛。
