## 决策背景

Prodivix 的核心产品形态是三编辑器架构：

1. Blueprint 负责 UI graph、布局、组件属性、事件入口和样式挂载。
2. NodeGraph 负责逻辑图、端口、连线、执行语义和自定义执行函数。
3. Animation 负责 timeline、track、binding、filter 和关键帧。

但前端项目中存在大量 **code-owned** 内容：它们不适合被普通 PIR 结构完整表达，也不应被三编辑器强行可视化。例如：

- 复杂 event handler、表单校验、数据转换和业务服务。
- NodeGraph 自定义 executor、条件函数和外部运行时代码。
- Animation easing、timeline script、GLSL / WGSL、复杂动效函数。
- React Flow、CodeMirror、Monaco、Three.js、图表、富文本等复杂库的 adapter。
- Workspace 中的普通 TS/TSX/CSS/JSON/Markdown 文件。

Prodivix 将代码作者态定义为三编辑器共享的基础设施。Canonical Workspace code document 持久化 code-owned 内容，Code Authoring Environment 提供编辑体验、投影、语言能力与引用协作；PIR 保持结构化 UI 语义边界。

## 决策目标

1. 明确 Prodivix 仍是三编辑器可视化架构；code-owned 内容由 Canonical Workspace code document 持久化，并通过共享 Code Authoring Environment 编辑。
2. 让代码成为 Prodivix 项目模型中的一等公民：可编辑、可引用、可诊断、可预览、可被 AI 修改。
3. 保持 PIR 的边界：PIR 引用代码，但不吞并代码源码和复杂库内核。
4. 为 Blueprint、NodeGraph、Animation、Inspector、Resources、AI 和 Issues 面板提供统一代码入口。
5. 让 Code Authoring Environment 以 Code Semantic Provider 向 Workspace Semantic Index 发布代码符号、作用域、引用和诊断，并消费同一索引中的跨领域语义。

## 核心决策

### 1) Code Authoring Environment 是共享底座，不是第四编辑器

Prodivix 保持三编辑器架构：

```text
Blueprint Editor
NodeGraph Editor
Animation Editor
```

Code Authoring Environment 作为三者共同调用的代码作者态底座：

```text
Blueprint / NodeGraph / Animation / Inspector / Resources / AI / Issues
    -> Code Authoring Environment
    -> Workspace VFS
    -> CodeArtifact / CodeReference / CodeSlot

Code Authoring Environment / Language Service
    <-> Code Semantic Provider
    <-> Workspace Semantic Index
```

规则：

1. 不把 Code Authoring Environment 描述为第四个并列业务编辑器。
2. 三编辑器遇到 code-owned 边界时，通过统一入口打开代码工作区、代码片段或符号选择器。
3. Code Authoring Environment 不直接拥有 Blueprint、NodeGraph 或 Animation 的领域语义；领域语义由对应编辑器或 provider 提供。
4. Canonical Workspace 拥有 code document；Code Authoring Environment 负责其编辑体验、CodeArtifact 投影、代码片段、语言服务、代码语义贡献、引用、诊断和定位体验。它不拥有 Route、PIR、Component、Collection、NodeGraph 或 Animation 的全局 identity 与 visibility policy。
5. `@prodivix/code-language` 拥有具体语言 adapter 与 revision-bound capability session；`@prodivix/authoring` 只拥有稳定 contract 和 composition，不依赖 TypeScript、CSS 或 LSP 实现对象。

### 2) code-owned 不等于黑盒放弃

code-owned 表示代码是事实源，但 Prodivix 仍提供作者态能力。

| 能力 | Code Authoring Environment 职责                           |
| ---- | --------------------------------------------------------- |
| 编辑 | 文件树、tabs、dirty state、Monaco / CodeMirror、保存      |
| 引用 | `CodeReference` 指向 artifact、export、symbol 或 span     |
| 诊断 | TypeScript、lint、adapter contract、PIR 引用断裂、runtime |
| 定位 | 从 PIR 节点跳到代码，从诊断跳到 source span               |
| 预览 | 运行时 preview、sandbox、错误边界                         |
| AI   | 基于上下文生成、解释、修复、重构代码 patch                |
| 集成 | 绑定 event handler、executor、adapter、easing、CSS        |

规则：

1. 对 code-owned 内容，Prodivix 不承诺完整可视化编辑内部实现。
2. Prodivix 必须能管理 code-owned 内容的入口、引用、诊断和生命周期。
3. 代码必须进入 Workspace VFS 或稳定 CodeArtifact，而不是散落在 UI 局部状态中。
4. 代码能力必须能被 Issues 面板、AI 和相关编辑器定位。

### 3) Workspace VFS 是作者态文件系统

Workspace VFS 不只是 PIR 多文档容器。它是 Prodivix 项目的作者态文件系统，负责承载 PIR 文档、NodeGraph 文档、Animation 文档、代码文件、样式、shader、adapter 和普通 workspace module。

长期形态：

```text
/
├─ pages/
│  └─ home.pir.json
├─ components/
│  └─ dialog.pir.json
├─ nodegraphs/
│  └─ fetch-user.graph.json
├─ animations/
│  └─ intro.animation.json
├─ src/
│  ├─ actions/openDialog.ts
│  └─ node-executors/fetchUser.ts
├─ styles/
│  └─ home.mounted.css
└─ shaders/
   └─ wave.glsl
```

规则：

1. VFS 路径直接等于用户代码路径和作者心智路径，例如 `/src/actions/openDialog.ts`。
2. `.prodivix/...` 可以保存 workspace manifest、route manifest 和内部索引，但不吞并用户代码路径模型。
3. `WorkspaceDocument.id` 是稳定身份，不随移动、重命名或路径调整改变。
4. `WorkspaceDocument.path` 是用户可理解的路径，用于文件树展示、Git 导出和错误文案。
5. PIR、NodeGraph、Animation 和 code document 在 VFS 中并行存在，但事实源边界不同。
6. PIR / NodeGraph / Animation 负责可视化结构事实源；code document 负责 code-owned 内容事实源。
7. Code Authoring Environment 消费 VFS 中的 code document，并将其投影为 `CodeArtifact`。

前后端 Workspace 已支持 `code` document、内容校验、Command/Transaction、Atomic Commit 与 CodeArtifact 投影。

### 4) CodeArtifact 是 Workspace code document 的作者态投影

CodeArtifact 不与 WorkspaceDocument 竞争事实源。VFS 中的代码文件采用：

```text
CodeArtifact.id = WorkspaceDocument.id
```

```ts
type CodeArtifact = {
  id: WorkspaceDocumentId;
  path: string;
  language: CodeArtifactLanguage;
  owner: CodeArtifactOwner;
  source: string;
  revision: string;
};
```

规则：

1. `id` 用于稳定引用、诊断、历史和重命名安全。
2. `path` 用于用户心智、文件树、导出和显示，不作为稳定主键。
3. 一个 workspace code document 默认投影为一个 CodeArtifact。
4. 未来可以支持一个 document 内多个 artifact，但不得改变 `documentId` 作为稳定身份的基本原则。
5. `revision` 来源于 workspace document 的 content revision 或等价版本，不规定具体生成算法。

### 5) CodeReference 是持久化代码引用

PIR、NodeGraph、Animation 和 TriggerBinding 中的代码引用必须使用稳定身份，不依赖路径或临时名称。

长期模型：

```ts
type CodeReference = {
  artifactId: string;
  exportName?: string;
  symbolId?: string;
  sourceSpan?: SourceSpan;
};
```

规则：

1. `artifactId` 是主引用，默认等于 `WorkspaceDocument.id`。
2. `exportName` 适用于 TS / JS module export。
3. `symbolId` 表达 artifact 内稳定符号身份，适用于 expression、CSS symbol、shader entry、adapter entry 等非标准 export 场景。
4. `sourceSpan` 只用于定位辅助，不作为唯一身份。
5. `path` 可以作为显示和恢复线索，但不得成为持久化引用主键。
6. `{ name, scopeId }` 这类轻引用可以作为 resolver 查询输入，但不作为 PIR / slot / trigger 的长期持久化模型。

CodeReference 断裂、签名不匹配、作用域不可见等问题使用 `COD-xxxx` 诊断。

### 6) Code slots 是连接代码能力的稳定边界

三编辑器不应直接保存任意代码字符串。它们通过明确 code slot 声明“这里可以接入代码能力”。Code slot 是能力声明，不是代码内容，也不是事件动作本身。

规则：

1. 每个 code slot 必须声明 owner、输入、输出、能力约束和诊断落点。
2. slot 的绑定值应是 `CodeReference`，不是裸字符串。
3. slot contract 不保存代码源码，也不负责保存所有绑定状态。
4. slot 的签名校验属于作者态诊断，通常使用 `COD-xxxx`。
5. slot owner 删除后，相关 CodeArtifact 默认标记为 orphan，不自动删除。
6. orphan artifact 应保留在 VFS 中，并提供删除、重新绑定或转为 workspace module 的用户动作。

示例：

```ts
type CodeSlotContract = {
  id: string;
  ownerRef: DiagnosticTargetRef;
  kind:
    | 'event-handler'
    | 'validator'
    | 'node-executor'
    | 'animation-function'
    | 'animation-script'
    | 'shader'
    | 'external-adapter'
    | 'mounted-css'
    | 'route-loader'
    | 'route-action'
    | 'route-guard'
    | 'route-runtime'
    | 'workspace-module';
  inputTypeRef?: string;
  outputTypeRef?: string;
  capabilityIds: string[];
  defaultPlacement: DiagnosticPlacement[];
};
```

`DiagnosticPlacement` 使用诊断展示契约中的落点模型，不在 code slot 内另起一套展示协议。

### 7) CodeSlotRegistry 聚合 slot 与只读 binding projection，不拥有代码和绑定

CodeSlotRegistry 是 Code Authoring Environment 的聚合入口，用于查询当前上下文有哪些代码插槽。

```ts
type CodeSlotProvider = {
  id: string;
  source: AuthoringSource;
  listSlots(context: AuthoringContext): CodeSlotContract[];
  getSlot(id: string): CodeSlotContract | null;
  listBindingProjections(
    context: AuthoringContext
  ): CodeSlotBindingProjection[];
  getBindingProjection(id: string): CodeSlotBindingProjection | null;
};

type CodeSlotRegistry = {
  register(provider: CodeSlotProvider): void;
  unregister(providerId: string): void;
  listProviders(): CodeSlotProvider[];
  listSlots(context: AuthoringContext): CodeSlotContract[];
  getSlot(id: string): CodeSlotContract | null;
  listSlotsByOwner(ownerRef: DiagnosticTargetRef): CodeSlotContract[];
  listBindingProjections(
    context: AuthoringContext
  ): CodeSlotBindingProjection[];
  getBindingProjection(id: string): CodeSlotBindingProjection | null;
  listBindingProjectionsByArtifact(
    artifactId: string
  ): CodeSlotBindingProjection[];
};
```

分工：

| 模块                             | 职责                                                  |
| -------------------------------- | ----------------------------------------------------- |
| CodeSlotRegistry                 | 哪里可以接入代码，以及领域 binding 的只读查询投影     |
| CodeArtifactProviderRegistry     | 代码实际在哪里                                        |
| Workspace Semantic Index         | 全项目有哪些 symbol、引用能否解析、影响与诊断落在哪里 |
| PIR / NodeGraph / Animation 文档 | 保存领域结构和 slot binding                           |

规则：

1. Registry 通过 provider 聚合 slot，不直接扫描编辑器内部 store 或 UI 结构。
2. Registry 不保存源码。
3. Registry 不集中保存所有 binding；binding 由对应领域文档保存，provider 只生成 revision-bound、可重建的只读 projection。
4. Registry 可以按 owner、surface、capability 查询 slot，用于 Inspector、Issues、AI 和 Code Editor 入口。
5. CodeSlotProvider 可以随模块内部重构而替换，只要输出的 `CodeSlotContract` 稳定。

绑定示例：

```ts
type CodeSlotBinding = {
  slotId: string;
  reference: CodeReference;
};

type CodeSlotBindingProjection = Readonly<{
  binding: CodeSlotBinding;
  ownerRef: DiagnosticTargetRef;
  semanticReferenceId: string;
}>;
```

`CodeSlotBinding` 通常保存在对应领域文档中，例如 PIR node event、NodeGraph executor field、Animation binding field 或外部 adapter 配置。`CodeSlotBindingProjection` 只把该保存态绑定连接到同一 revision 的 semantic reference；二者都不是 registry 的全局保存状态。

### 8) Trigger / Action 不等于 CodeSlot

Trigger 表示入口，Action 表示事件要做什么，CodeSlot 只表示触达代码能力时的边界。Prodivix 不应把所有 trigger 都降级成 JavaScript handler。

```ts
type TriggerBinding =
  | { kind: 'open-url'; href: string }
  | { kind: 'navigate-route'; routeId: string }
  | { kind: 'run-nodegraph'; documentId: string; inputMapping?: unknown }
  | {
      kind: 'play-animation';
      documentId: string;
      timelineId: string;
      command: 'play' | 'pause' | 'seek';
    }
  | { kind: 'call-code'; slotId: string; reference: CodeReference };
```

规则：

1. 打开链接、路由跳转、执行节点图、播放动画等非代码 trigger 继续作为结构化 action 存在。
2. 只有触达 event handler、executor、adapter、mounted CSS、shader、timeline script 等 code-owned 内容时才进入 CodeSlot / CodeReference。
3. NodeGraph 和 Animation 仍是可视化运行时；其中的复杂 executor、transform、easing、shader 等代码叶子节点才属于 code-owned。
4. AI 生成行为时应优先使用结构化 action；只有确实需要代码能力时才创建或绑定 CodeReference。

### 9) Code Authoring Environment 是 Workspace Semantic Index 的 Provider 与 Consumer

`specs/decisions/25.authoring-symbol-environment.md` 定义 Workspace 级 Symbol、Scope、Reference、TargetRef、SourceSpan 与查询语义。Code Authoring Environment 只拥有其中的代码贡献和代码作者体验：

```text
Code Authoring Environment
├─ Workspace Code Editor
├─ Code Artifact / Reference / Slot Registry
├─ Language Service Adapter
├─ Code Semantic Contribution Provider
│  └─ complete code symbols / scopes / imports / reference facts
├─ Code Language Capability Provider
│  └─ native resolution / definition / reference / completion / rename
├─ Diagnostic Bridge
└─ AI Code Patch Bridge
        <-> Workspace Semantic Index
        <-> other domain semantic providers
```

规则：

1. Language Service 和 LSP object 不成为跨领域 contract。
2. Code Semantic Contribution Provider 发布完整 facts；Code Language Capability Provider 保留语言原生 lexical/module/type resolution，并把结果映射为 Workspace identity、统一 resolution status、`DiagnosticTargetRef` 和 `SourceSpan`。
3. Code Authoring Environment 消费 Workspace Semantic Index，以解析 Route、Component、Collection、NodeGraph、Animation、Token 和 Asset 等合法外部符号。
4. 跨 provider identity、聚合结果和跨领域 visibility/capability policy 由 Workspace Semantic Index 统一执行；Index 不重新实现 TypeScript/CSS/shader 的语言原生解析规则。
5. 三编辑器和 Code Editor 通过稳定查询接口消费语义，不扫描其他编辑器内部结构。

当前 TypeScript/JavaScript 与 CSS/SCSS provider 已以 immutable `CodeLanguageSession` 落地。session 绑定 `SemanticSnapshotIdentity` 与 artifact revisions，统一提供 definition、references、completion、diagnostics、hover、rename proposal 和 semantic contribution；Code Editor 与 Issues 消费同一 session 结果。CSS/SCSS 的 selector、keyframes、variable 与 mixin 通过稳定 Workspace semantic facts 发布，LSP object 不越过 package 边界。rename proposal 只产生带 revision 前置条件的 `CodeLanguageTextEdit`，最终由 Workspace planner 合并为可逆原子 Transaction。

### 10) 复杂库按能力等级接入，不逐库承诺完整可视化

Prodivix 对外部库采用能力分级：

| 等级      | 含义                                                             |
| --------- | ---------------------------------------------------------------- |
| Native    | Prodivix 内置能力，完整 Inspector / PIR / preview / codegen 支持 |
| Adapted   | 有 adapter，支持主要 props、slots、状态和 code slot              |
| Embedded  | 可运行和预览，但内部实现作为黑盒                                 |
| Code-only | 只能在代码中使用，Prodivix 只提供依赖、文件、诊断和构建层面支持  |

对 React Flow 这类复杂领域库，合理形态是：

```text
Embedded graph engine + Adapted node-card surface + Code-owned graph semantics
```

规则：

1. Prodivix 不追求可视化编辑所有前端库。
2. 对复杂领域引擎，优先适配能力类型，而不是适配库海。
3. adapter 可以暴露可视化 surface，例如 React Flow 的节点卡片模板。
4. 引擎内核、连接规则、性能边界、私有状态仍由代码拥有。

### 11) PIR 引用代码，不吞并代码

PIR 保存态不应内联完整代码文件或复杂库内部状态。PIR 只保存结构化引用。

规则：

1. PIR 中的 event、binding、adapter、executor、animation hook 应引用 `CodeReference`。
2. 代码源码保存在 Workspace VFS 的 code document 中，并由 CodeArtifact 投影给作者态环境。
3. CodeReference 断裂、签名不匹配、作用域不可见等问题使用 `COD-xxxx` 诊断。
4. Codegen 可以读取 CodeReference 生成目标框架代码，但不改变作者态代码归属。

## 典型场景

### Blueprint event handler

```text
Button node
  onClick -> TriggerBinding(call-code)
    slotId: blueprint.button-1.onClick
    reference: CodeReference(artifactId=code_open_dialog, exportName=openDialog)
```

流程：

1. 用户在 Inspector 中点击 `Create handler` 或 `Open handler`。
2. Code Authoring Environment 打开对应 artifact。
3. Workspace Semantic Index 结合 Code Semantic Provider 检查 handler 是否存在、是否在当前 scope 可见、签名是否匹配。
4. 诊断可以落到 Inspector 字段、Code Editor inline 或 Issues 面板。

### NodeGraph custom executor

```text
Custom Node
  executor -> CodeReference(artifactId=code_fetch_user_executor, exportName=execute)
```

流程：

1. NodeGraph 只保存 executor 引用和端口契约。
2. executor 源码由 Code Authoring Environment 编辑。
3. NodeGraph provider 暴露 graph input / output / port scope。
4. signature mismatch 进入 `COD-2003` 或后续更具体码位。

### React Flow node card template

```text
React Flow NodeShell
  -> code-owned handles / selection / graph events
  -> Prodivix-authored NodeCardTemplate
```

规则：

1. NodeShell 由 adapter 代码拥有。
2. NodeCardTemplate 可以是 PIR fragment 或生成的 React component。
3. React Flow `Handle`、edge validation、node id 和 graph events 不由模板任意创建。
4. 模板只能消费只读 scope 和受控 slot。

### 非代码 trigger

```text
Button node
  onClick -> TriggerBinding(open-url)
    href: https://example.com

Button node
  onClick -> TriggerBinding(run-nodegraph)
    graphId: graph_fetch_user

Button node
  onClick -> TriggerBinding(play-animation)
    timelineId: timeline_intro
    command: play
```

规则：

1. 这些 trigger 不需要 CodeSlot。
2. 它们仍应进入 PIR / NodeGraph / Animation 的结构化 action 模型。
3. 当 action 的某个叶子能力需要用户代码时，才通过 CodeSlot 连接 CodeReference。

## 直接影响

### 架构表述

项目架构明确为：

```text
三编辑器 + 共享 Code Authoring Environment
```

其中三编辑器仍是主要产品面，Code Authoring Environment 是跨编辑器基础设施。

### Inspector

Code Tab 不应只是 triggers 的容器。它应成为当前节点相关 code slots 和 code references 的入口，例如：

- event handlers
- mounted CSS
- external adapter hooks
- custom expressions

### NodeGraph

NodeGraph 不直接内联复杂 executor 源码。它通过 code slot 绑定 executor，并把 port / graph scope 作为领域 contribution 发布到 Workspace Semantic Index。

### Animation

Animation 不把所有复杂 easing、shader、timeline script 都塞入可视化 timeline。复杂逻辑通过 code slot 进入 Code Authoring Environment。

### AI

AI 生成代码时不直接改任意文件。它应基于 code slot、CodeArtifact、Workspace VFS 和 Authoring Environment 生成可审阅 patch。

### Backend Workspace

后端 Workspace 将 code document 与 PIR、NodeGraph、Animation 文档作为并列的 canonical document type。

要求：

1. `WorkspaceDocumentType` 包含 `code`。
2. code document 使用明确的内容模型、校验策略和 Command/Transaction 写入策略。
3. 支持 VFS 路径作为用户代码路径，例如 `/src/actions/openDialog.ts`。
4. 保持 `WorkspaceDocument.id` 作为稳定身份，不随 path 改变。
5. `CodeArtifact.id` 直接使用 `WorkspaceDocument.id`。

code document 使用 JSON wrapper 表达语言、源码与元数据：

```ts
type WorkspaceCodeDocumentContent = {
  language: CodeArtifactLanguage;
  source: string;
  metadata?: Record<string, unknown>;
};
```

纯文本内容扩展继续沿用 `WorkspaceDocument.id`、`path` 和 CodeArtifact 投影语义。

## 当前能力与 G1 交付

当前能力：

1. 前后端 Workspace 均支持 `code` document、JSON wrapper 校验、Command/Transaction 与 Atomic Commit。
2. Workspace code document 已能投影为以 artifact identity 为核心的 CodeArtifact / CodeReference。
3. `@prodivix/authoring` 已实现 CodeSlotContract/Provider/Registry、领域 binding projection、semantic relation query 与 TriggerBinding 基础。

## 实施顺序

### 已实现基础

1. 三编辑器的 code-owned 能力共享 CodeArtifact、CodeReference、CodeSlot 与 authoring registry 基础。
2. 前后端 Workspace 支持 `code` document，并投影为 CodeArtifact。
3. 持久化 CodeReference 以 `artifactId` 为主引用。
4. `CodeSlotContract`、`CodeSlotProvider`、`CodeSlotRegistry` 与 TriggerBinding / ActionContract 边界已经形成。
5. TypeScript/JavaScript/CSS/SCSS/GLSL/WGSL 使用 revision-bound Code Language session；semantic contribution、Code Editor 与 Issues 使用同一 snapshot identity。
6. Language rename proposal 通过 Workspace Transaction planner 进入 History、Outbox 与 Atomic Commit 写入链路。
7. Blueprint、Route、NodeGraph 与 Animation CodeSlot 通过 Workspace composition 生成只读 binding projection，并复用 Semantic Index 的 definition/reference/impact 查询。

### 已实现 Workspace code editor integration

1. 让 source/Git projection 中的 code path 直接等于用户代码路径。
2. 将 Workspace VFS 中的代码文件和 CodeArtifact 统一接入代码工作区。
3. Code Editor 通过 Code Language session 消费 TypeScript/JavaScript/CSS/SCSS/GLSL/WGSL completion、definition、references、hover 与 diagnostics；同一 session 提供可进入 Workspace Transaction planner 的 rename proposal。
4. Issues 通过统一 diagnostic provider snapshot 消费同一批 language diagnostics，并以 CodeArtifact `SourceSpan` 回跳源码。

## 长期不变量

1. Prodivix 不把所有代码和复杂库内核强行 PIR 化。
2. Code-owned 内容必须可被 Prodivix 编辑、引用、诊断和定位。
3. Workspace VFS 是作者态文件系统，VFS path 直接等于用户代码路径。
4. `WorkspaceDocument.id` 是稳定身份，`WorkspaceDocument.path` 是用户路径。
5. CodeArtifact 是 workspace code document 的作者态投影。
6. CodeReference 以 artifact identity 为核心，不以 path 或临时 name 为主键。
7. 三编辑器通过 code slot 与代码连接，不直接保存任意代码字符串。
8. Trigger / Action 不等于 CodeSlot；非代码 action 继续结构化存在。
9. owner 删除后 code artifact 默认 orphan，不自动删除。
10. Workspace Semantic Index 是跨领域共享查询层；Code Authoring Environment 是 Code Semantic Provider 与 consumer，不拥有全局索引。
11. adapter 暴露可视化 surface，但不泄漏复杂引擎私有状态。

## 非目标

1. 本决策不要求立即实现完整 IDE。
2. 本决策不要求把 Code Editor 作为第四个并列业务编辑器。
3. 本决策不要求深度适配所有外部库。
4. 本决策不定义目标框架最终代码文件布局。
5. 本决策不要求一次性实现完整文件系统 API、CRDT 或二进制资产管线。
