## 决策背景

Core package 抽取完成后，Web 侧仍同时存在三种 Workspace 表达：

1. `@prodivix/workspace` 的 indexed canonical snapshot；
2. Web API 使用 `tree + documents[]` 的 wire shape；
3. Zustand Store 展平保存的 `workspaceId`、`treeById`、`workspaceDocumentsById`、`routeManifest` 与 `pirDoc` 镜像。

三份状态会在路由切换、远端 mutation、离线项目恢复与编辑器局部持久化时产生不同步。尤其是 `pirDoc`、NodeGraph/Animation localStorage 和项目级 PIR fallback 都可能绕开 Workspace VFS、Command 校验与 revision 语义，成为第二事实源。

项目仍处于 alpha，本轮采用 Hard Cut：删除旧入口和 fallback，不保留双模型、兼容镜像或旧数据自动修复。

## 决策内容

### 1. 唯一 canonical WorkspaceSnapshot

进程内作者态只使用 `@prodivix/workspace` 的 `WorkspaceSnapshot`：

```ts
type WorkspaceSnapshot = {
  id: string;
  name?: string;
  workspaceRev: number;
  routeRev: number;
  opSeq: number;
  treeRootId: string;
  treeById: Record<string, WorkspaceVfsNode>;
  docsById: Record<string, WorkspaceDocument>;
  routeManifest: WorkspaceRouteManifest;
  activeDocumentId?: string;
  activeRouteNodeId?: string;
};
```

Web、Compiler 与其他消费者不得再声明自己的 `WorkspaceSnapshot`、`WorkspaceDocumentRecord` 或 `WorkspaceDocumentType`。派生视图通过 selector 或只读计算获得，不在 Store 中保存镜像。

### 2. Wire DTO 与 canonical snapshot 明确分层

后端 wire contract 保持面向传输的结构：

- `tree: { treeRootId, treeById }`；
- `documents: WorkspaceDocument[]`；
- `settings: Record<string, unknown>`；
- confirmed revisions 与 route manifest。

`decodeWorkspaceSnapshot` 是 wire 到 canonical 的唯一入口；`encodeWorkspaceSnapshot` 是 canonical 到 wire 的唯一出口。Mutation response 先解码为 `DecodedWorkspaceMutation`，再由 Store 应用到当前 snapshot。

Workspace settings 不并入 canonical snapshot。API 解码结果是 `{ workspace, settings }`；Web settings store 负责 settings，Workspace Store 只负责作者态 snapshot。LocalProjectStore 可以对外命名为 `workspaceSettings`，但持久化时仍写入同一个 wire envelope，不引入第四种数据模型。

### 3. Store 只保存一个 snapshot

Zustand Workspace slice 的权威字段是：

```ts
workspace: WorkspaceSnapshot | null;
```

Store 不再保存以下镜像：

- `pirDoc` / `pirDocRevision`；
- `workspaceId` / `workspaceRev` / `routeRev` / `opSeq`；
- `workspaceDocumentsById`；
- `treeRootId` / `treeById`；
- 独立 `routeManifest` / `activeDocumentId` / `activeRouteNodeId`。

capabilities、readonly、selection 等产品 UI 状态可以独立存在，但不得复制作者态内容。

### 4. confirmed revision 与本地 edit sequence 分离

`workspaceRev`、`routeRev`、`opSeq`、document `contentRev` 与 `metaRev` 只表示服务端已确认的 revision。即时本地 Command 不得伪造或递增这些值。

本地未确认的文档编辑由 `documentEditSeqById` 计数。它只用于脏状态、autosave 触发与 UI 协调，不进入 wire contract，也不能代替服务端 revision。服务端 mutation 到达后更新 canonical snapshot 中的 confirmed revisions；删除文档时同步移除对应 edit sequence。

### 5. 所有作者态写入进入 Command / Transaction

单文档或单作用域变更使用 `WorkspaceCommandEnvelope`，跨文档、RouteManifest、VFS 与 active document 的原子本地变更使用 `WorkspaceTransactionEnvelope`。

统一入口为：

- `dispatchWorkspaceCommand`；
- `dispatchWorkspaceTransaction`；
- 基于 Command 的 `updateActivePirDocument`；
- 远端 Intent/Patch/Atomic Operation Commit 返回的 decoded mutation。

Blueprint、Inspector、drag/drop、NodeGraph、Animation、Route 与 Resources 不得直接覆盖 snapshot 内容。Command/Transaction 必须经过 patch path policy、PIR/Route/VFS validator 与 target 校验。

本 ADR 的 Transaction 保证客户端内存中的多命令原子应用。远端原子性由 ADR 36 的 `/operations/commit` 单独定义；旧 `/batch` 从未具备该保证，不能用本地 Transaction 语义替它背书。

### 6. active document 必须与 route selection 对齐

显式选择 route 时，Route selector 在 composed route manifest 中解析目标 route：

- route 绑定 `pageDocId` 时，`activeDocumentId` 切换到该页面文档；
- route 不存在或未绑定有效页面时，不制造虚构文档，使用 canonical fallback；
- 创建、移动、绑定、解绑和删除 route 的变更与相关文档/VFS 变更通过 Command 或 Transaction 一起提交。

编辑器不得再用独立 route slice 镜像推断当前 PIR。

### 7. 拒绝 legacy、fallback 与局部事实源

以下行为被明确删除：

1. Workspace 加载失败后回退到 `project.pir` 或 `setPirDoc`。
2. API 返回非法 snapshot 时容错生成默认 PIR、默认文档或默认 VFS。
3. IndexedDB 旧 `pir` 字段、single-PIR 项目或损坏 snapshot 的自动迁移与修复。
4. NodeGraph/Animation 使用 localStorage 保存可编辑 authoring state。
5. 编辑器组件保存裸 PIR 或裸代码字符串并绕过 Workspace/Code Authoring Environment。

localStorage 仍可保存 tab、selection、panel、viewport 等纯 UI 偏好；它不得保存会参与编译、导出、同步或 Git projection 的作者态内容。

### 8. Workspace 加载门禁

编辑器只在以下条件满足后进入 authoring ready：

1. remote response 或 local persisted record 已通过 codec 解码；
2. Workspace、RouteManifest、PIR 文档和 VFS validation 通过；
3. canonical snapshot 已一次性写入 Store；
4. capabilities/readonly 状态已完成对应项目的装载或明确降级策略。

加载失败必须停留在 error/loading gate，不挂载携带默认 PIR 的编辑器，也不进行部分 hydration。切换项目时先清空旧 Workspace，防止跨项目状态泄漏。

## 非目标

以下同步能力不由本 ADR 定义完成状态；它们可由关联 ADR 在同一 canonical snapshot 基线上独立交付：

1. durable offline outbox；
2. 409 后自动 rebase、重放与冲突 UI；
3. 跨 tab、跨设备或多人实时协作；
4. 后端 Atomic WorkspaceOperation Commit；
5. confirmed revision 与 pending operation 的持久化恢复状态机。

这些能力由 ADR 07 与 `@prodivix/workspace-sync` 承担。Hard Cut 只保证它们建立在一个 snapshot、一个 wire codec 和一个写入入口之上。

截至 2026-07-12，ADR 07 已在该基线上完成 canonical 409、semantic diff3、Document bounded automatic rebase、conflict session 与 Code/NodeGraph review UI；`@prodivix/workspace-sync`、Web 与后端已完成 exact Commit planner、tree/route/document/mixed `/operations/commit` transport、单数据库事务与 strong replay。Durable outbox 仍由 ADR 07 的后续 Gate 收口，不属于本 ADR 的完成条件。

## 结果与影响

### 正向影响

- Web Store、API、IndexedDB 与 Core 不再各持有一套 Workspace 语义。
- revision 不再同时承担“服务端确认版本”和“本地编辑次数”两种含义。
- Route、PIR、NodeGraph 与 Animation 共享可校验、可回放的写入链路。
- outbox/rebase 可以基于 confirmed snapshot 和 Command log 建立，不必先解决三份状态漂移。

### 代价

- 旧 IndexedDB 数据和非法 wire response 会显式失败，用户需要重新创建或重新拉取项目。
- 调用方必须持有 canonical snapshot 才能解码并应用 mutation。
- Document conflict 已可自动 rebase 或进入显式 review；后端 Atomic Commit 已落地，durable outbox 的未完成状态仍必须显式暴露，本 ADR 不以 fallback 掩盖。

## 验收标准

- [x] Web 不再声明重复 Workspace snapshot/document/type。
- [x] API 与 IndexedDB 边界统一使用 workspace codec。
- [x] Zustand 只保存一个 canonical `workspace`，不存在 `pirDoc` 或 flattened authoring mirrors。
- [x] 本地 Command 保留 confirmed revisions，并由 `documentEditSeqById` 表示本地编辑推进。
- [x] Workspace Command 与 Transaction 都有 validator、path policy 和测试。
- [x] Route selection 会同步解析 active page document。
- [x] Blueprint、Inspector、drag/drop、NodeGraph、Animation、Route 与 Resources 写入已进入统一入口。
- [x] legacy project PIR fallback 与 NodeGraph/Animation authoring localStorage 已删除。
- [x] invalid remote/local wire data 会在 codec 边界失败。
- [x] 编辑器加载受 canonical Workspace gate 保护。
