## 背景

运行全部测试既不能说明为什么这些检查足够，也无法在大型 Workspace 中持续扩展。反过来，仅根据改动文件名
筛选少量测试会漏掉 Route、Component、Data、NodeGraph、Animation、CodeSlot 和 target projection 的跨领域影响。

G3 需要把“对这个 exact change 必须验证什么”建模为确定性、可审计的 Plan。Plan 不是手写任务列表，也不是
CI YAML；它由 exact revision range、Semantic Impact、Scenario coverage、VerificationPolicy、target matrix 和
provider capability 共同推导。

## 决策

新增 `@prodivix/verification`，拥有 transport-neutral current domain：

- `VerificationImpactSet`；
- `VerificationPolicy` 与 policy evaluation；
- `VerificationPlan`、cell/check DAG 和 plan digest；
- plan completeness、closure evaluation 与 stable failure taxonomy。

### Canonical policy 与 derived plan

Workspace 新增唯一 project-level typed document `verification-policy`，由 `core.verification` Command namespace
管理。它保存用户可审阅的目标矩阵、required check rules、预算、baseline/exemption policy 与 retention request，
但不保存某次运行结果。

`VerificationPlan` 是 exact revision-bound derived projection，不写回 Workspace。相同输入必须得到相同 canonical
plan 和 digest；输入包含显式 `policyEvaluationInstant`，用于判断 exemption/freshness，planner 禁止读取 ambient
current time。plan 可以持久化在 Verification service 作为 evidence provenance，但不能成为第二作者态。

```text
base/target revision vectors + operation identities
  -> semantic diff roots
  -> Workspace Semantic Index impact closure
  -> Scenario coverage reverse links
  -> VerificationPolicy evaluation
  -> target/control/provider matrix expansion
  -> deterministic VerificationPlan
```

### ImpactSet

`VerificationImpactSet` 至少记录：

- base 与 target Workspace/partition revision；
- root operations、changed documents/symbols/source spans；
- impacted Route、PIR document/node、Component/Collection、CodeArtifact/CodeSlot；
- impacted DataSource/Operation、NodeGraph/node、Animation/timeline/track；
- impacted BehaviorScenario/checkpoint/assertion；
- affected framework target、runtime zone、capability 和 risk flags；
- broaden reason 与 unresolved/stale facts。

Impact 只由 exact Semantic Index 和 domain diff provider 计算。若 provider missing、schema/provider set drift、
index stale、dynamic CodeSlot、adapted unknown source 或 unresolved reference 使精确闭包不可信，planner 必须扩大到
document、route、target 或全 Workspace suite。它绝不能以“不知道”为理由少跑检查。

### Policy

Policy 是声明式规则，不包含 shell command、供应商 token 或任意代码。规则可按以下事实选择检查：

- scenario id/tag/criticality；
- impacted domain/target/runtime zone/capability；
- change risk 与 ownership class；
- matrix axis：framework target、surface、browser engine、viewport、color scheme、motion、locale；
- check kind 与 required evidence trust class；
- timeout、attempt、artifact、parallelism 与 total cost budget。

稳定 check kind：

```ts
type VerificationCheckKind =
  | 'diagnostics'
  | 'build'
  | 'unit'
  | 'integration'
  | 'e2e'
  | 'visual'
  | 'accessibility'
  | 'performance'
  | 'security';
```

Policy 可以把 check 标记为 `required` 或 `advisory`。required check 的 skipped、missing、unsupported、blocked、
infrastructure-error、stale 或 incompatible evidence 都不能满足 closure。advisory 结果仍进入 Evidence，不能从 UI
和审计中消失。

### Plan DAG 与 matrix cell

Plan 由稳定 cell 和 check 组成：

```ts
type VerificationPlanCell = Readonly<{
  id: string;
  scenarioId?: string;
  targetId: string;
  surface: 'preview' | 'export' | 'ci';
  browserEngine?: 'chromium' | 'firefox' | 'webkit';
  controlsDigest: string;
  checkIds: readonly string[];
}>;
```

check 明确依赖关系、输入 digest、provider/capability、预算、required/advisory、选择原因和 expected evidence kind。
依赖安装、build、app start、fixture provision、scenario execution、report normalization 与 evidence promotion形成 DAG；
共享安全的 immutable build/install artifact，不共享 active process、fixture namespace 或可变浏览器状态。

Plan ID 由 canonical normalized input digest 派生，其中包含调用方提供并被记录的 `policyEvaluationInstant`；不在 planner
内部读取当前时间或生成随机 UUID。attempt/run identity 单独生成。

### Matrix expansion 与预算

Planner 在 policy 允许的 axes 上做确定性展开，并有硬性 cell/check/byte/cost 上限。超预算时返回
`VER-3004` blocked plan，列出导致爆炸的规则；不能任意截断最后若干 cell，也不能偷偷降级 required matrix。

最小 G3 closure matrix：

1. critical scenario 在 Preview、Export、CI 三个 surface 运行共享 semantic assertions；
2. React/Vite 与受控 Vue/Vite target 至少运行同一关键 CRUD/behavior scenario；
3. primary engine 运行完整 matrix，Firefox/WebKit 运行 policy 指定的 critical subset；
4. impacted Animation/Route transition 至少覆盖 full-motion 与 reduced-motion；
5. impacted route/component 至少执行 accessibility；声明 visual checkpoint 时执行兼容 baseline cell；
6. build/unit/integration/security/performance 由 change impact 与 policy 加入，不通过无关笛卡尔积扩大。

### Retry 与 flaky semantics

- infrastructure error 可以按 policy 进行有界 retry，每次产生独立 attempt 和完整 evidence lineage；
- deterministic assertion failure 默认不自动重试以寻找绿色结果；
- 显式 stability sampling 必须保留所有 attempt，结果不一致标记 `unstable` 并阻断 required closure；
- retry 不能覆盖、删除或重写先前 failure；
- mutation scenario 每次 attempt 使用新的 fixture namespace/idempotency scope，旧副作用不自动重放。

### Exemption

G3 允许 project-level、可审计的有限 exemption，用于 accessibility rule、visual region 或 advisory policy：

- 必须绑定 exact rule/check、stable target、reason、actor reference、createdAt 和 expiry；
- 不能包含自由 shell、selector 或 credential；
- 过期、target drift 或 policy 不允许时 fail closed；
- evidence 必须同时记录 raw finding 与 applied exemption，不得把 finding 从报告删除；
- critical security、Secret leak、unknown network、missing evidence 和 stale revision 不可豁免。

组织级审批与多人 review 属于 G5；G3 只冻结可供后续授权层消费的 exemption contract。

### Closure

`VerificationClosure` 是对 exact plan 的派生判断：每个 required cell/check 都有 current、compatible、满足 trust
要求且 outcome passed 的 Evidence。它绑定 target Workspace revision、plan digest 和 evidence digests；任一 Workspace
revision、Scenario、Policy、baseline、tool compatibility 或 required provider set 改变都会使 closure stale。

Closure 不写 Workspace，不自动允许发布，也不代表 G5 deployment approval；它为 G4/G5 提供可信输入。

## 拒绝的方案

### 只按文件 glob 选测试

拒绝。它看不到 typed reference、Route composition、Component instance、CodeSlot 和 target projection 影响。

### 所有改动永远跑全矩阵

拒绝。它无法扩展，也掩盖了 coverage 与 policy 缺口；unknown impact 应扩大，但已知安全影响应可精确选择。

### CI YAML 是 VerificationPlan

拒绝。YAML 属于 adapter/deployment，缺少 Workspace revision、semantic impact、scenario 和 evidence identity。

### 自动重跑失败直到通过

拒绝。它销毁失败证据并把 flaky 伪装成 passed。

## 后果

- Workspace 增加 `verification-policy` typed document 与 validator。
- Semantic Index 增加 Scenario coverage reverse query 和 domain diff/impact provider。
- Web、CLI 和 CI adapter 只消费 canonical Plan，不自行选择检查。
- G4 Agent proposal 必须附带 planner 生成的 Plan，不能自报“已测试”。
- G5 review/deploy 可以引用 Closure，但不得修改其结果。

## 验收

- [ ] Policy current/wire/codec、Command/Transaction 与 Backend validator 完成。
- [ ] 相同 revision/diff/index/policy/policyEvaluationInstant 得到 byte-stable plan digest。
- [ ] stale/missing/unknown semantic fact 保守扩大而不漏选。
- [ ] matrix expansion、DAG、预算和 provider capability fail closed。
- [ ] retries 保留所有 attempt；unstable 与 required skipped 不能通过 closure。
- [ ] Plan/Impact/Closure 不成为 Workspace 作者态或第二写路径。
