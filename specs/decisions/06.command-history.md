## 决策背景

Prodivix 的 Blueprint、NodeGraph、Animation、Code Authoring、Route 与 Workspace VFS 共享同一个作者态 Workspace。History 如果围绕组件局部 state、裸 PIR 或某一个编辑器建立，就无法正确表达跨文档和跨领域操作。

Command、Transaction 与 History 必须有单向边界：

- Command 描述并校验一个最小可逆变更；
- Transaction 将多个 Command 组成一次原子用户操作；
- History 只管理成功 Operation 的本地时间线；
- Outbox 负责 Operation 的远端投递、ACK、冲突与重放。

Command Apply 不依赖 History；History 可以调用 Command / Transaction Apply，但不得直接改写 WorkspaceSnapshot。

## 决策内容

### 1. History 记录 Operation

History Entry 不只保存单个 Command，而是保存统一 Operation：

```ts
type WorkspaceOperation =
  | {
      kind: 'command';
      command: WorkspaceCommandEnvelope;
      undoOf?: string;
      redoOf?: string;
      sourceOperationIds?: string[];
    }
  | {
      kind: 'transaction';
      transaction: WorkspaceTransactionEnvelope;
      undoOf?: string;
      redoOf?: string;
      sourceOperationIds?: string[];
    };
```

Transaction 是一次用户动作。它在 History、未来 Outbox 和 operation log 中都只能作为一条 Operation，不能把内部 Command 分散入栈。

### 2. Apply 成功后才能入史

统一顺序是：

```txt
Domain Intent
  -> Command / Transaction
  -> Apply + Validator
  -> Snapshot + editSeq + History（同一次 Store 提交）
  -> Outbox（后续阶段）
```

任何 Command、Transaction 或最终 Workspace validation 失败时，Snapshot、History、editSeq 与 Outbox 都不得变化。远端 ACK mutation 只更新 confirmed revision，不重复加入用户 History。

### 3. Scope 与 Workspace barrier

History scope 为：

```ts
type WorkspaceHistoryScope =
  | {
      kind: 'document';
      workspaceId: string;
      documentId: string;
      domain: 'pir' | 'nodegraph' | 'animation' | 'code';
    }
  | { kind: 'route'; workspaceId: string }
  | { kind: 'workspace'; workspaceId: string };
```

规则：

1. 单文档、单领域 Operation 使用 document scope。
2. 纯 Route Operation 使用 route scope。
3. 跨文档、跨领域或 VFS Operation 使用 workspace scope。
4. workspace scope Entry 是 barrier；document/route scoped undo 不得越过与自己有依赖的 barrier。
5. 编辑器可以提交一组候选 scope，History 必须从全局时间线选择最新合法 Entry，不能由 Web 自行扫描栈。

### 4. Undo / Redo 仍走 Apply Engine

Undo 不直接恢复快照，也不调用 UI setter：

- Command undo 交换 `forwardOps` / `reverseOps`；
- Transaction undo 逆序排列内部 Commands，再交换每个 Command 的 ops；
- undo/redo 每次生成新的 Operation ID；
- `undoOf` / `redoOf` 保存因果关系；
- inverse/replay 仍经过 target、path policy、reverse 与 domain validator。

失败的 undo/redo 不移动 History Entry。

### 5. Merge 与容量

Command / Transaction 的 `mergeKey` 是 domain helper 提供的合并提示，History 决定是否合并。

只有同时满足以下条件才合并：

1. 相邻 Operation；
2. 相同 scope；
3. 相同非空 `mergeKey`；
4. 位于 merge 时间窗口内，默认 750ms；
5. Command IDs 无冲突。

合并结果是一个 Transaction，因而 redo 按原顺序执行，undo 按反序执行。History 默认保留 80 条，并由全局 `undoSteps` 设置在 0～500 范围内调整。

合并 Transaction 使用独立 Operation ID，并保留全部 `sourceOperationIds`。任何成功的 undo/redo 都推进 replay sequence，关闭旧 merge group；因此“undo → redo → 继续编辑”不会把新编辑重新并入旧 Entry。

### 6. Selection 是 History replay 的临时投影

当前 `activeDocumentId` / `activeRouteNodeId` 位于 canonical WorkspaceSnapshot，同时也会被非历史的页面选择动作修改。

History replay 对它们采用以下规则：

1. Apply 前临时对齐原 Operation 所需的 selection baseline；
2. Operation 完成后优先恢复用户当前仍然有效的选择；
3. 当前选择已被 undo 删除时，使用 canonical document/route fallback。

因此“创建 Route → 用户切换页面 → undo 创建 Route”不会因 selection 漂移而失败，也不会强行跳回旧页面。

### 7. Store 生命周期

History 是进程内编辑会话状态，不属于 WorkspaceSnapshot、Wire DTO、IndexedDB VFS 或 Git projection。

- `setWorkspaceSnapshot` 与 `clearWorkspaceState` 清空 undo/redo；
- hydration 时保留 History 深度与 merge 配置；
- Command / Transaction dispatch 成功后原子记录；
- undo/redo 成功后原子更新 Snapshot、History 与受影响文档 editSeq；
- remote mutation 不入史、不清空已有 History。

同步请求已经带有本地 editSeq baseline 时，ACK 只覆盖请求发出后未再变化的文档内容；如果期间发生了继续编辑或 History replay，则保留当前作者态内容，只接受服务端 revision metadata。尚未接入 durable Outbox 的远端 CodeDocument 保存采用 ACK-first bridge：请求成功后由 Store 将 Command、confirmed Snapshot 与 History 一次提交，请求失败时草稿仍可重试。

### 8. 编辑器与代码文本边界

编辑器快捷键为：

- `Mod+Z`：undo；
- `Mod+Shift+Z`、`Ctrl+Y`：redo。

Blueprint 使用 PIR + Route + Workspace scopes；NodeGraph、Animation 使用各自 document scope + Workspace scope；Code Resources 使用选中 CodeDocument + Workspace scope。

input、textarea、contenteditable 和 `data-editor-native-history` 边界保留 CodeMirror/未来 Monaco 的原生文本 History。Workspace shortcut 只有在 replay 成功时才消费键盘事件。拖拽或未提交 code draft 期间暂停 Workspace undo。

## 同步边界

本 ADR 实现的是本地可逆时间线，不把以下能力塞进 History：

1. durable offline outbox；
2. confirmed revision 推进；
3. 409 rebase、冲突 UI 与多人协作；
4. 远端 Atomic WorkspaceOperation Commit。

这些能力消费相同的 WorkspaceOperation，但属于 Workspace Sync；实现状态分别由 ADR 07 与 ADR 36 记录。没有进入本地 Command/Transaction 的远端 Intent-only 操作不会被 History 伪造 reverse ops；它们必须迁入本地 Operation planner，或由服务端在同一 commit transaction 内返回权威 Operation。

Mounted CSS 首次创建必须最终规划为“新增 CodeDocument + 更新 PIR binding”的单一 WorkspaceTransaction，并通过 `/operations/commit` 持久化；不得再以旧 `/batch` 或两个独立请求宣称原子。
