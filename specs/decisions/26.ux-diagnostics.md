## 决策背景

Prodivix 既是可视化 IDE，也是页面与应用的生成环境。PIR、Editor、Code、Codegen 等诊断可以回答“结构是否有效”“编辑命令是否成功”“代码是否可解析”“导出是否失败”，但它们不能稳定表达“这个页面是否能被用户理解和使用”。

UX 质量问题有独立的长期语义：

1. 它们可能在 PIR、CSS、主题、组件 metadata、运行时 DOM、viewport 和交互状态共同作用后才出现。
2. 它们常常不阻断保存、预览或导出，却会影响真实用户完成任务。
3. 它们需要引用 WCAG、WAI-ARIA、项目设计策略和交互可用性规则。
4. 它们需要进入 Issues、Inspector、画布标记、导出前检查、AI 修复上下文和文档站。

如果把这些问题塞进 `EDT-xxxx`，会把“编辑器操作失败”和“产物体验质量问题”混在一起。如果塞进 `COD-xxxx`，会把“用户代码错误”和“跨层体验诊断”混在一起。如果直接复用 WCAG 或 TypeScript 等外部编号作为 Prodivix 主错误码，则无法承载 Prodivix 自己的定位、分组、严重程度、修复建议和长期兼容策略。

因此需要建立独立的 `UX-xxxx` 诊断域。

## 决策目标

1. 为可访问性、交互、布局、内容、视觉和体验检查器建立稳定 `UX-xxxx` 命名空间。
2. 明确 `UX` 与 `PIR`、`EDT`、`COD`、`PLG`、`GEN` 的边界。
3. 支持 WCAG、WAI-ARIA、Lighthouse、axe、TypeScript 等外部编号作为 evidence，而不是 Prodivix 主 code。
4. 让 UX 诊断能定位到 PIR node、Inspector field、route、viewport、theme token、CodeArtifact 或运行时 DOM snapshot。
5. 让 UX 诊断适合 Issues 面板、画布标记、导出前 gate、AI 修复和文档站。
6. 避免把纯审美偏好、临时建议或无法稳定检测的问题编码成错误码。

## 核心决策

### 1) 新增 `UX-xxxx` 诊断域

`UX-xxxx` 表达用户体验质量问题，而不是编辑器内部错误。

首批阶段：

```ts
type UxDiagnosticStage =
  'accessibility' | 'interaction' | 'layout' | 'content' | 'visual' | 'checker';
```

分段：

| 段位      | 阶段            | 说明                                       |
| --------- | --------------- | ------------------------------------------ |
| `UX-10xx` | `accessibility` | WCAG、ARIA、语义、替代文本和辅助技术可读性 |
| `UX-20xx` | `interaction`   | 键盘、焦点、指针目标、输入反馈和交互状态   |
| `UX-30xx` | `layout`        | 响应式、溢出、滚动、遮挡和安全区域         |
| `UX-40xx` | `content`       | 文案、标签、说明、错误建议和状态消息       |
| `UX-50xx` | `visual`        | 对比、层级、密度、主题、动效和可读性       |
| `UX-90xx` | `checker`       | UX 检查器配置、执行和结果生命周期          |

码表见 `specs/diagnostics/ux-diagnostic-codes.md`。

### 2) UX code 表达 Prodivix 产品语义，外部标准编号进入 meta

Prodivix 主 code 必须是稳定产品语义，例如：

```ts
type UxDiagnostic = ProdivixDiagnostic & {
  code: `UX-${number}`;
  domain: 'ux';
  meta?: {
    standardRef?: UxStandardRef[];
    evidence?: UxEvidence;
  };
};

type UxStandardRef = {
  standard: 'WCAG' | 'WAI-ARIA' | 'HTML' | 'axe' | 'Lighthouse' | 'internal';
  version?: string;
  criterion?: string;
  ruleId?: string;
  level?: 'A' | 'AA' | 'AAA';
};
```

示例：

```json
{
  "code": "UX-1001",
  "domain": "ux",
  "severity": "warning",
  "message": "文本对比度不满足 WCAG。",
  "meta": {
    "standardRef": [
      {
        "standard": "WCAG",
        "version": "2.2",
        "criterion": "1.4.3",
        "level": "AA"
      }
    ],
    "evidence": {
      "foreground": "#737373",
      "background": "#ffffff",
      "contrastRatio": 4.1,
      "requiredRatio": 4.5
    }
  }
}
```

规则：

1. WCAG success criterion、axe rule id、Lighthouse audit id、TypeScript diagnostic code 都不能替代 Prodivix 主 `code`。
2. 一个 `UX-xxxx` 可以关联多个外部标准编号。
3. 外部标准升级时优先更新 `meta.standardRef` 和规则实现；只有 Prodivix 产品语义变化时才新增或废弃 `UX-xxxx`。
4. 外部工具返回的诊断若无法映射到稳定 Prodivix 语义，使用 `UX-9010` 预留方向，而不是临时创造主 code。

### 3) UX 诊断必须携带证据，而不是只给意见

UX 诊断进入稳定码表的条件：

1. 有可复现证据，例如 computed style、DOM snapshot、PIR node、theme token、viewport、focus walk、ARIA tree 或交互状态。
2. 有稳定判断标准，例如 WCAG、WAI-ARIA、HTML 语义、项目 viewport policy、design token policy 或 command metadata。
3. 有明确用户影响，例如无法读、无法点、无法键盘完成、无法理解下一步或关键操作不可见。
4. 有可执行修复方向。

`meta.evidence` 可以包含：

```ts
type UxEvidence = {
  viewport?: { width: number; height: number; device?: string };
  themeId?: string;
  tokenPaths?: string[];
  computedStyle?: Record<string, string | number>;
  foreground?: string;
  background?: string;
  contrastRatio?: number;
  requiredRatio?: number;
  domPath?: string;
  role?: string;
  accessibleName?: string;
  focusOrder?: string[];
  boundingBox?: { x: number; y: number; width: number; height: number };
  missingEvidence?: string[];
};
```

规则：

1. `message` 和 `hint` 可以本地化；`code`、`stage`、`meta.standardRef`、`targetRef` 必须稳定。
2. 不确定结果不能伪装为失败；应使用 `UX-1015` 或 `UX-9004` 表达证据不足。
3. 纯审美建议不能进入 `UX-xxxx`，除非有项目规则或用户任务证据支撑。

### 4) UX 与其他诊断域的边界

| 场景                                          | 主 code      | 说明                                      |
| --------------------------------------------- | ------------ | ----------------------------------------- |
| PIR graph 缺少节点、引用断裂、保存态非法      | `PIR-xxxx`   | UX checker 可以聚合但不改写               |
| Inspector 写入被 schema 或 capability 拒绝    | `EDT-xxxx`   | 字段导致体验问题时另报 `UX-xxxx`          |
| 用户 CSS/TS/表达式解析失败                    | `COD-xxxx`   | 代码成功运行后导致体验问题才报 UX         |
| Mounted CSS selector 超出作用域               | `COD-3011`   | CSS 造成对比不足可另报 `UX-1001`          |
| Official plugin package/contribution 无法安装 | `PLG-xxxx`   | 外部组件可渲染但体验不合规才报 UX         |
| 导出 React/Vue/Svelte 项目失败                | `GEN-xxxx`   | 导出前体验质量 gate 使用 UX               |
| 后端保存 UX 检查配置失败                      | `API-xxxx`   | 配置语义非法可返回 `UX-9002` 作为下游诊断 |
| Prodivix 自身 UI 崩溃                         | 未来运行时域 | 不使用 UX 表示应用崩溃                    |

规则：

1. 上游结构、代码或运行时错误不能被 UX code 吞掉。
2. UX checker 可以聚合下游 `PIR-xxxx`、`COD-xxxx`、`PLG-xxxx`，但 Issues 面板必须保留原始 code 列表。
3. 同一现象可以有多个诊断，前提是每个诊断表达不同稳定语义。例如 `COD-1001` 表示 CSS 解析失败，`UX-1001` 表示渲染结果对比度不足。

### 5) UX TargetRef 扩展产品定位

UX 诊断应尽量落到用户能修复的位置：

```ts
type UxTargetRef =
  | DiagnosticTargetRef
  | { kind: 'theme-token'; themeId: string; tokenPath: string }
  | { kind: 'viewport'; routeId?: string; width: number; height: number }
  | { kind: 'runtime-dom'; routeId?: string; stablePath: string }
  | { kind: 'component-slot'; nodeId: string; slotName: string };
```

定位优先级：

1. 能映射到 Inspector 字段时，落到 `inspector-field`。
2. 能映射到 PIR 节点但字段不唯一时，落到 `pir-node`。
3. 与主题变量有关时，附加 `theme-token`。
4. 只在某个断点出现时，附加 `viewport`。
5. 只在 materialized DOM 中出现且无法反向映射时，使用 `runtime-dom` 并保留可复现 snapshot。

### 6) UX 检查模式分为静态、预览和交互

不是所有 UX 规则都能静态判断。

| 模式        | 输入                                         | 典型码位                        |
| ----------- | -------------------------------------------- | ------------------------------- |
| Static      | PIR、component metadata、theme token、schema | `UX-1003`、`UX-4005`、`UX-9002` |
| Preview     | materialized DOM、computed style、viewport   | `UX-1001`、`UX-3001`、`UX-5003` |
| Interaction | focus walk、keyboard path、state transition  | `UX-1014`、`UX-2001`、`UX-2010` |
| Export gate | route set、target framework、theme matrix    | 聚合 `UX-xxxx` 与 `GEN-xxxx`    |

规则：

1. Static checker 不能声称验证了运行时 focus trap。
2. Preview checker 必须记录 viewport、theme 和 route。
3. Interaction checker 必须记录用户路径或自动化步骤。
4. Export gate 可以阻断发布，但不应阻断本地保存。

### 7) Severity 表达用户影响和流程影响

建议：

| Severity  | UX 语义                                      |
| --------- | -------------------------------------------- |
| `info`    | 可读性、说明、结构或一致性改进，不阻断流程   |
| `warning` | 影响部分用户完成任务，建议在发布前修复       |
| `error`   | 关键任务不可完成、严重可访问性风险或频闪风险 |
| `fatal`   | 保留给检查器或预览上下文无法继续的极端情况   |

规则：

1. WCAG A/AA 不合规通常是 `warning`，但键盘陷阱、关键操作不可见和频闪风险可以是 `error`。
2. `UX-90xx` 检查器失败不代表页面体验失败，应独立展示。
3. 团队可在发布策略中把某些 `warning` 提升为导出 gate，但不能改变码位本身的语义。

### 8) UX 豁免必须结构化

某些组件、实验页面或第三方内容可能需要暂时豁免规则。

```ts
type UxExemption = {
  code: `UX-${number}`;
  targetRef: UxTargetRef;
  reason: string;
  owner?: string;
  expiresAt?: string;
  createdAt: string;
};
```

规则：

1. 豁免不是删除诊断；它把诊断状态改为 acknowledged。
2. 豁免必须有原因和范围，不能全局关闭整个 UX 域。
3. 到期豁免必须重新触发诊断或进入 `UX-9005` 信息提示。
4. 豁免不能覆盖 `UX-5005` 这类安全风险，除非发布策略显式允许。

### 9) UX 诊断展示位置

主落点规则：

| Target            | 主展示位置                     |
| ----------------- | ------------------------------ |
| `inspector-field` | Inspector 字段、Panel、Issues  |
| `pir-node`        | 画布节点标记、组件树、Issues   |
| `theme-token`     | Theme/token editor、Issues     |
| `viewport`        | Preview viewport badge、Issues |
| `runtime-dom`     | Preview overlay、Issues        |
| `operation`       | 导出前检查、发布 gate、Issues  |

规则：

1. UX 诊断默认进入 Issues 面板。
2. 有具体字段或节点时应就地显示，避免只在全局 toast 中出现。
3. 对用户代码导致的 UX 诊断，可以同时在 Code Editor 显示相关 `sourceSpan`，但主 code 仍为 `UX-xxxx`。
4. AI 修复应优先使用 `targetRef`、`meta.evidence` 和相关上游 diagnostics，而不是只读自然语言 `message`。

## 首批码位策略

首批码位不是最终完整 WCAG 映射，而是 Prodivix 需要长期稳定处理的体验问题集合：

1. Accessibility：对比度、替代文本、标签、可访问名称、heading、landmark、ARIA 引用、role 冲突、live region、颜色唯一表达、焦点指示、媒体、语言方向、键盘陷阱和证据不足。
2. Interaction：键盘路径、Tab 顺序、目标尺寸、状态表达、禁用原因、输入反馈、loading、destructive action、手势替代、弹层焦点、hover-only 和就地反馈。
3. Layout：横向溢出、遮挡、文本截断、关键操作不可见、可读行宽、嵌套滚动、safe area、空/错状态、响应式约束和弹层碰撞。
4. Content：控件文案、链接文本、错误建议、空状态、表单约束、状态标签、破坏性操作影响、本地化、数字单位和状态一致性。
5. Visual：非文本对比、视觉层级、主题变量、reduced motion、频闪、状态区分、字号行高、密度分组、skeleton 布局跳动和图表编码。
6. Checker：未知异常、配置非法、结果过期、证据不足和显式豁免。

## 分阶段落地

### Phase 1：规范和码表

1. 新增 `UX-xxxx` 诊断域。
2. 更新 Diagnostics 总规范、决策索引和文档生成器。
3. 建立首批 `UX-xxxx` 码表。
4. 明确外部标准编号进入 `meta.standardRef`。

### Phase 2：静态检查

1. 基于 PIR、component metadata、Inspector schema 和 theme token 实现静态规则。
2. 优先接入 `UX-1003`、`UX-1004`、`UX-4005`、`UX-9002`。
3. Issues 面板展示 UX domain。

### Phase 3：预览检查

1. 基于 materialized DOM 和 computed style 检查 contrast、overflow、target size、theme readability。
2. 记录 viewport、theme、route 和 targetRef。
3. 接入 Preview overlay 和 viewport badge。

### Phase 4：交互检查

1. 自动化 focus walk、keyboard path 和 overlay focus management。
2. 接入 `UX-1014`、`UX-2001`、`UX-2002`、`UX-2010`。
3. 为画布和复杂 widget 定义等价键盘命令策略。

### Phase 5：导出前 UX gate

1. 对 route set、theme matrix 和 target viewport matrix 运行 UX checks。
2. 将严重 `UX-xxxx` 作为发布或导出前 gate。
3. 导出报告保留原始 diagnostics、豁免和 evidence。

## 长期不变量

1. UX 诊断是产品级稳定语义，外部工具编号只作为 evidence。
2. UX 诊断不吞掉 `PIR`、`EDT`、`COD`、`PLG`、`GEN` 的原始错误。
3. 没有证据的体验意见不能进入稳定码表。
4. 不确定检查结果必须表达为证据不足，不能伪装成通过或失败。
5. `targetRef` 和 `meta.evidence` 是 UX 自动修复、Issues 聚合和文档排障的核心。
6. UX checker 的执行失败属于 `UX-90xx`，不等同于页面体验失败。
7. 已发布 `UX-xxxx` 即使废弃也必须保留文档说明。

## 非目标

1. 本决策不实现完整 WCAG checker。
2. 本决策不把 WCAG success criteria 一对一复制成 Prodivix 错误码。
3. 本决策不定义具体 UI 组件样式。
4. 本决策不要求立即支持所有第三方 accessibility engine。
5. 本决策不把纯审美偏好、品牌创意方向或人工设计评审意见变成稳定错误码。
