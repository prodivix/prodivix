## 决策背景

跨文档、VFS、RouteManifest 与多 revision partition 的一次用户动作需要具备数据库原子性、强幂等 replay、单一 commit identity、完整 delta response 与可验证的 forward/reverse semantics。

Atomic WorkspaceOperation Commit 是生产作者态远端写入的唯一边界。

## 决策内容

跨分区 commit 入口：

```http
POST /api/workspaces/:workspaceId/operations/commit
```

请求只接受一个已经规划并能在 confirmed WorkspaceSnapshot 上通过本地 Apply/validation 的 `WorkspaceOperation`：

```ts
type WorkspaceOperationCommitRequest = {
  expected: {
    workspaceRev?: number;
    routeRev?: number;
    documents: Array<{
      id: string;
      contentRev?: number | null;
      metaRev?: number | null;
    }>;
  };
  operation: WorkspaceOperation;
};
```

## WorkspaceOperation wire kinds

Wire contract 只允许两种 kind：

```ts
type WorkspaceOperation =
  | {
      kind: 'command';
      command: WorkspaceCommandEnvelope;
      undoOf?: string;
      redoOf?: string;
      sourceOperationIds?: string[];
    }
  | {
      kind: 'transaction';
      transaction: WorkspaceTransactionEnvelope;
      undoOf?: string;
      redoOf?: string;
      sourceOperationIds?: string[];
    };

type WorkspaceTransactionEnvelope = {
  id: string;
  workspaceId: string;
  issuedAt: string;
  commands: WorkspaceCommandEnvelope[];
  label?: string;
  mergeKey?: string;
};
```

规则：

1. `command` kind 必须只携带 `command`；`transaction` kind 必须只携带 `transaction`。
2. Transaction 至少包含一个 Command，Command IDs 在 Transaction 内唯一。
3. Path workspaceId、Transaction workspaceId 与所有 Command target.workspaceId 必须一致。
4. Operation/Transaction/Command、namespace/type/version、Workspace 与可选 target Document/Route 标识符必须非空且已经 canonical：`value === value.trim()`；不额外限定 UUID 或字符集格式。
5. Transaction 与每个 Command 的 `issuedAt` 必须是合法 RFC3339 timestamp。
6. `undoOf` 与 `redoOf` 在 trim 后互斥；它们和 `sourceOperationIds` 只表达因果与审计，不改变 Apply、权限或 revision 语义。
7. `sourceOperationIds` 在 canonical request digest 计算前逐项 trim，空值以 422 拒绝，并按首次出现顺序稳定去重；不能用 `uniqueItems` 拒绝本可规范化的重复输入。
8. Atomic Commit 的所有 domain、forwardOps 与 reverseOps 都禁止 `move` / `copy`。调用方必须把它们展开为显式 granular `add` / `remove` / `replace` / `test`，避免 TypeScript 与 Go 的 JSON Patch 数组语义漂移。
9. Intent 是“把用户意图规划为 Operation”的输入。Atomic Commit wire union 只接收已经规划和校验的 Command / Transaction。

## 精确 revision preconditions

`@prodivix/workspace-sync` 的 `planWorkspaceOperationCommit(confirmedSnapshot, operation)` 是客户端唯一 expected-vector planner。Web 不手写或猜测 revision 集合。

服务端必须从 Operation 的实际持久写集与显式 `test` 读依赖独立推导相同 requirements，并要求 request.expected 与该受影响分区集合精确匹配：

| 写入范围                             | 必需 expected                                                       |
| ------------------------------------ | ------------------------------------------------------------------- |
| 已有 Document content                | `documents[{ id, contentRev }]`                                     |
| 已有 Document name/path/capabilities | `workspaceRev` + `documents[{ id, metaRev }]`                       |
| 新增 Document                        | `workspaceRev` + `documents[{ id, contentRev:null, metaRev:null }]` |
| 删除 Document                        | `workspaceRev` + 当前 `contentRev`、`metaRev`                       |
| Workspace tree                       | `workspaceRev`                                                      |
| RouteManifest                        | `workspaceRev` + `routeRev`                                         |
| 显式 `test` 读依赖                   | 与被测试 path 对应的同一分区 baseline                               |
| Mixed Transaction                    | 上述所有受影响分区的并集                                            |

`expected.documents` 始终存在；没有 Document 写入时是空数组。它按 document id 的 Unicode code-point 顺序稳定排序，并且每个 id 只能出现一次；该顺序在 TypeScript 与 Go 两端必须一致，不能依赖 locale。

规则：

1. `contentRev` / `metaRev` 省略表示 Operation 不写该分区。
2. 显式 `null` 表示该 Document 必须不存在，只用于新增 identity。
3. 正整数表示该分区必须与 confirmed baseline 相同。
4. 同一 Transaction 多次修改同一 Document 时只提交一次初始 baseline；客户端不得手工构造 `N + 1` 链式 precondition。
5. 缺少必需分区、重复 document id、额外 document precondition、非法 revision 或 unsupported path 都在任何写入前以 422 拒绝。
6. Server-derived requirements 是安全边界；不能相信客户端 planner 的判断而跳过服务端写集分析。
7. Workspace document 新增/删除使用 `/docsById/<escaped-document-id>` 的 granular identity patch。
8. `settings` 不属于作者态 WorkspaceOperation Commit；它不能借用 `workspaceRev` 混入该 endpoint。`activeDocumentId` / `activeRouteNodeId` 是本地 ephemeral selection，服务端不持久化，也不形成 revision precondition；只包含 ephemeral path 的 Operation 以 422 拒绝为空写集。
9. `asset` 与 `project-config` 只接受正式 `resource` domain 的可逆 document command，并使用独立 path policy 与 content validator；其他 domain 修改这两类文档必须 422，不得退回任意 JSON Patch。
10. `test` 可以作为含真实持久变更的 Operation 内部读前置条件，并形成对应 revision dependency；但纯 `test`、同值 `replace` 或多个 Commands 最终互相抵消都不构成可见 Commit。整笔 Apply 与 canonical final validation 后若持久作者态相对 baseline 没有 delta，必须在推进 revision、`opSeq`、operation log 或幂等记录前以 422 拒绝。
11. `workspaceRev`、`routeRev`、`contentRev`、`metaRev` 与 `opSeq` 必须始终位于 JSON safe-integer 正整数范围；应用层在递增前拒绝越界，数据库约束作为所有写入路径的最终防线。

## 强幂等语义

Operation stable id 是 commit id：

- command operation 使用 `command.id`；
- transaction operation 使用 `transaction.id`。

服务端以 `(workspaceId, commitId)` 建立唯一幂等记录，并保存 canonical request digest 与成功 mutation result。

规则：

1. 首次请求只有在数据、revision、operation log 和幂等结果能在同一数据库事务提交时才返回成功。
2. 相同 commit id 与相同 canonical request 重试时，返回首次提交的同一 mutation result；不得再次 Apply、推进 revision/`opSeq` 或追加 operation log。
3. 相同 commit id 携带不同 canonical request 时返回 422 identity-reuse failure；不得把它伪装成 revision conflict。
4. 幂等查找必须发生在 revision conflict 返回之前，否则已成功但响应丢失的请求会在重试时被错误报告为 stale revision。
5. 并发相同 commit id 只能有一个事务成为首次提交者；其他请求等待或读取已提交结果。

## 数据库原子性不变量

Atomic Commit 必须满足：

1. 在读取或构造 revision details 前验证 Workspace owner；不存在与非 owner 使用相同的 metadata-free 404。
2. 整个 request strict decode、Operation envelope validation 与静态 path/write-set analysis 在写入前完成。
3. 一个 Commit 只使用一个数据库事务。
4. 事务按稳定顺序锁定 Workspace 与全部受影响 Documents，再在锁内一次性校验所有 preconditions。
5. Commands 在事务私有 canonical state 上按顺序 Apply；每个 Command 的 reverse ops 必须恢复其输入状态。
6. 最终 state 必须通过 Workspace VFS、Document content 与 Route reference validation。
7. Document rows、Workspace tree、RouteManifest、所有 revisions、单一 operation-log record 与幂等 result 必须在同一事务提交。
8. 任一 Command、precondition、validator、SQL 或 operation-log 写入失败时整体 rollback；不得留下 partial content、tree、route、revision、opSeq 或成功幂等记录。
9. 一条 WorkspaceOperation 是一个可见 commit：`opSeq` 只推进一次，operation log 只追加一条记录；内部 Commands 不是独立远端 commit。
10. 每个受影响 revision partition 在一次 commit 中最多推进一次；不受影响分区保持不变。
11. 成功响应必须聚合整笔 Commit 的全部 mutation delta，不能只返回最后一个 Command 的结果。

Canonical Workspace 数据库是权威事实源。Git、索引、通知、Project publication 与其他外部投影通过同一数据库事务写入的 transactional outbox 可靠驱动。

## 冲突规则

Atomic Commit 仍使用 ADR 07/11 的 canonical 409：

1. `WKS-4001 / WORKSPACE_CONFLICT`。
2. `WKS-4002 / ROUTE_CONFLICT`。
3. `WKS-4003 / DOCUMENT_CONFLICT`，可以由 `contentRev` 或 `metaRev` 过期触发。

Conflict contract 只使用以上三种 canonical 分区类型。多个 precondition 同时过期时，服务端按稳定优先级返回一个具体分区；客户端随后获取最新 canonical snapshot，用 semantic diff3 分析整笔 Operation 的全部差异。

409 仍不得返回 document content。Document conflict 的 expected/current metadata 必须足以区分 content 与 metadata baseline，但完整 remote state 只能通过 owner-authorized snapshot read 获取。

`expected.document` 必须包含 id，并至少包含 `contentRev` 或 `metaRev` 之一；metadata-only conflict 仍使用 `WKS-4003 / DOCUMENT_CONFLICT`。`current.document` 属性必需：远端存在时返回当前 contentRev/metaRev 安全 metadata，远端已删除时为 `null`。Document-only stale commit 不得把 delete-modify 折叠成 404；客户端读取最新 snapshot 后继续 semantic diff3。客户端不增加第四种 conflict type。

同理，新增 Document 的显式 null absence precondition 若发现远端 identity 已存在，属于 concurrent-add 的 `WKS-4003`，不是 422 envelope validation failure。

## 成功响应

成功沿用 canonical Workspace mutation envelope：

```ts
type WorkspaceOperationCommitSuccess = {
  workspaceId: string;
  workspaceRev: number;
  routeRev: number;
  opSeq: number;
  updatedDocuments?: WorkspaceDocument[];
  removedDocumentIds?: string[];
  tree?: WorkspaceTree;
  routeManifest?: WorkspaceRouteManifest;
  acceptedMutationId: string;
};
```

`acceptedMutationId` 必须等于 commit id。Response 必须包含客户端重建 confirmed snapshot 所需的完整受影响 delta；没有变化的可选 projection 不返回。

`opSeq` 不是 CAS precondition。不相交分区的 Commit 可以在客户端 base snapshot 之后、当前 Operation 之前成功，因此响应的 `opSeq` 可能跨越多步，未纳入 expected vector 的 Workspace/Route revision 也只保证单调不回退。Web 发现 `opSeq > base.opSeq + 1` 时必须重新读取 canonical snapshot 后再采用 ACK，不能把只含当前 Operation delta 的响应冒充为完整 catch-up stream。

## 与 History、Outbox、Planner 的边界

1. History 记录本地成功 Apply 的 WorkspaceOperation；远端 ACK 不新增一条用户 History。
2. Atomic Commit 是 Operation 的远端持久化边界，不负责快捷键、merge window 或 selection replay。
3. Durable Outbox 持久化的单位是一条 exact `WorkspaceOperationCommitRequest`。
4. Intent planner 生成 WorkspaceOperation，Operation Commit wire union 接收 Command / Transaction。
5. Conflict resolution 必须从最新 remote snapshot 生成 fresh Operation，再由同一 planner 与 Commit endpoint 写回。
6. 服务端 strong idempotent replay 与客户端 Durable Outbox 共同闭环：网络失败、刷新和崩溃恢复复用 exact request/id；只有 canonical 409 的 semantic recovery 才生成 fresh rebased Operation。

## 非目标

本 ADR 不实现：

1. 多端实时订阅、presence 或 CRDT。
2. Git remote、部署与社区系统的跨系统分布式事务。
3. 任意脚本或未注册 plugin command 的服务端执行。

## 验收标准

- [x] `/operations/commit` strict decode `command | transaction` WorkspaceOperation。
- [x] Server 从 Operation 推导并精确校验 workspace/route/document preconditions。
- [x] Document add 使用显式 null absence CAS；metadata 使用 metaRev CAS。
- [x] 任一后续 Command 失败时数据库、revisions、opSeq 与 operation log 全部不变。
