# PIR-current 稳定模型与低成本演进

## 状态

- DecisionStatus：Accepted
- 日期：2026-07-14
- ImplementationStatus：PIR-current Architecture Implemented
- ProductGateStatus：G1 Passed
- Global Phase：G1-G6

## 决策摘要

Prodivix 的生产代码只面向一套无版本号的 `PIR-current` 领域模型。数字 PIR
版本属于序列化协议，不属于 Renderer、Compiler、Workspace、Semantic Index 或
Web 作者界面的架构。

`PIRDocument`、`PIRNode`、`PIRRenderer`、`decodePirDocument` 和领域 Command
等公开名称保持稳定。生产目录、类型、函数与组合链路统一使用无版本命名。

版本升级通过集中 wire codec 与确定性 migration 完成：旧 wire document 先在边界
升级为当前领域模型，所有下游只消费当前模型；保存时只编码为当前 wire contract。

## 三层模型

```text
immutable versioned wire schema / persisted document
  -> version dispatch + strict wire decode
  -> deterministic migration chain
  -> unversioned PIR-current domain model
  -> semantic validation
  -> Workspace / Semantic Index / Renderer / Compiler / Web

unversioned PIR-current domain model
  -> deterministic normalization
  -> current wire encoder
  -> current schema validation
  -> persisted document
```

### 1. PIR-current 领域模型

`@prodivix/pir` 拥有唯一活动领域模型。它表达当前作者态语义，公开 API 不含数字
版本前缀或后缀。

- 领域对象使用稳定 identity、typed reference、scope 和 capability。
- mutation、projection、semantic contribution、extraction 与 validator 都消费同一模型。
- 领域模型在同一套 current 实现中演进。
- 领域行为由 current model 的类型与 capability 决定，数字 wire version 只用于边界分派。

### 2. Current wire contract

`PIR-current.version.json` 是 activation manifest，指向一份不可变
`PIR-v<version>.json` snapshot。同步脚本从该 snapshot 生成
`specs/pir/PIR-current.json` 这一 active schema alias、generated wire types 与后端
current schema；这些派生产物不作为手工编辑的版本源。

- generated wire type 只允许被 PIR codec 边界消费。
- wire type 不作为 Workspace、Renderer、Compiler 或 Web 的领域类型导出。
- 后端 current version 常量用于 transport/persistence 校验，不定义前端领域架构。
- `PIR-current` 可以在 G1 内继续演进；数字版本只在形成新的持久化格式时递增。

### 3. Versioned schema 与 migration

`PIR-v<version>.json` 是不可变 wire 快照。版本化代码只允许出现在
`@prodivix/pir` 的 wire/migrations 边界。

- migration 是纯函数、确定性、可组合并且 fail-closed。
- migration chain 从输入版本逐步到 current，下游始终只读取 current model。
- migration 完整保留可表达的作者态语义；无法安全转换时返回稳定诊断。
- 历史数据由 wire boundary 的 decoder 与 migration 进入同一 current model。

仓库中的 `1.0`、`1.1`、`1.2` snapshot 是建立 canonical migration baseline 之前的
归档契约，只用于追溯历史格式，不承诺可作为生产导入输入。`1.3` 是 migration chain 的
最低 canonical baseline；从 `1.3` 起，每一个已 activation 的后续 wire version 都必须
保留到 current 的完整确定性 migration chain，任何缺失 step 都在 decode boundary
fail-closed。

## 稳定公开 API

生产领域入口使用以下无版本命名：

```text
PIRDocument
PIRNode
PIRComponentContract
decodePirDocument
encodePirDocument
validatePirDocument
createPIRProjectionPlan
PIRRenderer
compileWorkspacePirReactModules
```

数字版本常量、versioned wire type 与 migration step 统一由显式
`@prodivix/pir/wire` 子入口提供；`@prodivix/pir` 默认入口只提供 current 领域能力。

| --------------------- | ------------------------------------------------------------------- | ----------------------------------------------- |

| `@prodivix/pir/wire`  | version dispatch、strict decode/encode、migration chain             | decoder、encoder、migration                     |

| `apps/web`            | current 作者界面                                                    | 仅新增产品能力                                  |
| backend               | wire envelope、current version 与持久化校验                         | current version、schema/migration orchestration |

## 版本升级流程

一次普通 PIR wire 升级必须收敛为以下步骤：

1. 新增下一份不可变 `PIR-v<version>.json` snapshot，不修改既有 snapshots。
2. 在 codec 边界增加从已激活版本到新 snapshot 的确定性 migration 与轻量边界 fixture。
3. 运行 `pnpm run pir:activate-wire -- <version>`；该命令先验证 snapshot 与 immutable
   边界，再原子更新 `PIR-current.version.json`，同步 `PIR-current.json`、wire-only
   TypeScript types 与后端 current schema/constants，并执行 migration/codec 测试和边界
   检查；任一步失败时回滚派生产物。
4. 保存端开始只写 manifest 选中的 current wire contract。

## Canonical persistence rollout gate

Repository contract activation 与 production data rollout 是两道独立 gate。
`pnpm run pir:activate-wire -- <version>` 只原子更新仓库中的 schema alias、generated
contracts 与边界测试，不代表 Canonical Backend 中已经保存的旧 wire document 已迁移。

旧 wire Workspace 在前端 decode 为 current domain 后，后端保存态仍可能是旧结构。此时禁止
把 current domain 的路径级 patch 直接应用到旧 wire document；即使路径本身重合，最终整文档
也无法通过 current schema，且不同 wire 表达之间不存在可推断的局部 patch 等价关系。

production rollout 必须选择并完成以下一种 canonical 策略：

1. **Backend coordinated migration**：activation/deploy gate 先由 Canonical Backend 按
   content revision/CAS 迁移所有旧 PIR document，使用同一确定性 migration fixture 验证结果，
   通过 current schema 后才允许 current 路径 commit。
2. **Atomic first-write upgrade**：hydration 在 transport metadata 中保留 source wire version；
   首次写入的 exact request 先携带完整 current wire document replacement，再在同一后端原子
   transaction 中应用后续路径 patch。后端以旧 content revision 为前置条件，并只提交通过
   current schema 的最终文档。domain Command、History 与 local apply 仍不感知数字版本。

不得通过客户端私有镜像、后台 lazy fallback、双版本 Renderer/Compiler 或仅在内存 decode 后
继续 patch 旧保存态来绕过这道 gate。迁移失败必须 fail-closed，并保留旧文档与稳定诊断。

G1 closure 已覆盖 TypeScript codec 的 1.3→1.4 deterministic migration、current wire
encoding 与 Atomic Commit request projection，并建立低扩散的仓库升级架构。未来启用新的
numeric wire production rollout 时，在该边界上组合 Backend coordinated migration 或
atomic first-write upgrade orchestration。

若升级只改变 wire 表达而不改变领域语义，Workspace、Renderer、Compiler、
Semantic Index 与 Web 保持不变。若升级新增领域语义，只扩展无版本的
current 模型及真正消费该语义的稳定 API。

普通 wire 升级的默认验收结果是：除 schema snapshot、activation manifest、generated
wire contracts、migration 与相应边界测试外，生产消费者 diff 为空。

## 自动约束

仓库边界检查应保证：

1. `apps/**` 与常规 `packages/**` 的 PIR 生产目录与公开符号使用无版本命名。
2. generated wire type 的导入范围限定于 wire/migration 模块。
3. PIR 数字版本分派限定于 codec/persistence 模块。
4. 默认 package entry 只导出 current production API。
5. Renderer 与 Compiler 使用同一 current projection contract。

测试保持轻量：codec/migration 使用属性测试覆盖 deterministic、idempotent current
decode/encode 与 migration chain；Renderer/Compiler 使用少量 conformance 覆盖共享语义。
不为每个历史版本复制整套产品测试。

## G1 稳定架构

G1 的唯一生产架构满足：

1. Component、Collection、Binding、Extraction 与 Semantic contribution 统一由无版本
   `@prodivix/pir` API 提供。
2. Workspace projection、Transaction 与 Semantic Index 只使用默认 current 行为。
3. Renderer、Compiler 与 Blueprint 表面是无版本唯一实现。
4. 默认领域入口、生产目录与公开符号保持无版本命名。
5. S6 使用同一 current 架构验证 G1 Golden 产品旅程。

## 结果

- PIR 数字版本可以频繁演进，而不会扩散到整个项目。
- 已保存 Workspace 可以在一个明确边界升级，不产生第二作者态真相。
- 所有 G1 后续功能从一开始就在长期稳定的 current 架构上实现。
- 版本升级成本与变更的真实语义规模相关，而不与消费者数量相关。
