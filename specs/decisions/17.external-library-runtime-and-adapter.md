## 决策背景

早期 Blueprint 通过 esm.sh 在编辑器主线程动态加载第三方组件库，由此暴露出典型扩展性问题：

1. 每接一个组件库都手写一份库专属加载逻辑，调用链路重复、维护成本高。
2. 若强依赖人工 manifest，用户无法快速接入“任意库 + 任意版本”的现实需求。
3. 运行时存在生态差异风险（React 单例、portal 行为、全局 CSS/reset 污染、上下文依赖组件）。
4. 组件库可展示、可拖拽、可渲染、可生成生产代码需要统一协议，不应散落在 UI 侧分支逻辑中。

旧 remote loader/scanner/profile 链路已在 official plugin cutover 中删除。当前决策重点是让组件库通过可验证 package、JSON-only contribution、受信任 Host projection 和显式 compiler policy 接入；未来若恢复任意库发现能力，也必须作为受沙箱治理的 generic plugin 实现，不能恢复编辑器主线程 esm.sh fallback。

## 决策内容

采用“**Plugin Contribution + JSON-only External Descriptor + Host Projection + Serializable Codegen Policy**”架构：

0. **Plugin Contribution（外部能力归属层）**
   - official / community plugin 必须具备 Plugin Manifest，并通过 `externalLibrary`、`renderPolicy`、`codegenPolicy`、`paletteContribution` 等扩展点接入。
   - Plugin Manifest、权限、生命周期和 contribution owner 由 ADR 29 的 Extension Host 管理。
   - `externalLibrary@1.0`、`renderPolicy@1.0`、`codegenPolicy@1.0`、`iconProvider@1.0` 与 `blueprintTemplate@1.0` 已冻结，并完成 Host-controlled package resolution、owner/generation implementation binding、Canvas projection 与 compiler snapshot。
   - Ant Design、MUI、Radix 已迁移为 bundled official plugin；旧 core profile、manifest、remote official loader、Renderer/Compiler 特判与 Radix placeholder 已删除，不再是生产路径。

1. **External Library descriptor（可序列化事实层）**
   - `externalLibrary@1.0` 保存 library id、exact package、dependencies、component path、runtime type、props/slot/behavior metadata 与 Host implementation identity。
   - descriptor 不保存 React component、callback、module namespace、DOM handle 或 executable URL。
2. **Host projection（受信任实现层）**
   - bundled official package 通过 build-attested React Host Module 提供 component map、Palette preview、Render wrapper、Icon implementation 与 surface provider。
   - community runtime 默认无权绑定主线程 React/DOM projection；未来实现必须经过 sandbox RPC 与 capability policy。
3. **Render/Codegen Policy（声明式消费层）**
   - Canvas 从 Render Policy 生成 owner-scoped projection；Compiler 只消费 immutable、serializable `CodegenPolicySnapshot`。
   - exact dependencies、license、import kind、element path、children policy 与 icon codegen 均来自 policy，不从运行时对象反射。
4. **Generic arbitrary-library discovery（未来插件能力）**
   - 当前产品只安装 bundled catalog 中的 verified package；未知或未 bundled library id 不触发远程执行。
   - 后续 generic external-library plugin 若实现零配置发现，必须自行声明 Manifest/capability，在 opaque sandbox 中运行，并把结果转换成同一组 JSON-only contribution；不得恢复旧 app-local RuntimeLoader/ExportScanner。

## 术语收敛（2026-02-17 修订）

1. 渲染侧继续使用 Adapter 概念，但对外统一称为 **Render Policy**。
2. 代码生成侧统一称为 **Codegen Policy**，不再使用 “generator adapter” 对外命名。
3. Render、Palette、Template 与 Codegen 通过同 owner/generation 的 cross-point identity 保持语义闭合，不允许各自维护不可追踪的库专属分支。

## 接入分层模型

任何可安装 official / community plugin 都必须通过 `plugin-manifest-v1.schema.json` 声明身份、权限和 contribution。不存在绕过 Plugin Manifest 的 remote profile。

### L0：零配置自动发现（未实现）

若未来实现，由 generic external-library plugin 在 sandbox 内扫描受策略允许的 package，并输出严格 JSON contribution。当前 Host 不执行任意 URL/npm package，也没有 app-local fallback。

### L1：约定推断增强（未实现）

未来可以在 generic plugin 内按命名和行为标签推断分组、props option 与展示信息，但推断结果仍需通过 exact contribution Schema、size limit 和 Host semantic validation。

### L2：声明式 Plugin Contribution（已实现）

1. Plugin Manifest 声明 External Library、Palette、Blueprint Template、Render Policy、Codegen Policy 与可选 Icon Provider resource。
2. 所有 resource 为 closed JSON object，并在同一 installation transaction 中完成 cross-point validation。
3. 缺少创建、render 或 codegen 必需规则时 package 整体拒绝安装，不使用静默 fallback。

### L3：Official React Host Projection（已实现，受信任边界）

1. 对上下文依赖或副作用强的组件提供受限 provider、prop mapping、portal/container 和 transient canvas state。
2. implementation 只接收冻结的 React surface context，不获得 Workspace store、PIR graph、Host mutation API 或 package reader。
3. Ant Design、MUI、Radix 已通过这一边界接入；该能力不自动开放给 community package。

## 标准调用链路

`Extension Host -> externalLibrary/palette/template/render/codegen contribution -> resolved contribution registry -> Palette/Tree/Inspector -> CanvasRenderer -> serializable CodegenPolicySnapshot -> CodeGenerator`

链路约束：

1. 组件库面板、建节点、Inspector 与 Canvas 使用同一个 revisioned Host contribution snapshot，不允许各自维护私有解析分支。
2. 画布渲染失败必须降级为可解释占位节点，并输出诊断，不得直接崩溃整个编辑器。
3. 代码生成基于显式 Codegen Policy snapshot，确保产物是可生产的静态源码，而非运行时快照。

## 运行时安全与隔离约束

1. **React 单例约束**：official package 使用 Web build 的 React peer 和 build-attested Host Module，避免多 React 实例导致 context/hook 崩溃。
2. **CSS 污染控制**：默认禁止注入全局 reset；仅允许显式通道加载，且可按作用域隔离。
3. **Portal 类组件安全模式**：在画布预览时默认关闭破坏性全屏行为（如强制挂载 body、不可关闭遮罩）。
4. **上下文依赖组件兜底**：Form.Item、Menu.Item 等缺失上层上下文时，自动补最小 provider 或渲染占位提示。

## Diagnostics 标准

旧 main-thread remote loader 的 `ELIB-*` 码表已随实现删除，不保留无消费者的诊断域。Plugin Manifest、artifact、contribution、Host binding 与 lifecycle failure 使用 `PLG-*`；compiler policy/output failure 使用 Compiler diagnostics；PIR/composition 与用户体验问题分别使用对应 `PIR-*` / `UX-*` 域。新增 generic external-library plugin 时必须先按真实失败边界选择或建立诊断 contract，不能复活旧码位占位。

## 非目标

1. 当前不承诺零配置任意库接入；该能力必须等 generic external-library plugin 和 sandbox contract 完成。
2. 不在本决策内定义第三方库许可扫描与供应链审计流程。
3. 不要求一次性覆盖所有外部库的高级交互模式与可访问性细节。

## 分阶段落地

### Phase 1：早期通用 runtime（Superseded）

早期 app-local RuntimeLoader、ExportScanner、profile registry 与 ELIB diagnostics 已删除，不能作为当前实现入口。迁移结论保留在本 ADR 与 `official-component-plugins-phase46-48.md`；已删除的 execution plan/backlog 不再作为历史入口。

### Phase 2：Manifest/Adapter 增量增强

1. [x] 建立 External Library、Render Policy、Codegen Policy、Icon Provider 与 Blueprint Template exact contract。
2. [x] 建立 build-attested React Host ABI、owner-scoped style/overlay host 和 immutable compiler policy snapshot。
3. [x] 以 Ant Design、MUI、Radix 完成 direct/template、Modal/Dialog/portal 与 compound component 验证。

### Phase 3：生态化与稳定性

1. 通过后续 generic external-library plugin 支持用户注册受策略允许的库，并把项目状态持久化到 Workspace VFS；当前未实现。
2. [x] 将 Ant Design、MUI、Radix 迁移为使用同一宿主协议的 official plugin；社区插件模板仍未实现。
3. 将导入成功率、渲染成功率、代码生成成功率纳入可观测指标。

## 当前实现结果（2026-07-11）

1. Ant Design 以 81 个 Palette item、6 个 contribution point 和 exact `antd@5.28.0` / `@ant-design/icons@6.1.0` 进入 bundled catalog。
2. MUI 以 18 个 Palette item、2 个 template-only runtime component、6 个 contribution point 和 exact Material/Emotion dependency closure 进入同一 catalog。
3. Radix 以 10 个 Palette item、7 个 compound template、37 条 runtime/codegen rule 和 5 个 contribution point 进入同一 catalog，不声明 icon provider。
4. `blueprintTemplate@1.0` 通过单一 Workspace Command 原子实例化 Form.Item、Accordion 与 Radix compound fragment；click/drop/undo/redo 共用同一创建链路。
5. Compiler 只消费 serializable `CodegenPolicySnapshot`，并只为实际使用的 policy 输出 exact dependency closure。AntD-only、MUI-only、Radix-only 和三库组合导出项目均通过 install、build 与 browser behavior gate。
6. official package 默认随 Web build 可发现，但 workspace 默认不启用任何外部库；disable/update/shutdown 清理 owner contribution、implementation、style 与 overlay lease，不删除用户 PIR 节点。
7. Phase 4.9 已通过 Chromium、Firefox、WebKit sandbox matrix、official component Chromium smoke、四种 export fixture 与全仓 hardening gate；production sandbox 作为独立 GHCR/Compose service 部署。
8. ADR 仍保持 Draft/Partial，因为无需手写库专属 enhancement 的 L0/L1 流程、generic external-library plugin、可观测指标、community SDK 与供应链分发尚未完成。

## 验收标准（Draft）

- [ ] generic external-library plugin 可以在不手写库专属 enhancement 的情况下发现受策略允许的 React package，并发布可拖拽 contribution。
- [x] official component plugin 的 Palette、建节点、Inspector 与 Canvas 使用同一 Host snapshot，不出现私有 profile 分叉。
- [ ] generic external-library plugin 的加载/渲染失败在面板可见标准化 Diagnostics，包含阶段与修复提示。
- [ ] generic external-library plugin 不默认注入全局 reset 或逃逸受控 style/overlay host。
- [x] official component plugin 不默认注入全局 reset，style/overlay 由 surface host 约束并随 owner cleanup。
- [x] official component plugin 的代码生成可输出稳定 import、组件调用和 exact dependency closure，不依赖运行时动态对象结构。
