## 决策背景

ADR 32 完成 blueprint feature 目录迁移后，对 `blueprint/data/` 与其消费层做了一次只读盘点，发现两处结构性负债。

### 1. `editor/model/data.ts` 是一个腐烂的 mega-barrel

该文件转发 8 个源，但盘点证实**只有 3 个源被 barrel 消费者真正引用，其余 5 个是死转发**：

| 转发的源                   | 经 barrel 被消费？ | 实际消费方                                                                     |
| -------------------------- | ------------------ | ------------------------------------------------------------------------------ |
| `data/viewport`            | ✅                 | canvas / controller / viewportBar / SidebarPreviewFrame / SidebarComponentList |
| `data/sampleData`          | ✅                 | palette（18 个 symbol）                                                        |
| `data/helpers`             | ✅                 | SidebarComponentList                                                           |
| `data/options`             | ❌ 死              | 7 个 `data/groups/*` 直接 import                                               |
| `data/placeholders`        | ❌ 死              | 直接 import                                                                    |
| `data/ComponentGroups`     | ❌ 死              | 直接 import                                                                    |
| `blueprint/registry`       | ❌ 死              | 直接 import                                                                    |
| `blueprint/layoutPatterns` | ❌ 死              | 直接 import                                                                    |

冒烟枪：`editor/model/palette.ts` 在**同一文件内**对同类静态目录数据使用两种约定——`sampleData` 走 barrel，而 `registry` / `layoutPatterns` 直连：

```ts
import { SIDEBAR_ITEMS, TABLE_DATA, ... } from '@/.../editor/model/data';   // 走 barrel
import { getComponentItemById } from '@/.../blueprint/registry';            // 直连
import { buildLayoutPatternNode } from '@/.../blueprint/layoutPatterns';    // 直连
```

这证明该 barrel 是历史遗留的偶然便利，不是有意设计的边界。

此外 `editor/model/index.ts` 中的 `export * from './data'` 也是死的——没有任何文件通过裸路径 `@/.../editor/model` import。该 barrel 名为 `model/data`，却不是 model 逻辑，只是静态目录数据的转发站，层名误导。

### 2. `data/viewport.ts` 混了 3 类关注点（且含死代码）

| 导出                                                                         | 关注点               | 消费方                                                                 |
| ---------------------------------------------------------------------------- | -------------------- | ---------------------------------------------------------------------- |
| `DEFAULT_ROUTES`                                                             | 路由（domain）       | **0 个（死代码）**                                                     |
| `VIEWPORT_QUICK_PRESETS` / `VIEWPORT_DEVICE_PRESETS` / `VIEWPORT_ZOOM_RANGE` | 视口 UI 配置         | canvas / controller / viewportBar                                      |
| `DEFAULT_PREVIEW_SCALE` / `COMPACT_PREVIEW_SCALE`                            | sidebar 组件预览比例 | SidebarComponentList / SidebarPreviewFrame / `helpers.getPreviewScale` |

其中 preview-scale 常量已被 `data/helpers.ts` 的 `getPreviewScale` 反向引用（`helpers → viewport` 跨文件依赖），而 `getPreviewScale` 只服务 sidebar。三件事彼此无关却共处一文件。

## 决策目标

1. 删除误导且大半死亡的 barrel，让 import 路径自文档化。
2. 把 `data/viewport.ts` 拆成单一关注点，各归各位；顺手删除死代码 `DEFAULT_ROUTES`。
3. 不引入运行时逻辑变更，仅做常量/路径搬家与 barrel 删除。

## 决策内容

### 1) 删除 `editor/model/data.ts` barrel

- 删除文件 `editor/model/data.ts`。
- 6 个 barrel 消费者改为直连各自真实源：

  | 消费者                                       | 改为直连                                                    |
  | -------------------------------------------- | ----------------------------------------------------------- |
  | `canvas/BlueprintEditorCanvas.tsx`           | `data/viewport`                                             |
  | `controller/useBlueprintEditorController.ts` | `data/viewport`                                             |
  | `viewportBar/BlueprintEditorViewportBar.tsx` | `data/viewport`                                             |
  | `sidebar/SidebarPreviewFrame.tsx`            | `data/viewport`（见决策 2，preview-scale 迁出后改指新位置） |
  | `sidebar/SidebarComponentList.tsx`           | `data/viewport` + `data/helpers`（见决策 2）                |
  | `model/palette.ts`                           | `data/sampleData`                                           |

- 删除 `editor/model/index.ts` 中的 `export * from './data'`（第二层死转发）。

约束：

- 不引入新的聚合 barrel 替代它。消费者各自直连源文件。
- `editor/model/` 只保留真正的 model 层模块（autosave / dragdrop / graphExecutor / palette / tree / types / blueprintText）；旧 Radix factory 已在 official plugin cutover 中删除。

### 2) 拆分 `data/viewport.ts`

按关注点拆三处：

- **`DEFAULT_ROUTES`**：0 消费者，**删除**。
- **`VIEWPORT_QUICK_PRESETS` / `VIEWPORT_DEVICE_PRESETS` / `VIEWPORT_ZOOM_RANGE`**：留在 `data/viewport.ts`，使该文件成为诚实的"视口配置"单一职责文件（消费方：canvas / controller / viewportBar）。
- **`DEFAULT_PREVIEW_SCALE` / `COMPACT_PREVIEW_SCALE`**：迁出至 sidebar，与 `getPreviewScale` / `WIDE_PREVIEW_SCALE_BOOST`（现位于 `data/helpers.ts`）合并为 `editor/sidebar/previewScale.ts`。

  理由：这四项构成同一关注点（sidebar 组件预览缩放），且 `helpers.getPreviewScale` 已经反向依赖 `DEFAULT_PREVIEW_SCALE`。合并后：
  - 消除 `helpers → viewport` 的跨文件依赖。
  - `data/helpers.ts` 只保留真正的目录辅助函数（`buildVariants` / `getDefaultSizeId` / `getDefaultStatusIndex` / `isWideComponent`）。
  - `data/viewport.ts` 只剩视口配置。

### 目标结构（相关部分）

```text
blueprint/
├── data/
│   ├── ComponentGroups.tsx          # 不变
│   ├── groups/                      # 不变

│   ├── options.ts                   # 不变（本轮不拆）

│   ├── sampleData.tsx               # 不变（本轮不拆）
```

## 直接影响

### 文件删除

- `editor/model/data.ts`
- `DEFAULT_ROUTES`（死代码导出）

### 文件迁移

- preview-scale 常量：`data/viewport.ts` → `editor/sidebar/previewScale.ts`
- `getPreviewScale` / `WIDE_PREVIEW_SCALE_BOOST`：`data/helpers.ts` → `editor/sidebar/previewScale.ts`

### import 路径更新

- 6 个原 barrel 消费者改直连。
- `SidebarPreviewFrame` / `SidebarComponentList`：preview-scale 引用改指 `editor/sidebar/previewScale`。
- 其余原 `data/helpers` 中 `getPreviewScale` 的引用（仅 `SidebarComponentList`）改指 `editor/sidebar/previewScale`。

### 测试

- 无专门针对 barrel 的测试；现有 vitest 套件（含 `BlueprintGraphExecutor.test.ts` 与 inspector 内 `__tests__`）作为回归。

## 取舍说明

### 为什么不顺便拆 `data/options.ts` 与 `data/sampleData.tsx`

两者是 flat grab-bag（`options.ts` 14+ 组件变体枚举、`sampleData.tsx` 18 份组件示例数据，每个 export 属于不同组件）。不在本轮拆，原因：

1. 一旦 barrel 删除，消费者直连 `data/options` / `data/sampleData`，路径已诚实，flat 表的危害大幅降低。
2. 拆分它们涉及所有 `data/groups/*` 的 import 改动，收益与成本不匹配。
3. 留作后续单独决策，按组件组拆分时再评估（如 `data/options/form.ts` 等）。

### 为什么不把 `data/` 重命名为 `catalog/`

`data/` 大多数内容确实是组件目录（ComponentGroups / groups / options / sampleData / placeholders / helpers），唯独 `viewport.ts` 不是。本轮把 `viewport.ts` 收敛为纯视口配置后，`data/` 的异类程度下降但未归零（视口配置仍非"目录"）。重命名 `catalog/` 是更大的语义收敛，触及大量 import，不在本轮。留作后续决策。

### 为什么 preview-scale 迁到 sidebar 而非留在 data/

preview-scale 的全部消费者都在 sidebar（`SidebarComponentList` / `SidebarPreviewFrame`），且 `getPreviewScale` 只服务 sidebar。它是 sidebar 的渲染关注点，不是领域数据。迁到 sidebar 让关注点与消费者共处，并消除 `helpers → viewport` 的反向依赖。

### 为什么 `VIEWPORT_*` 留在 `data/` 而非迁到 `editor/viewportBar/`

`VIEWPORT_ZOOM_RANGE` 被 canvas / controller / viewportBar 三处消费，不是 viewportBar 独占。留在 `data/viewport.ts`（诚实的单一职责视口配置文件）比塞进某个消费子系统更中性。

## 非目标

1. 本决策不改任何运行时逻辑、常量值或组件 props。
2. 本决策不动 `data/options.ts`——其溶解（`SIZE_*` 抽共享 + 变体内联进 group）已由 ADR 29 内置收敛完成。
3. 本决策不重命名 `data/` 目录——已由 ADR 29 内置收敛重命名为 `catalog/`。
4. `data/sampleData.tsx`（现 `catalog/sampleData.tsx`）保留为共享 demo 数据（group 预览 + `palette.ts` 建节点默认值两处消费），不在任何决策中拆分。
5. 本决策不调整 `data/groups/*` 的内部结构。
6. 本决策不改变 inspector 内部数据结构（ADR 21 范畴）。
