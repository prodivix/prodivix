## 决策背景

Prodivix 的核心架构中，Git 位于 PIR、编译器、部署、社区系统之间：

```text
Canonical Workspace VFS -> Workspace 文件投影 -> Git -> GitHub / Gitee / GitLab
PIR -> Compiler -> Framework-specific source
Framework-specific source -> Git -> GitHub / Gitee / GitLab
Framework-specific source -> Build -> Deploy
License -> Git
Backend <-> Community
LLM -> PIR
```

因此 GitHub App 不只是“推送代码”的外设，而应成为 Prodivix 与真实软件工程平台连接的核心集成层：

1. 将 Prodivix Canonical Workspace VFS 投影同步到 GitHub 仓库。
2. 将编译产物、源码、测试、License 信息纳入 PR 工作流。
3. 将 GitHub PR / Checks / Issues / Comments 反馈回 Prodivix。
4. 为 LLM agent 提供受限的 GitHub 操作通道。
5. 支持社区模板、主题、组件、插件的 GitHub 分发链路。

## 决策目标

1. 使用 GitHub App 作为 GitHub 集成的首选方式，而不是用户 PAT。
2. 建立 project/workspace 与 GitHub repository 的绑定模型。
3. 支持从 Prodivix 创建 branch、commit、pull request。
4. 支持 GitHub webhook 反馈 checks、reviews、comments、deployments。
5. 支持 Workspace VFS 投影和生成源码的独立版本控制。
6. 支持依赖 License / security / provenance 信息进入 PR。
7. 支持 LLM agent 在受限权限下参与 PR 和 issue 工作流。
8. 为未来 Gitee / GitLab 集成保留 provider 抽象。

## 核心决策

### 1) GitHub App 是 GitHub 集成的主入口

Prodivix 不以 Personal Access Token 作为首选集成方式。GitHub 集成使用 GitHub App installation。

原因：

- 权限可精细配置。
- 可按组织 / 仓库安装。
- 支持 webhook。
- 支持 Checks API。
- 更适合 SaaS / 本地后端代理。
- 用户离职或 token 泄露时风险更低。

允许的 fallback：

- 本地开发可使用 CLI git remote。
- 自托管或离线环境可使用手动导出。
- PAT 仅作为高级临时兼容模式，不作为产品主路径。

### 2) Git Provider 抽象必须存在

虽然首批实现 GitHub App，但架构上应保留 provider 抽象。

```ts
type GitProvider = 'github' | 'gitee' | 'gitlab';

type RepositoryBinding = {
  id: string;
  provider: GitProvider;
  installationId?: string;
  owner: string;
  repo: string;
  defaultBranch: string;
  projectId: string;
  workspaceId: string;
  status: 'active' | 'disabled' | 'revoked' | 'error';
};
```

GitHub 专有能力可以通过 capability 暴露：

```ts
type GitProviderCapabilities = {
  supportsAppInstallation: boolean;
  supportsChecks: boolean;
  supportsPullRequests: boolean;
  supportsDeployments: boolean;
  supportsCodeScanning: boolean;
  supportsIssueComments: boolean;
};
```

### 3) Workspace 与 Repository 明确绑定

每个 Prodivix project 可以绑定一个或多个 repository，但首批只支持一个 active binding。

绑定内容：

- provider
- installationId
- owner/repo
- defaultBranch
- Workspace VFS 投影路径
- generated source 路径
- build/deploy workflow 路径
- sync mode

建议路径：

```text
.prodivix/
  workspace.json
  route-manifest.json
  docs/
    doc_home.pir.json
    doc_about.pir.json
  node-graphs/
  animations/
  metadata/
    dependencies.json
    licenses.json

src/generated/
  react/
  vue/
  web-components/
```

规则：

- `.prodivix/` 是 Prodivix 权威状态的 Git 表达。
- `src/generated/` 是编译器输出，不是 PIR 真相源。
- 人工源码、代码 artifact、外部代码可以放在项目约定目录，例如 `src/Prodivix-code/`。

### 4) Workspace VFS 投影与编译产物独立同步

Workspace VFS 投影和编译产物都应该能直接同步到 GitHub，并且可以分别同步。

双轨：

```text
PIR Track
  .prodivix/workspace.json
  .prodivix/route-manifest.json
  .prodivix/docs/*.pir.json
  .prodivix/node-graphs/*
  .prodivix/animations/*
  .prodivix/metadata/*

Artifact Track
  src/generated/*
  framework-specific source
  build/deploy config
  generated tests
```

同步模式：

```ts
type GitSyncTrack = 'pir' | 'artifacts' | 'both';

type GitSyncMode =
  'manual' | 'auto-pir' | 'auto-artifacts' | 'auto-both' | 'pr-only';
```

规则：

- `pir` track 是 Prodivix 可视化与结构化状态的 Git 表达。
- `artifacts` track 是编译器输出与工程交付物。
- 用户可以只同步 PIR，不生成或提交编译产物。
- 用户可以在 Workspace 作者态未变时重新编译并只同步 artifacts。
- 用户可以在同一个 PR 中包含两条 track，也可以拆成不同 commits / PR。
- 每条 track 应有独立 dirty 状态、last synced revision、last commit sha。

GitHub PR 可以同时表达：

1. PIR / workspace 变化。
2. 生成源码变化。
3. 测试和配置变化。
4. License / dependency metadata 变化。

提交策略：

- 默认一个用户动作批次或 LLM milestone 对应一个 commit。
- 如果同时同步两条 track，建议拆成两个 commit：
  - `feat(pir): ...`
  - `build(artifacts): ...`
- 大任务可以拆多个 commits，但同一个 PR。
- commit message 使用规范格式，例如 `feat(editor): add pricing section`。
- commit body 写入 Prodivix operation summary。

PR 描述应包含：

- 用户意图。
- PIR diff 摘要。
- 生成源码目标。
- 测试结果。
- License 变更。
- LLM 是否参与。
- 预览链接。

### 5) GitHub Checks 反馈回 Prodivix

GitHub App 应创建或读取 Checks。

首批 checks：

```text
Prodivix / PIR Validation
Prodivix / Compile
Prodivix / Typecheck
Prodivix / Unit Tests
Prodivix / Visual Regression
Prodivix / License Scan
Prodivix / Dependency Security
```

Checks 应标明适用 track：

- PIR validation 主要作用于 `pir` track。
- Compile / Typecheck / Unit Tests 主要作用于 `artifacts` track，但也可由 PIR 变化触发。
- License / Dependency Security 对两条 track 都可能生效。

反馈路径：

```text
GitHub webhook
  -> backend github module
  -> project integration event
  -> workspace diagnostics
  -> Prodivix UI status panel
```

用户在 Prodivix 中应能看到：

- 当前 PR 状态。
- 哪些 checks 失败。
- 失败 diagnostics 对应 PIR 节点 / 代码 artifact。
- 是否可由 LLM repair。

### 6) Webhooks 是状态同步入口

GitHub App 需要处理 webhook。

首批事件：

- `installation`
- `installation_repositories`
- `push`
- `pull_request`
- `pull_request_review`
- `pull_request_review_comment`
- `check_suite`
- `check_run`
- `issue_comment`
- `workflow_run`
- `deployment_status`

规则：

- webhook 必须验签。
- 事件处理必须幂等。
- 事件应写入 integration event log。
- 不在 webhook handler 中执行重任务，转入 queue。

### 7) PR Review 与 Comment 进入 Prodivix 任务系统

GitHub review/comment 不应只停留在 GitHub 页面。它们应映射回 Prodivix。

示例：

- PR comment 指向生成源码行。
- 系统映射到 code artifact。
- 如能关联 PIR source map，则进一步定位到 PIR node。
- 在 Prodivix 中显示为 review task。
- 用户或 LLM 可以生成修复 batch。

需要 compiler source map：

```text
generated code line -> PIR document -> nodeId -> property / code artifact
```

### 8) LLM Agent 可以参与 GitHub 工作流，但权限受限

结合 `22.llm-integration-architecture.md`，LLM 可以：

- 总结 PR diff。
- 生成 PR 描述。
- 解释 check failure。
- 根据 review comment 生成修复计划。
- 在 dry-run 后创建修复 commit。
- 生成测试。

LLM 不可以默认：

- 直接 merge PR。
- 修改 GitHub App permissions。
- 关闭安全告警。
- 绕过 failed checks。
- 删除仓库内容。
- 写入 secrets。

高风险操作必须用户确认：

- force push。
- 删除 branch。
- 修改 workflow。
- 引入新依赖。
- 修改 deployment 配置。
- merge / close PR。

### 9) License 与依赖信息进入 GitHub PR

AGENTS 架构中包含 `License -> Git`。GitHub App 应把依赖 license 处理纳入版本控制和 PR 检查。

功能：

- 记录外部库来源。
- 记录 npm / esm.sh / CDN 依赖。
- 生成 `.prodivix/metadata/dependencies.json`。
- 生成 `.prodivix/metadata/licenses.json`。
- PR 中标注新增依赖和 license。
- 对高风险 license 给出 warning。

License check 不应阻塞所有 PR，但应支持策略配置：

```text
allow
warn
block
```

### 10) Deploy 与 Preview 通过 GitHub 状态回流

Prodivix 的 Deploy 层包括 Vercel / Netlify / GitHub Pages / Cloudflare 等。GitHub App 应读取或创建部署状态。

功能：

- PR preview URL 回显到 Prodivix。
- deployment status 回写 project。
- build failure 映射为 diagnostics。
- 用户可从 Prodivix 打开 preview。

首批不要求 Prodivix 自己托管 CI/CD，但需要能读取 GitHub Actions / third-party deployment 状态。

### 11) Community 与 GitHub 共享链路

Prodivix 社区系统可通过 GitHub 分发：

- project template
- theme manifest
- component adapter
- plugin
- sample PIR
- node graph snippet
- animation preset

GitHub App 支持：

- 从 repository 导入模板。
- 将当前 project 发布为 template repository。
- 在社区条目中关联 GitHub repo。
- 使用 GitHub release/tag 管理版本。

社区导入必须经过：

- manifest validation。
- license check。
- sandbox / capability check。
- dependency scan。

## GitHub App 权限

首批建议最小权限：

```text
Repository permissions:
  Contents: Read and write
  Pull requests: Read and write
  Checks: Read and write
  Commit statuses: Read and write
  Issues: Read and write
  Metadata: Read-only
  Actions: Read-only
  Deployments: Read-only

Organization permissions:
  Members: Read-only (optional, only if team mapping is needed)

Subscribe to events:
  installation
  installation_repositories
  push
  pull_request
  pull_request_review
  pull_request_review_comment
  check_suite
  check_run
  issue_comment
  workflow_run
  deployment_status
```

权限原则：

- 不申请 `Administration`。
- 不申请 `Secrets`。
- 不申请 `Workflows: write` 作为默认权限。
- 如果未来需要修改 workflows，必须作为单独高级能力申请。

## Conflict Model

GitHub push 和 Prodivix workspace revision 可能冲突。

冲突类型：

- `REMOTE_BRANCH_MOVED`
- `PIR_FILE_CONFLICT`
- `GENERATED_CODE_CONFLICT`
- `WORKSPACE_REV_CONFLICT`
- `ROUTE_REV_CONFLICT`
- `CODE_ARTIFACT_CONFLICT`

规则：

- `.prodivix/` 文件冲突优先由 workspace/rev 机制解释。
- 生成源码冲突可以重新编译覆盖，但必须保留人工修改警告。
- 人工修改 generated source 不作为长期推荐路径。
- code artifact 冲突需要普通 Git merge 或用户确认。
- Workspace track 冲突不应阻塞 artifacts track 的状态读取。
- artifacts track 冲突不应阻塞 Workspace track 的保存或同步。
- `both` 同步时任一 track 失败都必须报告具体失败 track，不允许只显示 generic sync failed。

## Sync State Model

每个 repository binding 需要分别记录两条 track 的同步状态。

```ts
type GitTrackSyncState = {
  track: 'pir' | 'artifacts';
  dirty: boolean;
  lastSyncedRev?: number;
  lastSyncedAt?: string;
  lastCommitSha?: string;
  lastErrorCode?: string;
};

type RepositorySyncState = {
  bindingId: string;
  branch: string;
  pir: GitTrackSyncState;
  artifacts: GitTrackSyncState;
};
```

UI 需要分别展示：

- Workspace 投影未同步。
- 编译产物未同步。
- Workspace 投影已同步但产物过期。
- 产物已重新编译但 Workspace 作者态未变。
- 两条 track 都已同步。

## 取舍说明

### 为什么选择 GitHub App 而不是 PAT

GitHub App 更符合产品级集成：权限可控、支持 webhook、安装粒度清晰、易于组织管理，也更适合未来 SaaS 和 self-hosted 后端。

### 为什么 Workspace VFS 投影必须进入 Git

Canonical Workspace VFS 是 Prodivix 的唯一作者态真相源。它的 Git 投影同时覆盖 PIR、NodeGraph、Animation、Code、Assets、Config 与 RouteManifest，使 GitHub 可以承载可视化编辑的长期演进、review、回滚和社区分享。

### 为什么仍提交生成源码

GitHub 用户和 CI 需要看到真实框架代码，部署平台也通常消费源码。生成源码是工程交付物，但不是 Prodivix 真相源。

### 为什么 Workspace 投影和编译产物可以分别同步

Prodivix 同时服务可视化编辑和真实工程交付。Workspace VFS 投影保留作者态真相，编译产物是目标框架交付物。二者变更频率、review 对象和冲突模型不同，因此分别同步。

### 为什么不默认让 LLM 直接操作 GitHub

GitHub 操作有安全和协作风险。LLM 可以生成计划和修复，但必须经 dry-run、checks 和用户确认边界。

## 非目标

1. 本决策不要求首批支持 GitLab / Gitee，只保留 provider 抽象。
2. 本决策不定义完整 CI/CD 平台。
3. 本决策不要求实现实时多人协作。
4. 本决策不允许绕过 workspace/rev 直接把远端内容覆盖本地权威状态。
5. 本决策不把 GitHub 作为唯一存储后端。
