## 决策背景

MFE 的三编辑器架构不只是三个独立 UI：

1. Blueprint 维护 PIR UI graph、节点属性、事件、数据绑定和样式挂载。
2. NodeGraph 维护逻辑图、端口、执行结果和可复用函数能力。
3. Animation 维护 timeline、binding、track、filter 和关键帧。

Code Editor 又可以为三者提供复杂逻辑、表达式、Mounted CSS、GLSL、WGSL 和外部执行代码。后续 LLM 辅助开发、诊断定位、自动修复和代码补全都需要理解这些对象之间的引用关系。

如果每个编辑器各自扫描 PIR、各自解析 NodeGraph、各自维护补全和诊断，就会产生重复实现、过期上下文和冲突 UI。MFE 需要一个长期稳定的作者态符号环境，统一回答“当前上下文有哪些符号”“某个引用指向哪里”“错误应该落到哪个 UI”。

`specs/decisions/28.code-authoring-environment.md` 进一步明确：MFE 仍保持三编辑器可视化架构，但 code-owned 内容由共享 Code Authoring Environment 承载。本决策定义的是该环境中的符号、作用域、引用和诊断查询底座，而不是完整 IDE 外壳。

## 决策目标

1. 建立可长期沿用的核心概念，而不是一次性实现完整 IDE 内核。
2. 让 Blueprint、NodeGraph、Animation 和 Code Editor 共享符号、作用域和诊断定位语义。
3. 让 UI 通过稳定查询接口消费符号环境，避免直接跨域扫描内部模型。
4. 支持 `COD-xxxx` 诊断落到 Code Editor、Inspector、画布、节点图、动画轨道和 Issues 面板。
5. 为后续增量索引、worker、语言服务、LLM 上下文压缩和自动修复保留扩展边界。
6. 作为 Code Authoring Environment 的共享索引层，服务 code slot、Workspace 代码文件、外部库 adapter 和三编辑器上下文。

## 核心决策

### 1) 引入 Authoring Environment 作为作者态共享索引层

Authoring Environment 是 Code Authoring Environment 中的只读查询层。它可以由增量索引、worker、缓存和语言服务实现，但 UI 只依赖稳定接口。

```text
Code Authoring Environment
  -> Authoring Environment
    -> CodeArtifact / CodeSymbol / CodeScope / Diagnostics
```

```ts
type AuthoringEnvironment = {
  revision: string;
  querySymbols(context: AuthoringContext): CodeSymbol[];
  resolveReference(
    reference: CodeReference,
    context: AuthoringContext
  ): ResolvedReference | null;
  getCompletions(context: AuthoringContext): CodeCompletion[];
  getDiagnostics(context: AuthoringContext): ProdivixDiagnostic[];
  getDefinition(
    reference: CodeReference,
    context: AuthoringContext
  ): DefinitionLocation | null;
  getReferences(
    symbolId: string,
    context?: AuthoringContext
  ): ReferenceLocation[];
};
```

规则：

1. UI 不直接扫 PIR、NodeGraph、Animation 或 External Library 生成补全和诊断。
2. 各编辑器只把当前上下文描述给 Authoring Environment。
3. Authoring Environment 可以降级实现，但接口必须稳定。
4. 任何返回给 UI 的诊断必须保留稳定 `code` 和可定位上下文。
5. Authoring Environment 不负责文件 tab、编辑器外壳、语言服务 UI 或 code slot 生命周期；这些属于 Code Authoring Environment。

### 2) CodeArtifact 表示用户代码片段

CodeArtifact 是用户在 MFE 中编写或挂载的最小代码单元。

```ts
type CodeArtifact = {
  id: string;
  language: 'ts' | 'js' | 'css' | 'scss' | 'glsl' | 'wgsl' | 'expr';
  owner: CodeArtifactOwner;
  source: string;
  revision: string;
};

type CodeArtifactOwner =
  | { kind: 'pir-node'; documentId: string; nodeId: string }
  | {
      kind: 'inspector-field';
      documentId: string;
      nodeId: string;
      fieldPath: string;
    }
  | { kind: 'nodegraph-node'; graphId: string; nodeId: string }
  | { kind: 'nodegraph-port'; graphId: string; nodeId: string; portId: string }
  | { kind: 'animation-track'; timelineId: string; trackId: string }
  | {
      kind: 'animation-keyframe';
      timelineId: string;
      trackId: string;
      keyframeId: string;
    }
  | { kind: 'workspace-module'; documentId: string };
```

规则：

1. 一个代码片段必须有稳定 owner。
2. owner 决定诊断主落点，不由 parser 或 UI 临时猜测。
3. 删除 owner 时必须清理或显式标记相关 CodeArtifact。
4. CodeArtifact 不负责表示目标框架导出文件；导出文件属于 Codegen。

### 3) CodeSymbol 表示跨编辑器可引用对象

CodeSymbol 是作者态符号环境中的可引用对象。

```ts
type CodeSymbol = {
  id: string;
  name: string;
  kind:
    | 'state'
    | 'param'
    | 'data'
    | 'item'
    | 'node'
    | 'prop'
    | 'event'
    | 'route'
    | 'graph'
    | 'graph-input'
    | 'graph-output'
    | 'timeline'
    | 'track'
    | 'filter-primitive'
    | 'component'
    | 'module'
    | 'asset'
    | 'function';
  typeRef?: string;
  source: SymbolSource;
  scopeId: string;
  targetRef?: DiagnosticTargetRef;
};

type SymbolSource =
  | { kind: 'pir'; documentId: string }
  | { kind: 'route'; routeId: string }
  | { kind: 'nodegraph'; graphId: string }
  | { kind: 'animation'; timelineId?: string }
  | { kind: 'external-library'; libraryId: string }
  | { kind: 'workspace'; documentId?: string }
  | { kind: 'code'; artifactId: string };
```

规则：

1. `id` 是符号主键，`name` 只用于显示和源代码匹配。
2. `scopeId` 决定符号可见范围。
3. `targetRef` 用于跳转、定位和诊断 UI 投放。
4. 外部库、NodeGraph、Animation 和 PIR 都通过 SymbolSource 接入，不共享内部数据结构。

### 4) Scope 表示上下文边界

Scope 描述符号在哪些上下文中可见，尤其用于 `$item`、`$data`、route params、graph outputs 和动画 binding。

```ts
type CodeScope = {
  id: string;
  parentId?: string;
  kind:
    | 'workspace'
    | 'document'
    | 'route'
    | 'pir-node'
    | 'list-item'
    | 'inspector-field'
    | 'nodegraph'
    | 'nodegraph-node'
    | 'animation'
    | 'code-artifact';
  ownerRef: DiagnosticTargetRef;
};
```

规则：

1. 作用域必须显式建模，不能只靠字符串前缀判断。
2. 子作用域可以遮蔽父作用域中的同名符号，但必须可解释。
3. 列表渲染、数据作用域和 route params 必须进入 scope tree。
4. 诊断 `COD-2001` 必须基于 scope 解析失败，而不是全局搜索失败。

### 5) TargetRef 与 SourceSpan 是长期定位协议

TargetRef 指向产品对象，SourceSpan 指向代码片段中的文本范围。

```ts
type DiagnosticTargetRef =
  | { kind: 'workspace'; workspaceId: string }
  | { kind: 'document'; workspaceId?: string; documentId: string }
  | { kind: 'pir-node'; documentId: string; nodeId: string }
  | {
      kind: 'inspector-field';
      documentId: string;
      nodeId: string;
      fieldPath: string;
    }
  | { kind: 'route'; routeId: string }
  | { kind: 'nodegraph-node'; graphId: string; nodeId: string }
  | { kind: 'nodegraph-port'; graphId: string; nodeId: string; portId: string }
  | { kind: 'animation-track'; timelineId: string; trackId: string }
  | { kind: 'code-artifact'; artifactId: string }
  | { kind: 'operation'; operation: string };

type SourceSpan = {
  artifactId: string;
  startLine: number;
  startColumn: number;
  endLine: number;
  endColumn: number;
};
```

规则：

1. `targetRef` 用于产品 UI 定位。
2. `sourceSpan` 用于 Code Editor inline diagnostic。
3. 一个诊断可以同时有 targetRef 和 sourceSpan。
4. `path` 表示协议或文档路径，不替代 targetRef。

### 6) Editor Adapter 负责把查询结果落到具体 UI

Editor Adapter 是 UI 与 Authoring Environment 之间的薄适配层。

```ts
type EditorAdapter =
  | 'codemirror'
  | 'inspector'
  | 'blueprint-canvas'
  | 'nodegraph'
  | 'animation-timeline'
  | 'issues-panel';
```

主落点规则：

| 场景                             | 主落点                         |
| -------------------------------- | ------------------------------ |
| 有 `sourceSpan` 的解析和类型错误 | Code Editor inline diagnostic  |
| `inspector-field` 诊断           | Inspector 字段或 Panel         |
| `pir-node` 引用断裂              | 画布节点标记 + Issues          |
| `nodegraph-port` 类型问题        | NodeGraph 端口或连线           |
| `animation-track` 绑定问题       | Animation track 或 binding row |
| 无法定位到单一对象的聚合错误     | Issues 面板                    |

规则：

1. 同一诊断可以被多个 adapter 展示，但必须有一个主落点。
2. Inspector 只承载当前节点可编辑属性导致的诊断。
3. API client、parser、validator 和 indexer 不直接 toast。
4. Issues 面板可以聚合诊断，但必须保留原始 code 列表。

## 符号来源

首批符号来源：

| 来源             | 典型符号                                          |
| ---------------- | ------------------------------------------------- |
| PIR graph        | nodeId、component props、region、event target     |
| Blueprint        | selected node、layout role、className、CSS hook   |
| NodeGraph        | graph input、graph output、端口、函数节点         |
| Animation        | timeline、track、binding target、filter primitive |
| Route            | route params、loader data、navigation target      |
| Workspace        | document、asset、module、public resource          |
| External Library | component props、events、slots、capabilities      |
| CodeArtifact     | exports、locals、functions、imports               |

## 诊断关系

`COD-xxxx` 与其他域的关系：

1. 代码片段解析、符号、类型、宿主绑定和作者态运行错误使用 `COD-xxxx`。
2. PIR 保存态结构和 ValueRef 契约错误仍使用 `PIR-xxxx`。
3. Inspector 字段写入被拒绝仍使用 `EDT-3002`；字段中的代码表达式错误可以附带 `COD-xxxx` 下游诊断。
4. NodeGraph 端口和连线结构错误仍使用 `NGR-xxxx`；代码访问 graph output 失败使用 `COD-2001`。
5. 目标框架导出失败使用 `GEN-xxxx`；作者态代码转译失败使用 `COD-5001`。
6. 外部库加载和注册失败使用 `ELIB-xxxx`；代码 import 已注册外部库失败使用 `COD-2002`。

## 分阶段落地

### Phase 1：稳定类型与诊断

1. 定义 `CodeArtifact`、`CodeSymbol`、`CodeScope`、`AuthoringContext`、`TargetRef`、`SourceSpan`。
2. 接入首批 `COD-xxxx` 码位。
3. Code Editor 和 Inspector 能展示带 `sourceSpan` 或 `inspector-field` 的诊断。

执行说明见 `specs/implementation/authoring-symbol-environment-phase1.md`。Phase 1 只实现长期稳定契约，不绑定具体索引器、语言服务、worker 或 UI 组件结构。

### Phase 2：轻量符号索引

1. 从 PIR graph、Route、Workspace 和 CodeArtifact 生成内存符号表。
2. 支持 `querySymbols`、`resolveReference` 和 `getDiagnostics`。
3. 使用 revision 标记过期环境，必要时返回 `COD-2004`。

### Phase 3：三编辑器适配

1. NodeGraph 提供 graph input、output、port type 到 SymbolSource 的映射。
2. Animation 提供 timeline、track、filter primitive 到 SymbolSource 的映射。
3. Blueprint、Inspector、NodeGraph、Animation 和 Issues 面板共享诊断投放规则。

### Phase 4：语言服务与自动修复

1. 接入 TypeScript、CSS、GLSL、WGSL 或表达式 parser。
2. 增量索引移入 worker。
3. LLM 使用 Authoring Environment 生成可定位上下文和修复建议。

## 长期不变量

1. UI 通过 Authoring Environment 查询符号和诊断，不直接跨域扫描内部模型。
2. CodeArtifact 必须有 owner。
3. 诊断定位必须优先使用 targetRef 和 sourceSpan。
4. `code`、`domain`、`severity` 是稳定协议，文案可本地化。
5. 符号来源可以增加，但 Symbol、Scope、TargetRef 的边界不能被绕过。
6. 实现可以从简单内存 Map 演进到 worker 和增量图索引，外部查询接口保持稳定。
7. Authoring Environment 是 Code Authoring Environment 的共享索引层，不是第四编辑器或完整 IDE 外壳。

## 非目标

1. 本决策不要求立即实现完整 TypeScript language service。
2. 本决策不要求一次性支持 rename、find references 和 go to definition 的完整 IDE 体验。
3. 本决策不改变 PIR 保存态结构。
4. 本决策不定义目标框架导出代码结构。
5. 本决策不覆盖 MFE 自身前端应用崩溃的错误域。
6. 本决策不定义 code-owned 内容的产品级编辑外壳、文件树、tab 模型或 code slot registry；这些由 `specs/decisions/28.code-authoring-environment.md` 约束。
