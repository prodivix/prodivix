## 决策摘要

Prodivix 建立一个 Workspace 级、可重建、带 revision 的 **Workspace Semantic Index**。它统一索引 Workspace document、Route、PIR node、Component、Collection scope、NodeGraph、Animation、Code、Token、Asset 和 Contract 的符号与引用。

“整个项目共享符号”表示同一 snapshot 中的符号具有唯一 address，可以被统一查询、定位、引用和分析；持久化领域对象还具有跨 revision 的 durable identity。这不表示所有符号都处于一个扁平全局作用域。可见性仍由显式 Scope、类型和 capability 约束。

Workspace Semantic Index 是 Canonical Workspace VFS 的派生投影，不是第二真相源，不接受领域写入。领域文档继续持久化各自的类型化引用；索引只把这些引用规范化为统一的 Symbol / Scope / Reference graph。

TypeScript、JavaScript、CSS、GLSL 和 WGSL Language Service 只通过 **Code Semantic Contribution Provider** 与 **Code Language Capability Provider** 接入。LSP 或任何具体语言服务都不是 Prodivix 跨领域符号协议。

`@prodivix/code-language` 是具体语言 adapter 的 owner。当前 TypeScript/JavaScript、CSS/SCSS 与 GLSL/WGSL provider 均已建立 revision-bound session，并将 definition、references、completion、diagnostics、hover、rename proposal 与 semantic contribution 映射为本文的稳定 contract。Shader adapter 先归一化为 Prodivix 自有的 parser-neutral symbol/occurrence model；GPU/目标后端 compile validation 是独立能力。

## 决策背景

Blueprint、NodeGraph、Animation 和 Code Authoring Environment 需要共同回答：

1. 当前上下文能引用哪些对象。
2. 某个引用的定义和所有使用位置在哪里。
3. 重命名、删除、移动或提取组件会影响哪些文档。
4. `$item`、route param、component prop、graph output 和 code export 在何处可见。
5. 诊断应落到画布、Inspector、节点图、动画轨道、Code Editor 还是 Issues。

如果这些能力只建立在 Code Language Service 上，非代码对象会变成二等公民；如果各编辑器自行扫描其他领域文档，则会形成多套索引、作用域和重命名规则。二者都不符合 Canonical Workspace 和单一写路径。

`@prodivix/authoring` 负责本文定义的 Workspace 级 contract、provider composition 与稳定查询 API。G1 以单一 Workspace Semantic Index 连接所有领域 provider 和作者界面。

## 核心决策

### 1. Workspace Semantic Index 是共享只读投影

```text
Canonical Workspace VFS @ WorkspaceRevisions
  -> Workspace / Route / PIR / Component / Collection providers
  -> NodeGraph / Animation providers
  -> Code Language Service provider
  -> Contract / Token / Asset providers
  -> Workspace Semantic Index snapshot
     -> Blueprint / Inspector / NodeGraph / Animation / Code / AI queries
     -> semantic diagnostic provider snapshot -> Diagnostics / Issues
```

索引必须满足：

1. 每个 snapshot 绑定明确的 `SemanticSnapshotIdentity`，包含 partitioned `WorkspaceRevisions`、semantic schema version 和 provider-set digest。
2. 同一 snapshot identity 和同一组 provider 输入产生确定性结果。
3. snapshot 可被丢弃并由 Canonical Workspace 重建。
4. stale snapshot 必须显式标记，不得伪装成当前结果。
5. 索引不保存用户只能在这里找回的作者态数据。
6. UI、AI 和插件不得直接修改索引以间接修改 Workspace。

### 2. 统一符号、作用域与引用图

以下类型表达稳定语义，不承诺最终字段命名：

```ts
type WorkspaceSymbol = {
  id: string;
  stability: 'durable' | 'revision-scoped';
  kind: WorkspaceSymbolKind;
  name: string;
  providerId: string;
  scopeId: string;
  ownerRef: DiagnosticTargetRef;
  typeRef?: string;
  capabilities?: string[];
};

type WorkspaceScope = {
  id: string;
  kind: WorkspaceScopeKind;
  ownerRef: DiagnosticTargetRef;
  parentId?: string;
  imports?: string[];
};

type WorkspaceReferenceEdge = {
  id: string;
  kind: WorkspaceReferenceKind;
  sourceRef: DiagnosticTargetRef;
  targetSymbolId?: string;
  scopeId: string;
  status: SemanticResolutionStatus;
  sourceSpan?: SourceSpan;
};

type SemanticResolutionStatus =
  | 'resolved'
  | 'missing'
  | 'not-visible'
  | 'ambiguous'
  | 'type-incompatible'
  | 'stale';

type SemanticResolutionResult =
  | { status: 'resolved'; symbol: WorkspaceSymbol }
  | {
      status: Exclude<SemanticResolutionStatus, 'resolved'>;
      referenceId: string;
      candidateSymbolIds?: string[];
    };

type SemanticSnapshotIdentity = {
  workspaceRevisions: {
    workspaceId: string;
    workspaceRev: number;
    routeRev: number;
    opSeq: number;
    documentRevs: Record<string, { contentRev: number; metaRev: number }>;
  };
  schemaVersion: string;
  providerSetDigest: string;
};
```

首批 `WorkspaceSymbolKind` 至少覆盖：

- workspace document、route、component、component prop/event/slot/variant；
- PIR node、region、state、data、collection item/index/error；
- NodeGraph、node、port、input、output；
- Animation timeline、track、binding；
- code artifact、module、export、function、type、CSS symbol、shader entry；
- token group、token、asset 和 external contract。

持久化领域实体、Component contract member、Collection scope 和显式 CodeArtifact 等 durable symbol 的 `id` 必须由稳定身份派生，例如 `documentId + local entity id`；rename、path 变化和 UI 重排不得改变其 identity。语言服务推导的 local、CSS selector 或 shader-local symbol 若没有持久 id，可以使用仅在当前 `SemanticSnapshotIdentity` 内稳定的 revision-scoped address；它们不能成为长期领域引用的唯一目标。显示名称不是主键。

### 3. 全局可寻址不等于全局可见

Workspace Semantic Index 可以按当前 snapshot 内唯一 id 找到整个项目中的任意符号，但 `queryVisibleSymbols(context)` 必须遵守：

1. Workspace、document、route、component、node、collection、graph、animation 和 code scope 的父子关系。
2. 显式 import、component contract、slot、graph port 和 runtime zone 边界。
3. 同名遮蔽、类型约束和 capability 约束。
4. 不允许仅通过全局名称搜索绕过不可见或不兼容的引用边界。

Definition 与全局搜索可以定位不可见符号；completion 和 bind 操作只能返回当前上下文可合法引用的符号。完成上下文 composition 后仍无法解析、歧义或不可访问的引用必须产生可定位诊断。对于只有 Route、Component instance 或其他宿主上下文加入后才能绑定的合法 provisional fact，owner provider 可以声明 `diagnosticPolicy: defer`；Index 仍返回真实的 `missing` 状态，但在上下文 provider 尚未加入前不制造误报。`defer` 不得用于掩盖持久化显式引用、领域校验错误或已完成 composition 后的失败。

### 4. 领域引用继续类型化持久化

Workspace 文档中的引用继续由 owner domain 定义，例如：

- Route target / Route document reference；
- ComponentInstance 的 `componentDocumentId` 与 contract binding；
- PIR target、ValueReference 与 Collection binding；
- NodeGraph graph / port reference；
- Animation target reference；
- CodeReference；
- TokenReference 与 AssetReference。

索引把这些引用投影为统一 `WorkspaceReferenceEdge`，但不得用一个通用 `{ symbolId: string }` JSON 结构吞并所有领域 contract。类型化保存态负责校验和编译；统一引用图负责 definition、references、impact、rename 和导航。

### 5. Provider 拥有领域解释权

| Provider                            | 首批索引内容                                                                    |
| ----------------------------------- | ------------------------------------------------------------------------------- |
| Workspace Provider                  | document、path、document type、资源身份                                         |
| Route Provider                      | route、params、loader/data contract、route-to-document reference                |
| PIR Provider                        | node、region、state、data、target reference                                     |
| Component Provider                  | component identity、props、events、slots、variants、instance edges              |
| Collection Provider                 | source、item/index/error scope、key 与 region edges                             |
| NodeGraph Provider                  | graph、node、ports、inputs、outputs、executor reference                         |
| Animation Provider                  | timeline、track、binding、target reference                                      |
| Code Semantic Contribution Provider | artifact、module、export、local scope、import/reference facts                   |
| Code Language Capability Provider   | language-native resolution、completion、definition、rename 与 source diagnostic |
| Contract/Token/Asset Provider       | external contract、design token、asset identity                                 |

G2 已增加 revision-bound Data source/schema/operation provider；PIR/Collection consumer binding、runtime adapter、Secret resolution 与 runtime-zone permission 继续按独立领域 owner 接入。G3 可以增加 BehaviorScenario 和 trace link provider。Provider 只能读取其被授予的 canonical snapshot 与稳定依赖，不得扫描 React store 或其他编辑器私有状态。

Semantic contribution provider 必须发布其 revision 下完整、未按调用方上下文预过滤的 symbol/scope/reference/dependency facts。跨 provider identity、聚合结果状态、跨领域 scope/capability policy 与最终 visibility 由 Index composition 统一执行，不能由各领域分别发明。

### 6. Language Service 通过两类 Code Provider 接入

```text
TypeScript / CSS / GLSL / WGSL Language Service
  -> Code Semantic Contribution Provider
       -> complete code symbols / scopes / imports / reference facts
  -> Code Language Capability Provider
       -> language-native resolution / completion / definition / rename
  -> WorkspaceSymbol / WorkspaceScope / WorkspaceReferenceEdge / semantic diagnostics
  -> Workspace Semantic Index
```

规则：

1. Prodivix core 不暴露 LSP object 作为跨领域 contract。
2. Contribution Provider 发布完整 facts，不按某个 UI context 预过滤。
3. Language Capability Provider 保留 TypeScript/CSS/shader 原生 lexical、module 和 type resolution；通用 Index 不重新实现语言规则，而是编排查询并规范化为统一 identity、status、`DiagnosticTargetRef` 与 `SourceSpan`。
4. Index 在语言原生结果之上应用跨领域 scope/capability policy，并统一 missing、not-visible、ambiguous、type-incompatible 与 stale 状态。
5. CodeReference 仍以稳定 artifact identity 为核心，不以 path 或当前名称为主键。
6. Language Service 建议的 source edits 必须转换为 Code Command / Transaction；不能直接写 VFS。
7. Language Service 暂时不可用时，非代码 provider 与已有 code artifact identity 仍可工作。

当前 `CodeLanguageSession` 是一个 immutable、revision-bound 能力视图。每个查询和 workspace edit proposal 都携带 snapshot identity 与 artifact revision；stale session 必须返回显式状态。session 生命周期不等同于底层 Language Service 生命周期：provider 可以在 package 内按 Workspace 复用有界的增量 engine，更新 document/version 后发布新的 immutable session；旧 session 被显式 supersede，`dispose()` 只释放视图 lease，不能把每次保存或无关 Workspace revision 变成 engine 卸载与重建。TypeScript/JavaScript rename proposal 由 `@prodivix/workspace` 严格校验 revision、`SourceSpan`、范围重叠和 no-op，再按 artifact 合并为一个原子 Workspace Transaction。

### 7. 查询接口按能力组织

稳定查询面至少包括：

```ts
type WorkspaceSemanticIndex = {
  snapshotIdentity: SemanticSnapshotIdentity;
  getScope(scopeId: string): WorkspaceScope | null;
  getSymbol(symbolId: string): WorkspaceSymbol | null;
  getReference(referenceId: string): WorkspaceReferenceEdge | null;
  getDependency(dependencyId: string): WorkspaceDependencyEdge | null;
  queryVisibleSymbols(
    context: SemanticQueryContext
  ): SemanticVisibleSymbolsResult;
  resolveReference(referenceId: string): SemanticResolutionResult;
  getDefinition(referenceId: string): SemanticResolutionResult;
  getReferences(symbolId: string): SemanticReferencesResult;
  getImpact(symbolIds: readonly string[]): SemanticImpactResult;
  getCompletions(context: SemanticQueryContext): SemanticCompletionsResult;
  getSemanticDiagnostics(): SemanticDiagnosticsResult;
};
```

`@prodivix/authoring` 已交付 immutable snapshot builder、带 `workspaceId` 的 partitioned revision identity、provider-set digest、完整 contribution validation、统一 resolution、lexical visibility、definition/reference/impact/completion 查询、`SEM-xxxx` semantic diagnostics，以及 revision-bound Code Language Capability session contract。`@prodivix/workspace` 通过 `createWorkspaceSemanticIndexFromSnapshot` 从同一 WorkspaceSnapshot 组合 Workspace、Route、PIR-current、standalone NodeGraph、standalone Animation 与注入的 code semantic contribution，并从同一 snapshot 组合领域 CodeSlot provider。`@prodivix/code-language` 发布 TypeScript/JavaScript、CSS/SCSS 与 GLSL/WGSL 的 symbol/reference facts 和 source diagnostics，并提供 native definition、references、completion、hover 与 rename proposal。Code Editor、Issues 与跨编辑器 CodeSlot 查询消费同一 revision identity，不再各自建立代码语义扫描器。具体索引器可以从内存 Map 演进到增量 graph、worker 或持久缓存，但查询结果的 identity、revision 和 visibility 语义必须稳定。

### 8. Rename、删除、移动和组件抽取使用影响图规划事务

Workspace Semantic Index 只回答影响范围，不执行写入：

```text
User / AI rename or extract intent
  -> semantic definition + reference + impact query
  -> domain Command planners
  -> one atomic Workspace Transaction
  -> validation / local History
  -> WorkspaceOperation / Durable Outbox / Atomic Commit
  -> new semantic snapshot
```

跨文档变更必须由 owner domain 生成类型化 Command。索引不得返回一组未经领域校验便直接落盘的通用 patch。

### 9. Semantic diagnostics 与全域 Issues 分层

Workspace Semantic Index 只产出 resolution、scope、reference、type compatibility 和 stale snapshot 等 **semantic diagnostics**，并把它们作为 revision-aware provider snapshot 交给 `@prodivix/diagnostics`。Outbox、conflict、runtime、export、plugin、security 等诊断不经过 Semantic Index。

`@prodivix/diagnostics` 继续拥有全域 provider snapshot lifecycle、去重、active/stale/resolved 状态、presentation、summary 与 Issues 查询。Workspace Semantic Index 不建立第二套 Issues aggregator。

`DiagnosticTargetRef` 指向产品对象，`SourceSpan` 指向 CodeArtifact 内文本范围。同一 semantic diagnostic 可以同时拥有二者：

- Code Editor 使用 `SourceSpan`；
- Blueprint、Inspector、NodeGraph、Animation 和 Resources 使用 `DiagnosticTargetRef`；
- Issues 聚合全部落点并保留原始诊断 code；
- Editor adapter 负责 presentation，不重新解析引用。

所有文档内部 target 必须包含 `documentId` 与实体局部 id。path、名称和展示 label 都不能替代稳定 target identity。

## 与 Code Authoring Environment 的边界

Canonical Workspace 拥有 code document；Code Authoring Environment 拥有代码作者体验、CodeArtifact 投影、CodeReference、CodeSlot、Language Service adapter 和代码 patch 交互。Workspace Semantic Index 属于跨领域 Semantic Authoring 基础设施：

```text
Workspace Semantic Index
  <- Code Authoring Environment / Code Semantic Provider
  <- Route / PIR / Component / Collection / NodeGraph / Animation providers

Code Authoring Environment
  -> consumes Workspace Semantic Index queries
  -> edits code through Command / Transaction
```

`@prodivix/authoring` 是稳定索引 contract、provider composition 和查询 API 的 owner；Code Authoring Environment 是最重要的 provider 与 consumer 之一，但不拥有全部项目符号。

## 分阶段落地

### Semantic Contract implementation

1. `WorkspaceSymbol`、`WorkspaceScope`、统一 reference graph 与 length-prefixed canonical address 已成为跨领域语义 contract。
2. `CodeArtifact`、`CodeReference`、`SourceSpan` 和 code slot 保持代码专属语义；`CodeReference.symbolId` 是 artifact 内稳定符号身份。
3. WorkspaceSnapshot composition 为所有作者界面提供同一 revision-bound 查询入口。

### Domain contribution 与 snapshot

Workspace、Route、PIR-current Component/Collection、standalone NodeGraph、standalone Animation、全部当前 Code Language、DTCG Design Token、DTCG Resolver 与 Asset provider 已接入 revision-aware WorkspaceSnapshot composition。NodeGraph typed port/executor CodeReference，Token group/root/alias/JSON Pointer、design system/set/modifier/context/source/resolution-order facts，以及 MIME/category-qualified Asset identity 使用同一 snapshot identity。`AssetReference` 通过 durable asset document id、可选 MIME constraint 与 `asset-reference` fact 接入同一 definition/reference/impact contract。

### Code language capabilities

TypeScript/JavaScript、CSS/SCSS 与 GLSL/WGSL 已完成 Language Capability 纵切：同一 revision-bound session 提供 definition、references、completion、diagnostics、hover、rename proposal 和对应 semantic contribution，并由 Code Editor、Issues 与 Workspace Semantic Index 共同消费。WGSL stage-aware entry 与 GLSL `main` 均投影为统一 `shader-entry`。

### Cross-domain refactor 与 UI

Code Editor 与 Issues 已共用 revision-bound Code Language snapshot、`DiagnosticTargetRef` 和 `SourceSpan`；Language rename edits 已通过 Workspace planner 形成可逆原子 Transaction。PIR event/mounted CSS、Route runtime、NodeGraph executor 与 Animation timeline CodeSlot 使用领域 binding projection 连接同一 Semantic Index；Blueprint Inspector、NodeGraph、Animation Inspector 与 Resources 已具备对应的 definition/reference/impact 入口。Token、Asset、external adapter、AI 与跨所有领域的 rename、删除和移动规划继续按同一 semantic query contract 收敛。

## 长期不变量

1. Canonical Workspace VFS 是唯一作者态真相，Semantic Index 永远可重建。
2. 全项目符号在同一 snapshot 中统一可寻址；durable identity 与 revision-scoped address 必须区分，可见性必须经过 scope、type 和 capability 检查。
3. 领域保存态保留类型化引用，不降级成通用 symbol JSON。
4. Language Service 是 provider，不是跨领域核心协议。
5. UI、AI 和插件只通过稳定查询接口消费语义，不扫描其他编辑器私有状态。
6. 所有 rename、move、delete、extract 和 Quick Fix 写入都走 Command / Transaction 与 Atomic Commit 链路。
7. semantic diagnostics 和导航以 snapshot identity、durable/revision-scoped identity、`DiagnosticTargetRef` 与 `SourceSpan` 为基础；全域 Issues ownership 仍属于 `@prodivix/diagnostics`。
8. PIR provider 只消费无版本 current model；数字 wire 版本升级不得复制 provider、symbol kind、scope 或 reference graph。

## 非目标

1. 本决策不要求 G1 一次性实现完整桌面 IDE。
2. 本决策不把 embedding、vector search 或模型缓存定义为 Semantic Index 的必需部分。
3. 本决策不让索引成为远端写协议、持久化入口或通用 JSON 数据库。
4. 本决策不要求所有符号在所有上下文可见。
5. 本决策不改变各领域 compiler 对类型化 contract 的 ownership。
