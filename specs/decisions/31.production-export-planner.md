## 决策背景

Prodivix 的作者态系统由 PIR、Workspace VFS、CodeArtifact、CodeReference、NodeGraph、Animation 和 Assets 共同组成。生产项目需要按组件、页面/路由、layout、runtime、domain module、global style 和 asset 组织稳定文件拓扑，同时保留作者态 identity 与 SourceTrace。

领域编译器负责产出语义化的模块、样式、资源和依赖 contribution；Production Export Planner 统一决定源码拓扑、文件聚合、import 关系、依赖声明、诊断映射和最终文件列表。

## 决策目标

1. 防止作者态实现细节泄漏到生成项目的长期文件结构中。
2. 让 Blueprint、NodeGraph、Animation、CodeArtifact 和 Assets 的导出走同一套产物规划模型。
3. 使 CSS、JS/TS、shader、adapter、runtime helper、assets 都能按生产源码边界稳定聚合。
4. 支持完整项目、单组件、单页面、nodegraph、animation 等不同 resource type，但不为每种导出路径复制临时规则。
5. 保留从产物代码回溯到 PIR / CodeArtifact / NodeGraph / Animation 的诊断和 source trace。

## 核心决策

引入 Production Export Planner，作为所有生产代码导出的统一中间层。

```text
Workspace VFS / PIR / CodeArtifact / NodeGraph / Animation / Assets
  -> Domain Compilers
  -> Export Artifact / Module / Style / Asset Contributions
  -> Export Program IR
  -> Production Export Planner
  -> Export Bundle files
```

### 1) 与 Code Authoring Environment 的关系

Production Export Planner 与 Code Authoring Environment 是同一条链路的两端：

```text
Blueprint / NodeGraph / Animation
  -> CodeSlot / CodeReference
  -> Code Authoring Environment
  -> Code Semantic Provider
  -> Workspace Semantic Index
  -> Export Program Builder
  -> Production Export Planner
  -> Target Bundle
```

Canonical Workspace VFS 是 code-owned 内容的作者态事实源；Code Authoring Environment 将 Workspace code document 投影为 CodeArtifact/CodeReference，并通过 Code Semantic Provider 发布代码 symbol、scope、reference、diagnostics、origin、ownership、SourceSpan 和外部导入语义。Production Export Planner 不重新扫描编辑器局部状态，也不把源码字符串或 Semantic Index 当成第二真相源；它消费领域 compiler、Code Authoring Environment 与 Workspace Semantic Index 的结构化投影。

规则：

1. 三编辑器通过 CodeSlot / CodeReference 连接代码能力，不把裸代码字符串直接交给导出器。
2. Export Program Builder 从 Code Authoring Environment 读取 artifact/source，从 Workspace Semantic Index 读取统一 symbol/reference/dependency/diagnostic query，再转换为 module/style/asset/dependency contribution。
3. Production Export Planner 负责 production topology、文件边界、import、dependency、asset copy 和 emit 顺序；它不负责判定作者态 ownership，也不绕过 Code Authoring Environment 自行扫描其他编辑器内部结构。
4. 外部导入源码、vendor file、plugin contribution、remote asset、workspace module 的 origin / ownership / license / hash / update policy 由各自领域 owner 持久化，并通过 Code Authoring Environment 或 Workspace Semantic Index 投影给 Export Program。
5. Source trace 从作者态引用继承到产物 contribution，用于 diagnostics、AI patch、导出预览和问题定位；默认不写入生产源码注释。

因此，Canonical Workspace 是导出输入的事实源，Code Authoring Environment 与 Workspace Semantic Index 是作者态投影，Production Export Planner 是导出的拓扑规划器和链接器。

### 2) Domain compiler 不直接拥有最终文件结构

Blueprint、NodeGraph、Animation 等领域编译器不得直接以“最终文件列表”为主要输出。它们应输出 Export Program IR 片段：

```ts
type ExportProgramContribution = {
  artifacts?: ExportArtifactContribution[];
  modules?: ExportModule[];
  styles?: ExportStyleContribution[];
  assets?: ExportAssetContribution[];
  files?: ExportFileContribution[];
  runtimeRequirements?: ExportRuntimeRequirement[];
  dependencies?: ExportDependency[];
  diagnostics?: CompileDiagnostic[];
};
```

领域编译器可以声明：

1. 某段 JSX/TSX 属于哪个组件、页面或 layout。
2. 某段 CSS 来源于哪个 CodeArtifact、owner、selector、scope。
3. 某个 NodeGraph 需要导出为 runtime callable module。
4. 某个 Animation 需要导出 TS timeline module、CSS keyframes、Web Animations helper 或 shader asset。
5. 某个 external adapter 或 runtime helper 需要被 import。

但领域编译器不应自行决定“一个 node 一个文件”或“一个 slot 一个文件”。

`ExportArtifactContribution` 是目标文件类型的统一上层入口。Workspace code document、project config、public asset、shader、adapter、deployment config、plugin template 和 remote/vendor asset 都应优先表达为 artifact，再由 planner 展开为 `ExportFileContribution`、`ExportStyleContribution` 或 `ExportAssetContribution`。底层 contribution 仍存在，但应作为 domain compiler 已经明确生产语义时的低层 IR，而不是页面层临时拼文件的入口。

### 3) Export Program IR 表达生产语义，而非作者态结构

Export Program IR 的核心实体应以生产源码语义命名：

```ts
type ExportProgram = {
  target: ExportTarget;
  roots: ExportRoot[];
  artifacts: ExportArtifactContribution[];
  modules: ExportModule[];
  styles: ExportStyleContribution[];
  assets: ExportAssetContribution[];
  files: ExportFileContribution[];
  dependencies: ExportDependency[];
  diagnostics: CompileDiagnostic[];
};

type ExportRoot = {
  id: string;
  kind: 'app' | 'route' | 'page' | 'component' | 'nodegraph' | 'animation';
  displayName: string;
  routePath?: string;
  sourceRef: DiagnosticTargetRef;
};

type ExportModule = {
  id: string;
  kind:
    | 'react-component'
    | 'react-entry'
    | 'nodegraph-runtime'
    | 'animation-runtime'
    | 'event-handler'
    | 'adapter'
    | 'workspace-module'
    | 'runtime-helper';
  ownerRootId?: string;
  suggestedName: string;
  sourceRef?: DiagnosticTargetRef;
  imports: ExportImportIntent[];
  body: string;
};
```

`ownerRootId` 表达产物归属，例如某段 mounted CSS 属于 `HomePage`，而不是属于 `PdxHeading-1.css`。

`ExportArtifactContribution.kind` 表达来源要导出的目标语义，例如 `source`、`style`、`runtime`、`domain`、`shader`、`asset`、`config`、`deployment`、`metadata`、`documentation`、`adapter`。Planner 负责把这些 artifact 放入 target preset 的 production path 中，并统一执行 path normalization、origin policy、asset delivery policy、hash 和 manifest 记录。

### 4) 文件边界由 production topology 决定

Planner 根据 resource type、route manifest、component boundaries、shared usage 和 runtime requirements 规划文件：

完整项目示例：

```text
src/main.tsx
src/App.tsx
src/routes/home/Home.tsx
src/routes/home/Home.css
src/routes/about/About.tsx
src/routes/about/About.css
src/components/ButtonCard/ButtonCard.tsx
src/components/ButtonCard/ButtonCard.css
src/logic/nodegraphs/fetchUser.ts
src/animations/heroIntro.ts
src/styles/global.css
src/runtime/prodivix-events.ts
src/assets/...
```

单组件导出示例：

```text
ButtonCard.tsx
ButtonCard.css
```

单 NodeGraph 导出示例：

```text
fetchUser.ts
```

同一份导出规则应由 target preset 决定，而不是由每个领域编译器各自拼路径。

### 5) CSS 以 stylesheet bundle 聚合

CSS 的输出单位是 stylesheet bundle，不是 authoring artifact。

默认聚合层级：

1. **Component CSS**：组件内部结构、节点 mounted CSS、组件私有 selector。
2. **Route/Page CSS**：页面级布局、页面节点 mounted CSS、route-specific animation keyframes。
3. **Layout CSS**：layout pattern 或 shared layout 的样式。
4. **Global CSS**：reset、font-face、theme variables、body/html、真正全局 utility。

Mounted CSS 的导出流程：

```text
CodeArtifact(mounted-css)
  -> resolve owner / scope / selectors / effective class usage
  -> attach to component/page/layout/global stylesheet bundle
  -> emit one import per stylesheet bundle
```

规则：

1. mounted CSS 不等于 production CSS file。
2. 未被当前产物引用、owner 已失效、className / selector 已清空、内容为空或只有占位注释的 CSS 不进入产物。
3. 同一 artifact 被多处引用时只合并一次。
4. CSS 排序应稳定：global -> layout -> route/page -> component，并在同一 bundle 内按 source order / tree order / artifact id 稳定排序。
5. Planner 可以保留 source map 或 source trace，但默认不把 node id 写入生产 CSS 注释。

### 6) JS/TS 也以 runtime module 聚合

NodeGraph、Animation、event handler、adapter 和 workspace module 都应按生产运行时边界组织。

NodeGraph：

```text
logic graph -> src/logic/nodegraphs/<graphName>.ts
shared executor helpers -> src/runtime/nodegraph-runtime.ts
```

Animation：

```text
timeline/easing/script -> src/animations/<animationName>.ts
CSS keyframes -> owning stylesheet bundle
shader asset -> src/shaders/<shaderName>.(glsl|wgsl)
shared animation runtime -> src/runtime/animation-runtime.ts
```

Event handlers：

```text
code-owned handler module -> keep workspace module path when production-safe
inline generated handler -> owner component/page module or runtime helper
```

规则：

1. 一个 NodeGraph node、animation track、keyframe 或 code slot 不直接等于一个 TS 文件。
2. 文件边界优先按 graph、animation、component、page、domain module 和 shared runtime 划分。
3. Planner 负责分析 shared usage：被多个 roots 使用的模块提升为 shared module；只被单 root 使用的模块放在 owner 目录。
4. CodeArtifact 原有 VFS path 可以作为 production path hint，但不是最终路径的唯一来源。

### 7) Export Planner 统一处理 imports 和 dependencies

Planner 应负责：

1. 相对 import 路径计算。
2. CSS import 插入位置。
3. package dependency 去重与版本解析。
4. runtime helper 只生成一次。
5. package.json、tsconfig、vite config、workspace metadata 等 scaffold 文件生成。
6. asset copy / rewrite / public path 处理。

领域编译器只声明 import intent：

```ts
type ExportImportIntent = {
  source: string;
  imported?: string;
  local?: string;
  kind: 'default' | 'named' | 'namespace' | 'side-effect' | 'asset-url';
};
```

Planner 最终决定 import source 是 package、relative module、CSS 文件还是 asset URL。

### 8) Source trace 是一等产物元数据

为了让 diagnostics、AI patch、导出预览和用户调试能回到作者态，Export Program 中的模块、样式和 asset 必须携带 source trace：

```ts
type ExportSourceTrace = {
  sourceRef: DiagnosticTargetRef;
  artifactId?: string;
  sourceSpan?: SourceSpan;
  ownerRootId?: string;
};
```

Source trace 用于诊断和 UI，不应默认进入生产源码注释。

### 9) Workspace 是完整项目导出的编译输入

完整项目导出由 compiler 的稳定 Workspace 入口负责，不由 Web 页面选择态拼接：

```ts
compileWorkspaceToExportProgram(workspace, options): ExportProgram;
generateWorkspaceReactViteBundle(workspace, options): ReactExportBundle;
```

`compileWorkspaceToExportProgram` 接收 canonical `WorkspaceSnapshot`，先执行 Workspace validation，再以稳定顺序编译全部 PIR page、layout 和 component document，并聚合 Workspace code document、route topology、route runtime reference、asset、project config、NodeGraph、Animation、dependency、deployment 与 source trace contribution。它生成的 React app entry 显式引用 Workspace PIR 文档模块集合，并从组合后的 RouteGraph 建立 route table；因此构建结果不依赖当前 active document 或对象插入顺序。同一个 `pir-component` 当前可以被多个 route `pageDocId` 复用，compiler 只生成一个共享模块并让多个 route entry 引用同一个 import；这是 route-level PIR artifact reuse，不是 Blueprint Component Instance。

Compiler 从 component dependency DAG 规划 definition module 与 imports，每个 Definition 只生成一个模块；Collection item template 复用同一 Instance contract。该 G1 能力与 React projection 共用 conformance。独立 `pir-graph` 与 `pir-animation` document 由对应 domain compiler 贡献 runtime module、style/asset 与 SourceTrace；Target 缺少必需 runtime capability 时产生 blocking diagnostic。

`generateWorkspaceReactViteBundle` 只负责把该完整 `ExportProgram` 交给 React/Vite preset 与 `ProductionExportPlanner`。单页面、单组件、单 NodeGraph 或单 Animation 的 resource export 仍可使用各自领域入口，但不能冒充完整 Workspace export。

`@prodivix/golden-conformance` 是这条入口的 Living Golden App。默认 Harness 会逐个 syntax-transform 所有生成的 JS/TS 模块，再外部化 bare package imports 执行无服务器的进程内 Vite/Rolldown build，用于证明生成语法、文件拓扑和相对 import。`verify:g1:standalone` 先构建当前 compiler，再从生成 `package.json` 的唯一 `packageManager` 契约选择 Corepack pnpm，在独立临时目录执行 dependency install、TypeScript typecheck、generated-project tests 与 production build。`verify:g1:browser` 重复同一独立项目序列，启动只服务 production `dist` 的临时 origin，并使用真实 Chrome 验证 runtime interaction、client routing、WebGL2 program 与 WebGPU shader compile。Visual regression 使用独立 Gate。

## 非目标

1. 不定义某个具体框架的最终目录模板细节。React / Vue / Svelte 等 target preset 可以各自扩展。
2. 不要求一次性实现 CSS Modules、tree-shaking、source map、code splitting 的全部能力。
3. 不把作者态 VFS path 简单等同于生产项目 path。

## 实施方向

### Phase 1：建立 Export Program IR 与 React planner

1. [已完成] 建立 `ExportProgram`、`ExportModule`、`ExportStyleContribution`、`ExportAssetContribution` 类型。
2. [已完成] 领域 CSS 通过 stylesheet contribution 接入 Planner。
3. [已完成] React project scaffold 消费 planner 输出的 `ExportBundle`。
4. mounted CSS 按 owner root 聚合到 component/page stylesheet。
5. 保持现有 project/component/nodegraph resource type，但通过 target preset 规划文件。

### Phase 2：接入 NodeGraph 与 Animation

1. NodeGraph compiler 输出 runtime module contribution。
2. Animation compiler 输出 animation module、keyframes style contribution、shader asset contribution。
3. Planner 统一生成 `src/runtime/*` helper。
4. 建立 shared usage 分析和 shared module 提升规则。

### Phase 3：生产质量增强

1. CSS selector usage validation。
2. optional CSS Modules target。
3. source map / source trace UI。
4. code splitting 和 route-level lazy import。
5. 多 framework target preset。

- [x] NodeGraph 和 Animation 能以 runtime module contribution 接入同一 planner。
- [x] 产物 import 路径由 planner 统一计算。
